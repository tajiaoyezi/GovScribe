# GovScribe — 智能公文 Agent（面向国企 / 央企 / 政府等事业单位）

## Problem
体制内单位的公文写作高度依赖人，专职文秘和兼职写公文的业务人员都受困于"起草耗时长、质量参差、格式规范难达标、文种选择与模板匮乏"。通用大模型（DeepSeek、豆包等）写出来的内容"不像公文"——口吻、结构、政治性表述不合规，改起来比自己写还累。问题留着不解决的代价是：核心岗位被低效重复劳动占据，关键公文的交付质量和时效无法保证。

## Evidence
- 一线访谈 / 观察：已接触过国企、政府文秘或办公室人员，掌握一手痛点反馈与工作场景观察。
- 招标 / 询单信息：见到过相关采购招标项目、客户询价或同行中标案例，存在真实采购动作。
- 竞品 / 市场数据：已有竞品在售，可佐证市场存在，但效果 / 价格 / 私有化能力不达标，留有缺口。
- 客户提供的需求材料《支持公文类型》PDF：界定了一份明确的"文种覆盖能力期望表"，含细分子类、模板生成范围、标题示例与数据量标注（标黄 = 训练数据 <100 条），代表某客户对能力边界的真实期望。

> 证据强度：问题侧为**已验证**（多来源交叉印证）。具体量化指标（采纳率、提效幅度的基线）仍为 `待验证 — 需通过客户访谈基线测量与试点 UAT 确认`。

## Users
- **Primary（核心）**：国企 / 央企 / 政府及事业单位的**专职文秘 / 办公室（综合）人员**。触发场景：日常承担大量通知、请示、报告、函、会议纪要、通报等公文的起草、核稿、排版任务，对文种规范、行文关系、机关口径要求最高。
- **Secondary（兼顾）**：业务部门**兼职写公文的员工**。触发场景：偶尔需要写通知 / 请示 / 总结，但不熟悉文种区分与行文规范，最怕"选错文种、写得不像"。
- **Not for**：MVP 阶段不服务于"只做审签 / 改稿、不从零起草"的领导核稿岗（智能核稿 / 审校能力后置，见 Out of scope）；不服务于无任何公文写作需求的一般个人用户。

## Hypothesis
我们相信，一个**懂公文规范（文种区分、行文关系、机关口径、规范结构）、覆盖高频文种且对任意文种有通用兜底**的智能公文 Agent，能够为体制内文秘与兼职写作者**显著缩短起草时间、产出"像公文、能直接用"的初稿**。
我们会通过以下信号确认假设成立：**生成初稿的采纳率达标、单篇起草时间显著下降、并通过首批试点客户的验收（UAT）**。

## Success Metrics
| Metric | Target | How measured |
|---|---|---|
| 初稿采纳率（主指标） | `待首版实测后评估`（初始参考锚点：≥ 70% 直接采用或小改后使用） | 试点期人工标注每篇初稿的采纳状态（直接用 / 小改 / 大改 / 弃用） |
| 单篇起草提效 | `待首版实测后评估`（初始参考锚点：较人工基线下降 ≥ 50%） | 试点用户分别记录"用 Agent"与"纯人工"两组的起草耗时对比 |
| 通过客户验收 | 首批试点客户完成 UAT 并签署验收（覆盖文种与质量达约定标准） | 下方"验收 rubric 草案" + 采纳率口径逐项验收 |

> 量化标准说明：客户要求**先看到实际效果再评估阈值**，故不预设硬指标；上表数值仅为初始假设锚点，待首版实测后与客户共同定档。

### 验收 rubric 草案（工作稿，待实测校准）
单篇初稿按以下维度打分，叠加采纳率口径作为 UAT 判定依据：
- **文种规范**：文种选择正确，标题 / 称谓 / 主送 / 落款符合行文规范。
- **结构完整**：符合该文种的标准段落结构与要素。
- **行文关系**：上行 / 下行 / 平行文的口吻与措辞得当。
- **机关口径**：政治性表述准确、不臆造、无敏感表述风险。
- **可用度（采纳口径）**：直接用 / 小改 / 大改 / 弃用四档，"直接用 + 小改"计入采纳。

## Scope

**MVP** — **高频文种深做 + 通用兜底 + 可配置模型层**：
- 选取一组**高频文种深做到"可直接交付"质量**：覆盖 **通知、请示、报告、函、会议纪要、通报、批复 + 讲话稿、方案**（共 9 个；前 7 为法定公文核心，讲话稿 / 方案依客户《支持公文类型》首批优先级与数据量增补，方案数据量最大）。文种按 **5 档能力阶梯**（模板生成深做 / 模版辅助写 / 框架写 / 待计划训练 / 通用兜底）分级，详见下方「文种覆盖矩阵」。开发过程中可适当补充或调整。
- **通用公文兜底**：上述高频文种无法满足时，任意场景均可走"通用公文生成"，不出现"无法处理"的死路（与 PDF 中"通用公文兜底"思路一致）。
- **文种选择引导**：用户描述场景 → Agent 正确路由到文种 / 细分子类，降低"选错文种"。
- 输出**规范正文内容**（标题、称谓、主送、正文结构、落款符合行文规范），版式排版后置。
- **在线编辑 / 核稿界面（MVP 纳入）**：文秘在浏览器内编辑 AI 初稿、核稿、决定采纳，可导出 .docx 带走精排；采用在线 Office 编辑器（OnlyOffice，见「技术选型方向」）。版式保真 / OFD 仍后置。
- **可配置的多供应商模型接入层（已决策）**：同时支持公网模型与私有化模型；公网须**同时支持 OpenAI 与 Anthropic**，`base_url / api_key / model` 等配置项灵活可配、可切换。
- **公网模型提质路线（已确认）**：公网商用模型不走微调，靠"**范文 / 模板知识库 + 检索增强（RAG）+ 文种化提示工程 + 采纳稿回流**"提升"像公文"程度。**范文库由客户提前提供大量存量公文数据完成初次建库**，并提供**补充入口**持续扩充。
- **发送前脱敏网关 + 密级路由（MVP 纳入）**：调用公网模型前，对敏感实体（人名 / 单位 / 地名 / 金额 / 文号 / 证件号 / 项目代号等）做"识别 → 占位替换 → 本地回填"，映射表不出域；范文库同样脱敏入库。配合按密级路由：非密直接走公网、敏感走脱敏公网、**涉密强制走私有化模型**。识别采用「正则规则 + **客户可维护脱敏库**（单位 / 人员名册、项目代号、涉密关键词黑名单，客户预导入 + 增删入口）+ NER 兜底」三层；每次出公网留"脱敏前后差异 + 命中项"审计供合规复核。详细工程设计见下方「技术选型方向」。⚠️ 定位为降低非密敏感数据出域风险的工程手段，脱敏非 100% 可靠，**不作为"涉密上公网"的合规承诺**（见 Risks 与 Open Question）。
- **人工最终把关（核心原则）**：Agent 定位为辅助起草，所有产出由人工评判质量、决定采纳，不自动发文。
- **速度（非功能需求）**：生成速度须"快"，无硬性最低值（阈值 `待定`）；建议以流式输出改善体感时延。
- 部署：MVP 先用公网环境快速验证效果，私有化部署形态后置（见 Post-MVP Roadmap）。

**Out of scope**（均已与干系人确认延后到 MVP 之后）
- **私有化 / 信创部署** — 私有化部署形态放在 MVP 后考虑；MVP 用公网环境验证效果与采纳率。
- **保密 / 合规适配** — 内网隔离、数据不出域、真实公文上模型的合规方案放在 MVP 后；MVP 验证仅用公开 / 脱敏 / 合成数据。
- **GB/T 9704 版式排版输出** — 不做红头文件版式 / OFD / Word 精确排版，MVP 只交付规范正文内容。
- **OA / 公文流转集成** — 不做收发文、签批流程、电子签章，只做"写"这一环。
- **智能核稿 / 审校** — MVP 只做起草生成，不做对存量文稿的合规检查与润色建议。

## Decisions（已确认）
- **部署形态** → 私有化部署形态放在 MVP 后；MVP 用公网环境验证。
- **模型接入** → 同时支持公网与私有化模型；公网同时支持 OpenAI 与 Anthropic，`base_url / api_key / model` 可灵活配置切换（MVP 范围 + 里程碑 1）。
- **公网提质路线** → 采用范文知识库 + RAG + 提示工程（不微调）；范文库由客户提供大量存量公文初次建库，提供补充入口。
- **私有化提质** → 提供可视化微调训练平台（Post-MVP），供客户使用人员自行微调。
- **高频文种清单** → MVP 深做 **9 个文种**：通知 / 请示 / 报告 / 函 / 会议纪要 / 通报 / 批复（法定公文核心）+ 讲话稿 / 方案（依客户《支持公文类型》首批优先级与数据量增补）；文种按 **5 档能力阶梯**分级（见「文种覆盖矩阵」）。开发中可适当补充 / 调整。
- **数据稀缺文种** → 标黄文种（<100 条）不纳入 MVP 深做，统一走通用兜底 + 模板辅助，不承诺"深做"质量。
- **指标阈值** → 不预设硬阈值，待首版实测后与客户共同定档（采纳率 / 提效 / 速度时延）。
- **验收口径** → 采用上方"验收 rubric 草案"为工作稿，实测后校准。
- **发送前脱敏** → MVP 纳入"脱敏网关 + 密级路由"（识别→占位→本地回填，映射不出域；范文库脱敏入库；涉密强制私有化）；定位为降低非密敏感数据出域风险，非"涉密上公网"承诺。
- **脱敏失败模式（fail-closed 为底线，允许降级）** → 网关默认 **fail-closed**：外置 NER 兜底不可用（超时 / 连接失败 / 5xx / 响应异常均算）时，**绝不静默发原文**。NER 不可用时处置优先级：① 优先自动转私有 / 本地模型；② 无私有模型可用时，**允许降级发送**（仅用 regex + 客户字典、跳过 NER 兜底）——作为按密级生效的可配置开关，**MVP 公网阶段无私有模型时降级即主回退路径**；③ 仍不满足则阻断公网调用并明确报错。**涉密永不降级**（恒走私有）；每次降级与阻断均入审计。详见 ADR-0001 D7。
- **保密合规（部署级）** → 内网隔离 / 数据不出域的部署方案放在 MVP 后；版式（GB/T 9704）放在 MVP 后。
- **技术栈** → 应用层 Golang（单二进制，利信创 / 气隙）；前端 React + Ant Design；模型接入 MVP 走统一网关、信创阶段切 Go 直连官方 SDK。详见「技术选型方向」。
- **存储与检索架构** → 向量库 Milvus（原生 BM25 + 稠密混合检索）；关系 / 审计 / 字典用 PostgreSQL；对象存储 MinIO；embedding / rerank 服务化。详见「技术选型方向」。
- **在线编辑 / Office** → MVP 采用 OnlyOffice Document Server，**先用社区版（免费）**（带 OnlyOffice 品牌、无企业级支持），经 JWT 回调与 Go 后端集成、文档存 MinIO；**正式政企交付再评估商业授权**（去品牌 / 合规 / SLA）。**OFD 暂不支持**（后续视客户需求再评估）。详见「技术选型方向」。
- **用户 / 角色 / 认证** → MVP **单租户私有部署**（每客户一套，数据按部门 / 用户 / 密级隔离，不引入多租户复杂度）；角色含 **系统管理员 / 文秘（核心）/ 业务兼职用户 / 审计员（可选，职责分离）**；MVP 用**本地账号 + RBAC（落 Postgres）**，认证做成**可插拔适配层**，信创 / 政企终局再接客户单点（CAS / OIDC / LDAP / 国产 IAM），业务不动。RBAC 管"能做什么操作"、密级管"能看 / 能外发什么数据"，两层正交。详见 ADR-0001 D10。

## 技术选型方向（已确认）
<!-- 摘要级；详细版本号/API/三库一致性方案/信创 PoC 清单见 ADR-0001：../adr/0001-tech-stack-and-architecture.md -->

经多轮评估与核验，MVP 至信创演进的技术方向已确认如下（实现细节后置到实施计划）：

- **应用层 / 后端**：**Golang**，应用层目标单二进制，利于后续信创 / 气隙部署。供应商接入用一层窄抽象封装：**MVP 走统一网关（LiteLLM Proxy）**对接公网 OpenAI / Anthropic 与私有兼容端点；**信创 / 私有化阶段切换为 Go 直连官方 SDK**，业务代码不动。
- **前端**：**React + TypeScript + Ant Design（antd）**，中后台管理 + 写作工作台；初稿流式输出走 SSE。
- **向量库**：**Milvus**，用其原生 **BM25 + 稠密向量混合检索**，一个引擎同时覆盖"文号 / 单位名等精确词召回"与"语义召回"。
- **关系库**：**PostgreSQL**，作为权威数据源（用户、模型配置、审计日志、脱敏字典、文档元数据、采纳稿回流）与访问控制判定。
- **对象存储**：**MinIO**（范文 / 生成稿原件 + 向量库后端，按用途分桶隔离）。
- **向量化 / 重排**：embedding 与 rerank 模型**服务化**（OpenAI 兼容，便于本地化 / 国产化），优先中文 embedding 模型。
- **在线编辑 / Office**：**OnlyOffice Document Server** 作为公文在线编辑/核稿界面，docx 原生保真 + 实时协同；与 Go 后端经 JWT 回调集成、文档读写走 MinIO 预签名 URL。**MVP 先用社区版（免费）**——带 OnlyOffice 品牌、无企业级支持/集群；只要不改其源码、仅经 API/JS 集成，一般不传染本应用（商用前建议法务确认 AGPL）。**正式政企交付再评估商业授权**（去品牌 / 合规 / SLA / 信创认证：麒麟有企业版认证、统信 / 龙芯待确认）。**OFD 暂不支持**（后续视客户需求再评估）。

**架构方向**
- **三库切分**：Postgres（权威源 + 访问控制）/ Milvus（可重建的派生混合索引，密级做分区、文种做标量过滤）/ MinIO（分桶）；Milvus 作派生索引，必要时可从 Postgres + MinIO 全量重建。
- **密级防越权**：以 Postgres 预过滤为权威访问控制，Milvus 标量过滤 / 分区为性能与纵深，过滤条件由后端注入。
- **文号 / 单位名精确召回**：入库时由脱敏网关的规则 + 字典能力抽取为结构化字段做精确匹配，不依赖分词器的 BM25 切分；与脱敏识别复用同一套词条来源。
- **跨库一致性**：关系库先落地、向量库异步建索引 + 软删 + 定期对账，不追求跨库强一致。
- **编辑器可替换**：在线编辑接入做成可替换适配层（前端组件 + 后端回调/导出接口），内容按公文语义结构化存储、与版式解耦，导出抽象 `export(docx)` 预留 `export(ofd)`，便于信创终局换 WPS / 接 OFD 服务时业务不动。

**部署路径**：MVP 公网快速验证（向量库可临时用托管服务、数据驻留境内）→ 信创 / 私有化全本地离线，LLM / embedding / rerank / 向量库 / 对象存储均私有化，脱敏映射表不出域。

## Open Questions
- [ ] **密级策略由客户设定（政策项，不阻塞工程）**：脱敏库 + 密级路由让策略"可配置、可强制执行、可审计"，但"哪些密级 / 类型允许脱敏后上公网、哪些强制私有化"属客户合规 / 保密部门的**政策决定**——建好工具 ≠ 自动合规，须客户拍板并接受脱敏漏识别的残余风险。工程侧按其策略落地，不阻塞 MVP 开工。
- [ ] **范文数据到位情况（开工前盘点，影响效果验证而非阻塞开工）**：公网提质走 RAG——写作时从范文库检索同类优秀公文当样例喂给模型，初稿"像不像公文"取决于库里有没有可检索的好范文。**库空或某文种范文过少 → 检索不到样例 → 退回通用模型"不像公文" → 价值假设无从验证**（建库"入口"是已纳入的功能；这里指的是"入口里到底灌进多少数据"这一前置条件）。须开工前与客户盘点：① 9 个高频文种**每种可提供多少篇优质存量范文**（客户 PDF 已给出部分数据量，如方案：工作 1815 / 活动 740 / 整治 315 条；参照标黄线 <100 条 = 数据稀缺，每文种宜 ≥100 篇方有"深做"底气，不足者降级走通用兜底）；② 数据格式（docx / 扫描件 / PDF 等）与**到位时间**（是否卡里程碑 2 / 5）；③ 清洗 / 脱敏入库责任方。不预设硬数字入合同，仅作开工前数据盘点 + 风险登记。

## Risks
| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| 私有化阶段国产 / 私有模型效果不达标（公网验证效果在交付环境复现不了） | 高 | 高 | 已落地可配置多供应商模型层 + 共用范文知识库降低对单一模型的依赖；私有线叠加微调 / 可视化训练平台兜底；尽早用目标国产模型做一轮高频文种效果摸底 |
| "不像公文"是主观判断，验收标准难量化，导致验收扯皮 | 高 | 高 | 启动前与客户共建可量化的文种质量 rubric（口吻 / 结构 / 行文关系 / 政治性表述分项打分）与采纳率口径 |
| 保密红线：真实公文经 RAG 调用公网模型导致数据出域 | 高 | 高 | 发送前脱敏网关（识别→占位→本地回填，映射不出域）+ 范文库脱敏入库 + 按密级路由（涉密强制私有化）；残余风险：脱敏漏识别，故涉密内容不走公网，密级口径须客户确认 |
| 脱敏过度导致初稿丢失单位口吻 / 称谓 / 上下文、生成质量下降 | 中 | 中 | 脱敏强度分级可配置；保留类型化占位符与指代一致性；非密公文可不脱敏直接走公网以保质量 |
| 数据稀缺文种（标黄 <100 条）生成质量不足 | 中 | 中 | 稀缺文种优先走通用兜底 + 模板辅助，不承诺"深做"质量；是否纳入 MVP 由客户确认 |
| 范文库数据不足或未按时到位，RAG 检索不到优质样例，初稿退回"不像公文"，MVP 效果无法验证 | 中 | 高 | 开工前按文种盘点客户存量范文数量（对照 <100 条稀缺线，每文种宜 ≥100 篇）；不足文种降级走通用兜底、不承诺"深做"；约定数据到位时间与清洗 / 脱敏入库责任方，避免"功能就绪、库空"导致 UAT 说不清是模型问题还是缺数据 |
| 竞品已在售，差异化不足 | 中 | 中 | 以"懂公文规范 + 私有化 / 信创能力 + 高频文种交付质量"为差异点，试点采纳率数据作为竞争证据 |
| 政治性 / 合规内容风险（公文涉政治表述，错误代价高） | 中 | 高 | 明确 SFW / 合规红线，敏感表述不臆造；保留人工核稿环节，Agent 定位为"辅助起草"而非"自动发文" |
| 在线 Office（OnlyOffice）商业授权成本与信创认证未落实 | 中 | 中 | 社区版 AGPL + 禁去品牌，政企交付须预算商业授权（Developer/Enterprise）；麒麟有企业版认证、统信 / 龙芯待书面确认（龙芯非 ARM64 官方镜像不覆盖）；编辑器做成可替换适配层，信创终局可切 WPS 私有化（OFD 原生）兜底 |
