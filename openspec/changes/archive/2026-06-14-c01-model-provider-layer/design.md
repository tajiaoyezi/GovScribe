## Context

GovScribe 的全部生成能力都建立在大模型之上，而交付环境跨度极大：MVP 阶段在公网调用 OpenAI / Anthropic，信创 / 私有化阶段则切到国产或私有 OpenAI 兼容端点，且供应商、`base_url`、`api_key`、`model` 都要由管理员运行时灵活配置切换（PRD Milestone 1「可配置模型接入层」、Decisions「模型接入」「技术栈」）。当前代码库尚无任何模型调用入口，`openspec/specs/` 下无既有 capability，本 change 是整条产品的最底层基础设施，无前序依赖。

关键约束来自 ADR-0001：

- **D1**：后端用 Go，应用层目标单二进制（利信创 / 气隙），供应商接入用一层**窄抽象接口**封装。
- **D2**：初稿流式输出走 SSE。
- **D3**：模型接入 MVP 走 **LiteLLM Proxy** 统一对接 OpenAI / Anthropic / 私有兼容端点；信创阶段切 **Go 直连官方 SDK**，业务代码不动。该决策同时锁定了三个已核验坑：`openai-go` 导入路径必须带 `/v3`（裸路径会拉到过期的 v1.12）、LiteLLM 热切换对在途流不生效且多 worker 有约 30s 传播延迟、Anthropic 经 LiteLLM 的 OpenAI 格式流式存在字段映射损耗与已知丢失 bug（#27946）。
- **D5**：关系型权威数据落 PostgreSQL。

本 change 只解决"如何可靠地把请求发给一个可配置模型并流式返回"这一件事，把它收敛成一层稳定接口，让上层（c02 脱敏 / 密级路由、c03 RAG、文种生成）只面向接口编程。具体的供应商抽象契约、配置生命周期、流式透传契约见同目录三份 specs（`model-provider-abstraction`、`model-config-management`、`model-streaming-passthrough`），动机见 `proposal.md`；本文只解释 HOW 与为什么这样选。

## Goals / Non-Goals

**Goals:**

- 定义并落地一层 Go 窄抽象接口，作为上层调用大模型的唯一入口，覆盖一次性 chat completion 与流式生成两种形态，且两种后端实现行为等价。
- 提供两个可互换后端：MVP 的 LiteLLM Proxy 后端、信创的 Go 直连官方 SDK 后端，切换为配置项而非代码改造。
- 模型配置落 PostgreSQL 作为权威源，支持管理员增改查 / 启停、运行时切换当前生效配置、保存或切换前的连通性探活。
- 统一流式归一：把不同后端的分片归一为统一事件 / 增量契约（文本增量 / 正常结束 / 异常结束），供 c05 / c07 的生成端点封装后向前端透传，本 change 不持有面向前端写作工作台的生成端点；并对非标字段做宽松解析、对流中断给出可观察行为。
- 向上层暴露"当前生效供应商是公网还是私有"的可读状态，为 c02 的密级路由与 fail-closed 提供事实依据。

**Non-Goals:**

- **不做脱敏与密级路由判定**（属 c02）：本层只暴露公网 / 私有事实，不决定"什么数据能否外发"。
- **不做 RAG 检索 / 上下文拼装**（属 c03），不做 embedding / rerank 调用（这两类调用走各自 capability，且 `openai-go` 接 embedding 的 `EncodingFormat="float"` 坑、rerank 自写 client 等属 ADR-0001 D7 范畴，不在本层）。
- 不做提示工程、文种模板、初稿质量评估等业务语义。
- 不做信创私有化 / 气隙部署本身（属后置里程碑）；本层只保证后端可替换，使部署切换时业务代码不动。
- 不引入第二套版本 / 许可口径：所有版本、许可、PoC 一律以 ADR-0001 D3 及其「版本与许可锚点」表为准。

## Decisions

### D-A：以 Go 窄抽象接口作为唯一模型调用入口（依据 ADR-0001 D1）

定义一个面向公文生成场景**裁剪过的**最小接口，而非镜像任何一家 SDK 的全量能力。接口只暴露两个方法形态——一次性生成与流式生成——入参为消息列表 + 生成参数，返回与具体后端无关的结构（生成文本 / 增量事件 + 统一的结束原因枚举 + 统一的错误归因分类）。上层业务包的导入图中不得出现 `github.com/openai/openai-go`、`github.com/anthropics/anthropic-sdk-go` 或任何 LiteLLM 专有客户端类型（见 `model-provider-abstraction` 的依赖扫描 Scenario）。

- **为什么"窄"而非"全"**：公文起草只需文本生成与流式两种形态；把接口收窄到业务真正用到的能力，才能让两家差异极大的后端（HTTP Proxy vs 原生 SDK）真正等价实现，也避免把某家 SDK 的特有概念（如 Anthropic 的 thinking 块、OpenAI 的 tool_calls 细节）泄漏成接口契约后在另一后端无法兑现。
- **替代方案**：(1) 上层直接 import LiteLLM 客户端——被否，信创阶段要重写全部调用链，违背 D1；(2) 直接以 `openai-go` 的类型作为内部通用类型——被否，会把 OpenAI 语义钉死成事实标准，Go 直连 Anthropic 时需做反向适配，且把 SDK 升级风险传导到全业务。

### D-B：后端可替换契约 = LiteLLM Proxy 后端（MVP）+ Go 直连 SDK 后端（信创）（依据 ADR-0001 D3）

抽象接口背后提供两个实现，二者对同一方法的可观察行为必须等价（结果结构、结束原因取值集合、错误归因分类一致，见 `model-provider-abstraction` 的「两后端行为等价」Scenario）。

- **LiteLLM 后端**：以 HTTP 调用 LiteLLM Proxy 的 OpenAI 兼容端点，由 Proxy 负责把请求路由到 OpenAI / Anthropic / 私有兼容端点。MVP 选它是因为一处接入即统一三类供应商，公网快速验证成本最低。
- **Go 直连后端**：信创阶段用 `github.com/openai/openai-go/v3`（**导入路径必须带 `/v3`**，Apache-2.0）与 `github.com/anthropics/anthropic-sdk-go`（MIT，需 Go 1.24+），均支持 `WithBaseURL` 指向私有兼容端点。选它是因为 LiteLLM 是 Python + Postgres、非单二进制，与 D1 的单二进制气隙目标冲突，信创阶段须退场。版本 / 许可一律以 ADR-0001 D3 与「版本与许可锚点」表为准，本文不另立。
- **接口归一的代价由后端吸收**：两家 SDK 的结束原因、错误形态各异，归一映射逻辑放在各后端实现内部，而非泄漏给上层。重度依赖 thinking 的场景按 ADR-0001 D3 走原生 `/v1/messages` 或直连 `anthropic-sdk-go`，Go 端用宽松 JSON 接收非标字段——这一处理在 Go 直连后端内部完成，不改接口。
- **替代方案**：(1) 只做 LiteLLM 后端、信创再说——被否，会让"信创不改业务代码"的承诺落空，等于把 D1/D3 的核心价值推迟到来不及验证；(2) 信创也继续用 LiteLLM——被否，非单二进制与气隙 / loong64 引擎缺失（ADR-0001 PoC 清单）使其不可行。

### D-C：启动期后端选择与模型配置解耦（依据 `model-provider-abstraction`）

"用哪条通道"（后端选择：LiteLLM / Go 直连）与"调哪个供应商的哪个模型"（模型配置）是两个正交维度。后端选择是启动期配置项：应用启动读取后，按值实例化对应后端并注入抽象接口；配置缺失或非法时**拒绝以不确定状态启动并报明确错误**，绝不静默回退默认后端（见对应 Scenario）。这样 MVP→信创的后端替换是改一个启动配置项 + 换部署形态，而模型的日常增改切换走 D-D 的运行时机制，互不干扰。

- **为什么后端选择放启动期、模型配置放运行时**：后端切换必然伴随部署形态变更（LiteLLM 进 / 退场、依赖与镜像不同），属于发布事件，没有运行时热切的现实需求；而模型配置（换 key、换 model、切公网 / 私有）是管理员日常操作，必须热切。把二者绑在同一开关上会过度复杂化。

### D-D：模型配置落 PostgreSQL 为权威源 + 运行时热切（依据 ADR-0001 D3、D5）

每条配置至少含：供应商类型、`base_url`、`api_key`、`model`、网络属性（公网 / 私有）、启用状态（见 `model-config-management`）。落 PostgreSQL（D5）作权威源；管理员可增改查、启停；支持运行时切换当前生效配置而不重启应用层（新请求即用新配置）。

热切语义**显式吸收 ADR-0001 D3 已核验行为**，不向管理员承诺"立即对所有在途请求生效"：

- 在途流不受切换影响，继续以原配置完成或按既定中断策略结束（见「在途流不受切换影响」Scenario）。
- LiteLLM 后端下存在多 worker 约 30s 传播延迟与 Redis 缓存 TTL 陈旧期，UI 须如实告知。LiteLLM 侧热切换需开 `store_model_in_db` 并经 `/model/new` API（ADR-0001 D3）。
- **网络属性在创建时确定且可被上层读取**：这是与 c02 的两个契约面之一——本层只暴露"公网 / 私有"事实，c02 据此做密级路由与 fail-closed，本层不替它判定。与 c02 的契约面共两个：① 公网 / 私有网络属性（可读状态）；② 随请求透传的出站内容密级（供 c02 拦截点读取，c01 不解读）——后者指窄抽象调用承载的**出站请求内容密级（取值：非密 / 敏感 / 涉密；允许缺省为「缺失/未知」）**，c01 原样透出、不解读不判定不改写（见 `model-provider-abstraction`「按请求承载出站内容密级供 c02 拦截点读取」需求）。

`api_key` 等敏感字段妥善保管：读取接口返回体掩码或省略明文，审计记录不含 `api_key` 明文（保留操作者 / 时间 / 目标配置标识 / 动作类型，见对应 Scenario）。

- **替代方案**：(1) 配置走环境变量 / 配置文件——被否，无法满足运行时切换与管理端增改，且气隙环境改文件 + 重启代价高；(2) 仅依赖 LiteLLM 自身的 DB 存模型而 Go 端不存——被否，信创 Go 直连后端没有 LiteLLM，权威源必须在 GovScribe 自己的 PostgreSQL，才能让两后端共享同一套配置语义。

### D-E：保存 / 切换前连通性探活（依据 `model-config-management`）

切换或保存配置前可对目标端点发起一次**最小代价探活**（不产生实质业务生成内容），返回可用 / 不可用及基础错误归因（鉴权失败 / 端点不可达 / 模型名不存在 / 超时）。探活通过应作为"允许设为当前生效配置"的前置条件；探活失败不得把该配置静默推上线。停用当前生效配置时，须先切到另一条已启用且通过探活的配置，**绝不允许进入"无生效配置"的不确定状态**（见「停用配置」Scenario）。这与 c02 的 fail-closed 红线同向：宁可拒绝上线，不静默放行不可用 / 不安全的通道。

### D-F：统一流式归一契约（依据 ADR-0001 D2、D3）

本 change 把后端流式分片归一为统一事件 / 增量契约，供上层 c05 / c07 的生成端点封装后向前端透传；本 change **不持有面向前端写作工作台的生成端点**（归 c05 / c07）。无论底层是哪个后端，消费方收到的事件类型与增量语义一致，归一契约至少表达三类：文本增量、正常结束（带结束原因）、异常结束（带错误归因）。两处坑按 ADR-0001 D3 显式处理：

- **宽松解析非标字段**：以宽松 JSON 接收后端分片，遇未知 / 缺失的非关键字段忽略并继续归一透传，整条流不因解析非标字段中断（吸收 Anthropic 经 LiteLLM 的字段映射损耗 / 丢失 bug #27946）。关键字段损耗时尽力透传已到达增量并记录可观察诊断（如损耗计数），**绝不向消费方伪造不存在的内容**。
- **流中断可观察**：后端断开 / 超时 / 上游报错时，主动向消费方发携带错误归因的异常结束事件，绝不让消费方的事件流无限期挂起；已透传增量不回滚，异常结束事件须让消费方能区分"内容已完整结束"与"因中断而不完整"。

## Risks / Trade-offs

- **[LiteLLM 热切换语义被误解为"即时全局生效"]** → 在管理端 UI 与切换接口响应中显式标注"对在途流不生效、多 worker 约 30s 传播延迟、缓存有 TTL 陈旧期"（ADR-0001 D3）；切换接口不返回"已全局生效"的强承诺，只确认"新请求将使用新配置"。
- **[Anthropic 经 LiteLLM 的 OpenAI 格式流式字段损耗 / 丢失（#27946）导致内容缺失或解析崩溃]** → 宽松 JSON 接收非标字段、损耗时尽力透传 + 记诊断计数、绝不伪造内容（D-F）；重度依赖 thinking 的场景按 ADR-0001 D3 走原生 `/v1/messages` 或信创阶段直连 `anthropic-sdk-go`。
- **[`openai-go` 裸导入路径拉到过期 v1.12]** → 信创后端导入路径强制带 `/v3`，并在依赖锁文件与 PoC 检查项中固化（ADR-0001 D3、PoC 清单）。
- **[`api_key` 等敏感字段泄漏到日志 / 接口返回 / 审计]** → 读取接口掩码、审计不落明文（D-D）；这是与 c02 保密红线同向的硬约束，违反即视为缺陷。
- **[两后端行为不等价，上层据 LiteLLM 行为写的处理逻辑在 Go 直连下失效]** → 把结束原因枚举与错误归因分类定义为接口契约的一部分，归一映射在各后端内部完成；信创切换前以同一组消息 / 参数做两后端行为等价回归（对应 Scenario）。
- **[停用 / 切换操作把系统推入"无生效配置"状态]** → 切换目标须为已启用且探活通过的配置；停用当前生效配置须先切走，否则拒绝（D-E）。

### 对脱敏网关、密级防越权、三库一致性的影响

- **脱敏网关（c02）**：本 change **不触及**脱敏与密级路由判定，逻辑由 c02 叠加在本接入层之上。本层向 c02 提供两个支撑面：① "当前生效供应商公网 / 私有"可读状态（D-A、D-D），c02 据此做密级路由与 fail-closed；② **按请求定向到指定私有配置的出站能力**（见 `model-provider-abstraction`「按请求定向到指定供应商配置」需求）——c02 的"转私有"在拦截点经此能力按单次调用定向到一条已启用的私有配置且不改全局当前配置；MVP 无任何已启用私有配置时该定向不可达，本层以"无可用私有配置"信号拒绝，由 c02 据此按 ADR-0001 D7 降级（仅 regex + 字典、需该密级显式开启）或阻断，涉密永不降级。本层的探活拒绝上线、停用须先切走等设计与 c02 的 fail-closed 同向，不与之冲突。**权衡**：本层不替 c02 做任何外发判定与降级 / 阻断决策，避免把保密决策下沉到基础设施层造成职责越界；代价是 c02 必须信任本层暴露的网络属性事实，故该属性须在创建时确定、与生效配置强一致、不返回模糊状态（对应 Scenario）。
- **密级防越权**：涉密恒走私有、永不降级的红线由 c02 执行，但本层须保证"私有"配置的网络属性不会被错误标成"公网"、也不会在切换 / 缓存陈旧期暴露不一致状态——这是防越权链条的事实源头。**权衡**：为此本层宁可在配置不确定时拒绝上线 / 拒绝启动（D-C、D-E），牺牲一点可用性换取网络属性事实的可靠性。
- **三库（PostgreSQL / Milvus / MinIO）一致性**：本 change 只写 **PostgreSQL**（模型配置权威源，D5），**不触及 Milvus 与 MinIO**，不引入跨库一致性问题。模型配置与三库中的业务数据（范文 / 生成稿）无外键或事务耦合；切换模型配置不影响已落 MinIO 的稿件与已入 Milvus 的向量。**权衡**：把配置权威源单独收在 PostgreSQL、与对象 / 向量库解耦，使运行时热切配置不会牵动三库一致性，代价是配置变更的传播（尤其 LiteLLM 多 worker）只能做到最终一致而非强一致，已在 D-D 显式接受。

## Migration Plan

本 change 是全新基础设施、无既有调用方迁移，"迁移"主要指 MVP→信创的后端替换路径与回滚。

1. **MVP 上线（LiteLLM 后端）**：部署 LiteLLM Proxy（Python + Postgres），开 `store_model_in_db`；GovScribe 应用以启动配置项选定 LiteLLM 后端（D-C）；PostgreSQL 建模型配置表（D-D）；管理员录入公网 OpenAI / Anthropic 与私有兼容端点配置，逐条探活通过后设当前生效配置。
   - 验证：依赖扫描确认上层包无具体 SDK / LiteLLM 客户端导入；一次性生成与流式生成各跑通；探活通过 / 失败归因正确；切换不重启即对新请求生效且 UI 如实标注传播延迟。
2. **信创切换（Go 直连后端）**：引入 `openai-go/v3`（路径带 `/v3`）与 `anthropic-sdk-go`（Go 1.24+）；把启动配置项切到 Go 直连后端；LiteLLM 退场。上层业务代码零改动（D-B 等价契约保证）。
   - 验证：以同一组消息 / 参数做两后端行为等价回归（结果结构 / 结束原因 / 错误归因一致）；私有兼容端点经 `WithBaseURL` 接通；流式非标字段宽松解析不崩。
   - 前置 PoC：按 ADR-0001「信创 / 私有化必做 PoC 清单」核验目标国产平台（含 loong64 等高风险项），不向客户提前承诺。
3. **回滚**：后端选择是启动配置项，回滚即改回原值 + 恢复对应部署形态（重启应用层，不改业务代码、不动 PostgreSQL 配置数据）。模型配置层面的回滚为切回上一条已探活通过的配置（运行时、不重启）。

## Open Questions

- 生成参数（如 temperature、max_tokens、是否启用 thinking）应固化为接口契约的一部分，还是作为可选透传字段交由各后端按需映射？影响两后端等价性的边界，待与文种生成 change 对齐。
