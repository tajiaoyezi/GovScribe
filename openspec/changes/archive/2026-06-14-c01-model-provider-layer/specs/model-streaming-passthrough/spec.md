## ADDED Requirements

### Requirement: 统一流式事件 / 增量归一契约

系统必须（MUST）把不同后端的流式分片归一为一套统一的事件 / 增量契约（对齐 ADR-0001 D2），供上层 c05 / c07 的生成端点封装后向前端透传；本 change 不持有面向前端写作工作台的生成端点（归 c05 / c07）。无论底层是 LiteLLM 后端还是 Go 直连 SDK 后端，消费方接收到的事件类型与增量文本语义必须一致。归一契约必须至少能表达：文本增量、正常结束、异常结束三类事件，使消费方无需感知后端差异即可封装其生成端点。

#### Scenario: 增量逐步归一

- **WHEN** 后端逐分片返回生成内容
- **THEN** 系统必须把每个文本增量按到达顺序归一为统一增量事件交付给消费方
- **AND** 消费方据此拼接得到的最终文本必须与一次性生成同一请求的完整文本在语义上等价

#### Scenario: 正常结束事件

- **WHEN** 后端流正常结束
- **THEN** 系统必须发送一个明确的结束事件，携带结束原因（如正常结束、达到长度上限）
- **AND** 结束事件之后必须不再发送该请求的任何文本增量

### Requirement: 宽松解析非标流式字段

系统必须（MUST）以宽松 JSON 接收后端流式分片中的非标准字段，显式吸收"Anthropic 经 LiteLLM 的 OpenAI 格式流式存在字段映射损耗 / 已知丢失 bug"等已核验坑（ADR-0001 D3）。遇到无法识别或缺失的非关键字段时，系统必须不中断整条流，而应当忽略未知字段并继续透传可用增量。

#### Scenario: 遇到未知字段不崩溃

- **WHEN** LiteLLM 后端返回的流式分片中包含抽象接口未预期的额外字段
- **THEN** 系统必须忽略这些未知字段并正常透传该分片中的文本增量
- **AND** 整条流必须不因解析非标字段而中断

#### Scenario: 关键字段映射损耗的可观察处理

- **WHEN** 某分片因 Anthropic 经 LiteLLM 的已知映射损耗而缺失关键字段
- **THEN** 系统应当在不丢弃已到达文本增量的前提下尽力透传
- **AND** 系统应当记录可观察的诊断信息（如损耗计数）以便定位，且必须不向消费方伪造该分片不存在的内容

### Requirement: 流中断的可观察行为

系统必须（MUST）为流式生成的异常中断（如后端连接断开、超时、上游报错）定义明确且可观察的行为。中断发生时，系统必须向消费方发送一个异常结束事件而不是静默挂起，使消费方能据此终止其向前端的透传并提示停止等待。

#### Scenario: 后端中途断开

- **WHEN** 流式生成进行中，后端连接断开或上游返回错误
- **THEN** 系统必须停止该流并向消费方发送携带错误归因的异常结束事件
- **AND** 系统必须不让消费方的事件流无限期挂起而无任何终止信号

#### Scenario: 已透传增量不回滚

- **WHEN** 流在已归一透传若干文本增量后异常中断
- **THEN** 已交付给消费方的增量必须保持有效，系统必须不要求消费方回滚已交付内容
- **AND** 异常结束事件必须使消费方能区分"内容已完整结束"与"内容因中断而不完整"
