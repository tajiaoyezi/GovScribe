## 1. 抽象接口契约设计（先于任何后端实现）

- [ ] 1.1 定义 Go 窄抽象接口包（如 `internal/llm`），只暴露两种调用形态：一次性 chat completion 与流式生成；入参为消息列表 + 生成参数，返回结构与具体后端无关（对齐 ADR-0001 D1、spec `model-provider-abstraction`）。
- [ ] 1.2 定义接口契约内的统一枚举：结束原因（正常结束 / 达到长度上限 / 被内容策略拦截）与错误归因分类（鉴权失败 / 端点不可达 / 模型名不存在 / 超时 / 上游报错），作为两后端必须一致的可观察行为基线。
- [ ] 1.3 在抽象接口上定义"读取当前生效供应商网络属性（公网 / 私有）"的方法，明确该状态必须与当前生效模型配置强一致、不得返回未知 / 模糊态（供 c02 密级路由与 fail-closed 读取，本 change 不做判定）。
- [ ] 1.4 明确生成参数（temperature、max_tokens、是否启用 thinking 等）在接口契约中的归属：MVP 先按 design.md Open Questions 落"可选透传字段、由各后端按需映射"，并在接口注释中标注待与文种生成 change 对齐；当前生效配置先做全局单值，不做按文种多路。
- [ ] 1.5 在抽象接口上定义"按请求定向到指定供应商配置"的能力（供 c02 转私有）：调用方可在单次调用粒度指定一条已启用的私有配置出站，且不改变全局当前生效配置、不影响其它请求；当不存在任何已启用私有配置时返回明确的"无可用私有配置"信号并拒绝定向、绝不静默回退公网，由上层 c02 据此按 ADR-0001 D7 降级 / 阻断（本 change 不做密级判定与降级 / 阻断决策）。
- [ ] 1.6 在窄抽象接口入参新增"出站请求内容密级（取值：非密 / 敏感 / 涉密；允许缺省为「缺失/未知」）"一项，其值即 c06 结构化场景上下文契约的「内容密级」项：c01 原样承载并在公网分支原样透出给 c02 拦截点，自身不解读 / 不判定 / 不改写该密级，也不映射进后端 SDK 的生成参数；调用方未传时如实呈现「缺失/未知」、不补默认值。验证：c02 拦截点可从调用读到该密级且能区分"已传入 vs 缺失/未知"。

## 2. 模型配置持久化与生命周期（PostgreSQL 权威源）

- [ ] 2.1 设计并迁移 PostgreSQL 模型配置表，字段至少含：供应商类型（OpenAI / Anthropic / 私有 OpenAI 兼容端点）、`base_url`、`api_key`、`model`、网络属性（公网 / 私有）、启用状态、唯一标识与时间戳（对齐 ADR-0001 D5、spec `model-config-management`）。
- [ ] 2.2 实现 `api_key` 的妥善保管：落库前不以明文进日志、读取接口返回体对 `api_key` 掩码或省略；按 design.md 锁定的契约只保证"不明文回显、不入审计明文"，具体保管强度（应用层加密 / 磁盘加密 / 外部 KMS）留待与 c02 保密体系协同确定。
- [ ] 2.3 实现配置增 / 改 / 查管理端接口；网络属性（公网 / 私有）在创建时确定、之后可被抽象接口读取；修改命中当前生效配置时其生效语义遵循第 4 组的运行时切换规则。
- [ ] 2.4 实现启用 / 停用接口：停用后该配置不可再被选为当前生效配置；停用恰为当前生效的配置时，必须拒绝或要求先切到另一条"已启用且探活通过"的配置，绝不允许进入"无生效配置"的不确定状态。
- [ ] 2.5 实现配置变更审计记录：保留操作者 / 时间 / 目标配置标识 / 动作类型，且审计记录中绝不含 `api_key` 明文。
- [ ] 2.6 模型配置管理端各动作（增 / 改 / 查 / 启停 / 切换 / 探活 / 审计入口）消费 c04 `rbac-authorization` 的授权决策（`model.config` 权限点），本 change 不自建 RBAC 判定。

## 3. 连通性探活（设为生效配置的前置条件）

- [ ] 3.1 实现连通性探活：对目标端点发起一次最小代价调用，绝不产生实质业务生成内容；返回可用 / 不可用结果。探活的统一形态（最小 chat 请求 vs models 列表查询）按各供应商真实行为在实现期确定（design.md Open Question）。
- [ ] 3.2 探活失败时返回基础错误归因分类（鉴权失败 / 端点不可达 / 模型名不存在 / 超时），复用第 1.2 组的统一错误归因枚举。
- [ ] 3.3 把"探活通过"接入为"允许设为当前生效配置"的前置校验：探活未通过的配置必须不能被静默推上线（与 c02 fail-closed 同向）。

## 4. 运行时切换语义（显式吸收 LiteLLM 已核验行为）

- [ ] 4.1 实现"运行时切换当前生效配置"：切换后新发起的请求即用新配置，全程不重启应用层；切换目标必须为已启用且探活通过的配置。
- [ ] 4.2 实现"在途流不受切换影响"：切换发生时正在进行的流式生成继续以原配置完成或按既定中断策略结束，不在流中途改用新配置。
- [ ] 4.3 切换接口响应与管理端 UI 须如实告知 LiteLLM 后端已核验行为：对在途请求不生效、多 worker 约 30s 传播延迟、Redis 缓存 TTL 陈旧期；切换接口绝不返回"已全局立即生效"的强承诺，只确认"新请求将使用新配置"（ADR-0001 D3）。

## 5. 启动期后端选择（与模型配置解耦）

- [ ] 5.1 实现启动期后端选择配置项：应用启动时读取该项，按值实例化对应后端（LiteLLM / Go 直连）并注入抽象接口；该选择随发布事件确定、非运行时热切，运行时热切仅作用于模型配置；后端选择与模型配置正交（后端=以何通道发起，配置=调哪个供应商的哪个模型，design.md D-C）。
- [ ] 5.2 后端选择缺失或非法时，必须拒绝以不确定状态启动并报明确错误，绝不静默回退任意默认后端；运行时不提供后端热切入口，切后端须改启动配置项并重启应用层。

## 6. LiteLLM Proxy 后端实现（MVP 默认通道）

- [ ] 6.1 实现 LiteLLM 后端：以 HTTP 调用 LiteLLM Proxy 的 OpenAI 兼容端点完成一次性 chat completion，由 Proxy 路由到 OpenAI / Anthropic / 私有兼容端点；把后端返回归一为第 1.2 组定义的结束原因与错误归因。
- [ ] 6.2 实现 LiteLLM 后端的流式调用，把分片归一为统一事件流（接入第 7 组的 SSE 透传契约）。
- [ ] 6.3 部署 / 配置 LiteLLM Proxy（Python + Postgres）并开启 `store_model_in_db`，模型热切换经 `/model/new` API；记录其为非单二进制、信创阶段退场的依赖（ADR-0001 D3）。

## 7. 统一流式归一契约（供 c05 / c07 生成端点封装）

- [ ] 7.1 实现把后端流式分片归一为统一事件 / 增量契约（对齐 ADR-0001 D2），至少表达三类事件：文本增量、正常结束（带结束原因）、异常结束（带错误归因）；不同后端在消费方看来事件类型与增量语义一致。该契约供上层 c05 / c07 的生成端点封装后向前端透传，本 change 不实现面向前端写作工作台的生成端点（归 c05 / c07）。
- [ ] 7.2 实现增量逐步归一：按到达顺序交付文本增量，保证消费方拼接结果与同一请求一次性生成的完整文本语义等价；正常结束事件之后不再交付该请求的任何增量。
- [ ] 7.3 实现宽松 JSON 解析非标字段：遇未知 / 缺失的非关键字段忽略并继续归一透传，整条流绝不因解析非标字段中断（吸收 Anthropic 经 LiteLLM 的字段映射损耗 / 丢失 bug #27946，ADR-0001 D3）。
- [ ] 7.4 实现关键字段损耗的可观察处理：尽力透传已到达增量、记录可观察诊断（如损耗计数），且绝不向消费方伪造分片不存在的内容。
- [ ] 7.5 实现流中断的可观察行为：后端断开 / 超时 / 上游报错时主动向消费方发携带错误归因的异常结束事件，绝不让消费方的事件流无限期挂起；已透传增量不回滚，异常结束事件须让消费方能区分"内容已完整结束"与"因中断而不完整"。

## 8. 上层调用约束验证（窄抽象的护栏）

- [ ] 8.1 增加依赖扫描 / 静态检查：确保上层业务包（脱敏、密级路由、文种生成、RAG 编排）的导入图中不出现 `github.com/openai/openai-go`、`github.com/anthropics/anthropic-sdk-go` 或任何 LiteLLM 专有客户端类型，全部模型调用只经窄抽象接口。
- [ ] 8.2 编写一次性生成与流式生成的契约级回归用例（覆盖结束原因、错误归因、网络属性读取），作为后续两后端行为等价的对照基线。

## 9. 信创 / 私有化后端（Go 直连 SDK，待 PoC 与里程碑触发）

- [ ] 9.1 实现 Go 直连后端：用 `github.com/openai/openai-go/v3`（导入路径必须带 `/v3`，否则拉到过期 v1.12；Apache-2.0）与 `github.com/anthropics/anthropic-sdk-go`（MIT，v1.50.x，需 Go 1.24+），均经 `WithBaseURL` 指向私有兼容端点；版本 / 许可一律以 ADR-0001 D3 及版本表为准。
- [ ] 9.2 把两家 SDK 各异的结束原因与错误形态在 Go 直连后端内部归一为第 1.2 组的统一枚举；重度依赖 thinking 的场景走原生 `/v1/messages` 或直连 `anthropic-sdk-go`，Go 端用宽松 JSON 接收非标字段（ADR-0001 D3）。
- [ ] 9.3 在依赖锁文件与 CI 检查项中固化 `openai-go` 的 `/v3` 导入路径，防止裸路径退化到 v1.12（ADR-0001 D3、PoC 清单）。
- [ ] 9.4 以同一组消息 / 参数做 LiteLLM 后端与 Go 直连后端的行为等价回归：结果结构、结束原因取值集合、错误归因分类必须一致，验证"切后端不改业务代码"。

## 10. 待 PoC（信创 / 私有化高风险，勿向客户提前承诺）

- [ ] 10.1 [PoC] 龙芯 LoongArch64：LiteLLM / Prisma 无 loong64 引擎（ADR-0001 PoC 清单），核验该平台下模型接入层的可行替代路径（信创阶段须切 Go 直连后端规避 LiteLLM），结论未验证前不向客户承诺。
- [ ] 10.2 [PoC] 鲲鹏 ARM64 + 麒麟：核验私有兼容端点 + Go 直连后端在 ARM64 / 64KB 页环境下的可用性与构建链，锁定已验证镜像 / tag（ADR-0001 PoC 清单）。
- [ ] 10.3 [PoC] 国产模型效果摸底：在目标私有 / 国产兼容端点上对一次性生成与流式生成做最小连通与基础效果摸底，仅验证接入层"可靠发请求并流式返回"，不涉及文种质量评估（属后续 change）。
