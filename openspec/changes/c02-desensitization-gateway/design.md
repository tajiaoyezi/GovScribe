## Context

GovScribe 的公网提质路径（RAG + 提示工程，见 ADR-0001 D4）必然把客户公文片段拼进 prompt 送往 OpenAI / Anthropic 等公网模型，这与体制内单位的保密红线正面冲突：真实公文里的人名 / 单位 / 文号 / 金额 / 项目代号一旦原文出域，代价极高（PRD Risks「保密红线」）。仅靠提示约束或事后审查都不可靠，必须在调用链上插入一道强制、可审计、按密级生效的拦截关口。

当前状态：c01 已落地模型接入调用链与公网 / 私有供应商的窄抽象（ADR-0001 D3，MVP 经 LiteLLM Proxy 统一 `base_url / api_key / model`）。`openspec/specs/` 仍为空，本 change 不修改既有 capability 的 spec 级行为，只新增三个 capability。

约束（均锚定 ADR-0001，不在本产物另立第二套口径）：

- 重 ML 算子不进 Go 进程：中文 NER 一律外置 Python 微服务（D1 边界、D7）。
- 映射表 / 原文 / diff / 审计全部留在 Go 网关进程内；外置 NER 无状态、只收文本片段、只回实体 span，绝不接触映射表与原文落盘（D7 铁律）。
- PostgreSQL 为权威源 + 访问控制判定（D5）；脱敏字典、密级路由配置、脱敏审计日志均落 Postgres。
- RBAC 与密级两层正交叠加（D10 边界）：脱敏库维护入口受 RBAC 约束，密级决定能外发什么数据。
- MVP 单租户私有部署（D10），暂不引入多租户复杂度。
- 关键失败点是外置 NER：它是唯一会"挂"的环节，必须默认 fail-closed。

利益相关方：文秘 / 业务兼职用户（被网关无感拦截）、系统管理员（维护脱敏库与密级路由配置）、审计员（只读复核出公网留痕）、客户合规 / 保密部门（拍板密级策略，PRD Open Questions）。

定位声明：本 change 是非密敏感数据出域的风险缓解手段，不是"涉密上公网"的合规承诺；残余风险（脱敏漏识别）由客户合规部门知悉并接受，故涉密内容恒不走公网。

## Goals / Non-Goals

**Goals:**

- 在 c01 公网分支上插入唯一、不可旁路的拦截点：原文离开 Go 进程前必经网关处置（放行 / 改写 / 改道私有 / 阻断）。
- 落地三层识别流水线（正则 + 字典 Aho-Corasick + 外置 NER 兜底），其中正则层与字典层在 Go 进程内运行、不依赖 NER 可用性。
- 类型化占位替换 + 本地回填，保留单次请求内的指代一致性，映射表绝不出域。
- SSE 流式输出用尾缓冲处理跨 chunk 截断的占位符，杜绝半截占位符输出与原值泄露。
- fail-closed 失败模式 + 按密级可配降级开关：处置优先级「转私有 → 降级 regex+字典 → 阻断」；涉密永不降级、恒走私有。
- 对外置 NER 接熔断器，故障时不逐请求干等超时拖死写作流。
- 按密级路由（非密 / 敏感脱敏 / 涉密强制私有），密级 → 模型映射不出域。
- 客户可维护脱敏库（Postgres），作为字典识别层与（c03）结构化精确召回的统一权威词条来源。
- 每次出公网或处置决策留审计（脱敏前后 diff + 命中项明细 + 处置事件），审计员只读。

**Non-Goals:**

- 不实现模型调用本身（属 c01；本 change 仅在其公网分支调用链上插入拦截点）。
- 不实现范文检索侧入库与精确召回字段写入（属 c03；c03 复用本 change 的抽取与词条能力）。
- 不交付私有化 / 信创部署形态本身（ADR-0001 D7、PRD Scope 后置）；本 change 只保证"涉密 / 高风险导向私有路径"的路由与 fail-closed 行为在调用链上成立，私有模型端点能力依赖 c01 提供。
- 不为 NER 模型选型下最终结论（HanLP / PP-UIE / GLiNER 实测后选，锚定 D7），本 change 只约定外置 NER 的接口契约与无状态边界。
- 不替客户制定密级策略（政策项，PRD Open Questions）；工程侧只保证策略"可配置、可强制执行、可审计"。

## Decisions

### D-1 网关切入点：在 c01 供应商窄抽象的公网分支前插一道拦截装饰器

**决策**：把网关实现为 c01 模型接入抽象（D3 窄抽象层）的一道前置拦截，包裹"目标解析为公网模型"的出站调用；私有 / 本地模型目标直接放行不强制脱敏（数据不出域，无需脱敏）。拦截点是 Go 进程内唯一的公网出站收口，不存在绕过网关直达公网的旁路。

**为何这样而非别的**：

- 备选 A「在 LiteLLM Proxy 侧做脱敏」被否：映射表 / 原文 / diff 会落到 Go 进程之外，违反 D7 铁律；且 D3 规划信创阶段切 Go 直连官方 SDK，Proxy 侧逻辑届时作废、业务代码反而要动。
- 备选 B「在每个业务模块各自调用脱敏」被否：无法保证"唯一、不可旁路"，任一模块漏调即泄露；拦截点应收口在调用链而非分散到调用方。
- 选 A' 装饰 c01 抽象：拦截点随调用链统一收口，信创切 SDK 时网关逻辑不动（与 D1 / D3 窄抽象同思路）。

**影响**：c01 的公网供应商必须经此装饰链出站；密级 → 路由与 fail-closed 转私有都在该装饰内完成目标重解析。

### D-2 三层识别流水线与命中合并

**决策**：对待出域文本顺序执行三层，结果合并为统一命中项集合（span：起止 offset、类型、来源层）后进入占位替换：

1. **正则层（Go 进程内）**：格式化实体——文号（如 `〔2024〕X号`，含六角括号 / 方括号变体）、金额、身份证号、统一社会信用代码等证件号。
2. **字典层（Go 进程内，Aho-Corasick 多模匹配）**：客户脱敏库词条——单位 / 人员名册、项目代号、涉密关键词黑名单。AC 实现封装为可替换接口，候选 `petar-dambovaliev/aho-corasick`（锁 commit）或保守 `cloudflare/ahocorasick`（锚定 D7）。
3. **外置 NER 层（Python 微服务兜底）**：抓字典未覆盖的新实体（人名 / 地名 / 机构等）；无状态、只收文本片段、只回 span（D7 铁律）。

合并规则：按 offset 区间合并，重叠时优先级 涉密关键词黑名单 > 正则格式化实体 > 字典其它 > NER（黑名单与格式化实体置信度最高且与处置 / 路由强相关）；嵌套 / 包含取较长 span。

**为何分层而非单一手段**：正则对格式化实体零依赖、零误漏；字典让客户掌握自己名册的权威召回且与 c03 共用词条；NER 只承担"未知新实体"的长尾兜底，把唯一会挂的环节限定到一层，便于 fail-closed 仅围绕该层设计。

**影响**：正则层与字典层不随 NER 宕机失效——NER 不可用时降级路径仍有这两层兜底（见 D-5）。

### D-3 类型化占位符设计与本地回填

**决策**：命中实体替换为类型化占位符，形如 `〖PERSON_01〗`（类型 + 单次请求内自增序号）。设计约束：

- 同一原值在单次请求内映射到同一占位符（指代一致性），避免脱敏过度丢失单位口吻 / 称谓（PRD Risks「脱敏过度」缓解）。
- 边界标记 `〖…〗` 选用公文正文中几乎不出现的成对全角符号，且与文号常用的六角括号 `〔〕` 区分，降低与正文及模型自由发挥产生的伪占位符碰撞概率；占位符整体便于完整匹配与回填（与 D-4 尾缓冲算法配合）。
- 映射表为「占位符 ↔ 原值 + 类型 + 命中来源」的单次请求内存结构，模型返回后在本地按映射表回填原值；映射表绝不出域、不落盘到 NER（D7）。

**为何类型化而非掩码 / 删除**：类型化保留语义骨架，模型能据"某单位 / 某人名"生成得体公文口吻；掩码 / 删除会破坏语法与指代，加剧脱敏过度风险。

**影响**：回填在 Go 进程内完成，下游（含 OnlyOffice 在线编辑、c03 入库）拿到的是已回填原文，与三库的密级 ACL 一致（原文从不以脱敏态进入权威源）。

### D-4 流式回填尾缓冲（tail-buffer）

**决策**：SSE 流式输出（D2、SSE）下，逐 chunk 回填时维护一个尾缓冲：

- 每个 chunk 到达，先与缓冲拼接，再扫描完整占位符并回填、输出已确定安全的前缀；
- chunk 末尾若是"占位符前缀的可能匹配"（出现起始标记 `〖` 但未见闭合 `〗`，或部分类型 token），暂缓输出该尾部、留待与后续 chunk 拼接再判定；
- 缓冲长度上界设为占位符最大可能长度，防止恶意 / 异常长尾把内存拖大；
- 流结束时对尾缓冲做最终回填并输出，不丢弃缓冲内容。

**为何需要**：占位符被流式切分到相邻两个 chunk 时，若逐 chunk 直接回填会出现"半截占位符无法匹配"，轻则回填失败、重则把占位符的明文片段当原文输出造成泄露。尾缓冲把回填判定窗口从"单 chunk"扩到"跨 chunk 拼接后"。

**备选**：缓冲到全文结束再统一回填——被否，破坏流式体验、首字延迟过大。尾缓冲只缓冲疑似占位符尾部，正常文本即时透传。

### D-5 fail-closed 处置状态机 + 按密级降级开关

**决策**：以"脱敏是否完成"为状态机输入，默认 fail-closed。对"NER 不可用"判定从宽——超时 / 连接失败 / 5xx / 响应格式异常（HTTP 200 但 span 格式不符约定）均算"脱敏未完成"，不只判进程死活。处置优先级：

1. **转私有 / 本地模型**：存在可用私有模型则自动改道（重解析 D-1 的出站目标），不向公网发送，记审计"转私有"。
2. **降级发送**：无私有可用且该非涉密密级显式开启降级开关 → 以 regex + 字典结果脱敏后发公网、跳过 NER 兜底，记审计"降级发送"。MVP 公网阶段（无私有模型）降级即主回退路径。
3. **阻断**：仍不满足 → 阻断公网调用并向调用方返回明确错误，绝不发原文。

**铁律**：涉密恒走私有、永不降级，即使降级开关被开启也不对涉密生效；降级开关默认关闭（fail-closed），仅客户为某非涉密密级显式开启后方对该密级生效。

**为何 fail-open 必须显式 opt-in**：NER 兜底正是抓字典未覆盖的新实体，它挂的时刻恰是最难识别的敏感信息最易泄露之时；默认放行等于在最危险时刻泄露，违反保密红线。把 fail-open 收紧为"按密级显式开启 + 涉密强制排除"，让残余风险由客户合规部门明确承接（PRD Open Questions 密级策略）。

### D-6 外置 NER 熔断器

**决策**：对外置 NER 调用接熔断器（半开探测）：短时间内连续失败达阈值则打开，打开期间后续请求不再逐个等超时、直接进入 D-5 的 fail-closed 回退路径；半开探测成功后闭合恢复。

**为何**：NER 单次超时只拖慢一个请求，但故障期逐请求干等会把写作流整体拖死；熔断把"故障期每请求各等一个超时"压成"快速失败 + 周期性探测"。阈值 / 超时 / 半开窗口设为可配（私有部署环境差异大），默认值在实施期据公文域调用量定档。

### D-7 密级路由决策器

**决策**：路由决策器依据数据密级在 D-1 拦截点内决定出站目标：非密直接走公网（仍过三层识别处置，但不强制改道）；敏感先经网关脱敏后走公网；涉密强制私有，禁止任何配置开关使其走公网或脱敏后公网。密级 → 模型路由映射与降级开关落 Postgres、可由系统管理员维护、不出 Go 进程边界。

与网关协同（防越权关键）：

- 网关因 fail-closed 触发"转私有"或密级判定为涉密 → 路由导向不出域私有路径；
- NER 不可用且无私有可用 → 路由与网关共同保证非降级路径默认阻断而非放行原文；
- 涉密在 NER 不可用时仍走私有，不因 NER 状态改变涉密的私有路由结论。

**密级数据来源**：网关读取 c01 窄抽象调用承载的**『出站请求内容密级』字段（取值：非密 / 敏感 / 涉密；允许缺省为「缺失/未知」）**——该密级由写作发起方在发起时确定（c06 结构化场景上下文契约的内容密级项），随场景上下文经 c05 / c07 由 c01 窄抽象在出站调用上原样承载该字段透出给本 change 拦截点（c01 不解读、不判定、不改写），与 D10「密级=Milvus partition key、Postgres 预过滤为权威 ACL」同一密级体系；具体定级权限与策略归客户密级策略，本 change 不在网关内另立密级判定口径。当该字段缺失或未知（c01 如实呈现「缺失/未知」、不补默认值）时，网关必须按涉密 fail-closed 处置——强制转私有，无可用私有则阻断，涉密永不降级，并入审计；不得缺省按非密放行公网。

### D-8 脱敏库作为统一权威词条来源

**决策**：脱敏库（单位 / 人员名册、项目代号、涉密关键词黑名单）落 Postgres，词条区分类型以便字典层生成对应类型化占位符；维护入口受 RBAC 约束，仅系统管理员（D10 角色）可增删。RBAC 判定不在本 change 自建——脱敏库维护入口消费 c04 `rbac-authorization` 的统一授权决策接口，按 `dict.manage` 权限点校验（c04 为该权限点的登记中心与判定权威），本 change 只作为消费方在维护入口前置该校验。同一套词条同时服务：① 网关字典层 Aho-Corasick 匹配；② c03 文号 / 单位名结构化精确召回字段抽取（锚定 ADR-0001 §架构「文号 / 单位名精确召回」与 D5）。不在两处各自维护以免口径漂移。

**为何统一来源**：D7 与架构注记明确要求脱敏识别与精确召回复用同一词条；分两套会导致"脱敏识别到、精确召回漏抽"或反之，且增加客户维护负担。

**词条更新生效**：增删后网关后续识别按更新后词条集合生效；AC 自动机重建采用副本切换（构建新自动机后原子替换指针），避免重建期阻塞在途请求。

### D-9 出公网审计留痕

**决策**：本 change 拥有的审计表为「脱敏审计表」，仅承载脱敏出公网留痕——每次涉及公网调用或处置决策落审计到 Postgres——脱敏前后 diff、命中项明细（类型 / 来源层 / 位置）、降级 / 阻断 / 转私有处置事件类型与触发原因；路由 / 降级配置变更也可被审计。账号 / 角色 / 认证 / 授权 / 登录失败类审计不属本表，归 c04「账号安全审计表」，本 change 不承载该类事件。审计供合规复核，审计员只读——「审计员」角色及其只读授权（`audit.read` 权限点）由 c04 `rbac-authorization` 登记与判定，本 change 的审计查询入口消费该决策、不自建角色或授权。审计数据与映射表 / 原文同样不出 Go 进程边界对外暴露。

**diff 自身的密级风险权衡**：脱敏前 diff 含原文敏感片段，脱敏审计表与原文同密级管控、仅审计员只读、不外发；这是"可复核"与"留痕即扩大原文存储面"的取舍——选择留痕以满足合规可追溯，代价是审计表纳入密级 ACL 与访问最小化（见 Risks）。该「同等密级 ACL」判定由 c04 `access-control-composition` 提供（对接 Postgres 预过滤为权威 ACL，ADR-0001 §2 / D10），本 change 不自建数据 ACL 密级判定，仅声明本表纳入该 ACL 管控。

## Risks / Trade-offs

- **[NER 漏识别长尾新实体导致脱敏不全]** → 三层互补（正则 / 字典零依赖兜底常见与已知实体），NER 仅承担长尾；占位替换 + 出公网审计 diff 留痕便于事后发现漏识别模式并补字典；定位明确为风险缓解非合规承诺，涉密恒不走公网。
- **[降级开关被误开成默认放行（fail-open）]** → 开关默认关闭且对涉密强制无效；开启需系统管理员显式配置并入审计；MVP 阶段把降级定位为"无私有时的回退"而非常态。
- **[占位符与公文正文 / 模型自由发挥产生伪占位符碰撞]** → 选用正文罕见的成对全角边界符并与文号括号区分；回填只认本次请求映射表内登记的占位符，未登记的相似串不回填，避免误把模型生成内容当占位符还原。
- **[流式尾缓冲增加首字延迟 / 缓冲膨胀]** → 只缓冲疑似占位符尾部、正常文本即时透传；缓冲长度上界 = 占位符最大长度，异常超长直接刷出。
- **[熔断打开期把本可成功的请求也快速失败]** → 半开探测周期性放行试探，NER 恢复即闭合；阈值 / 窗口可配以适配不同私有部署负载。
- **[脱敏审计 diff 扩大原文敏感数据存储面]** → 审计表纳入与原文同等密级 ACL（D5 Postgres 权威），仅审计员只读、不出 Go 进程边界、不外发；以"可复核"换"存储面扩大"，并以访问最小化收敛风险。
- **[脱敏网关影响]** → 本 change 即脱敏网关本体：把"非密敏感数据出域"风险降到工程可控范围，代价是公网调用链增加一段识别 + 占位 + 回填开销（正则 / 字典为进程内、毫秒级；NER 为网络调用、受熔断保护）。
- **[密级防越权影响]** → 路由决策器在出站收口处按密级强制改道，与 D10「Postgres 预过滤为权威 ACL、密级与 RBAC 正交叠加」一致；涉密永不走公网为硬约束，不受任何配置开关左右，构成纵深防御的"外发出口"一层。
- **[三库（Postgres / Milvus / MinIO）一致性影响]** → 本 change 只新增 Postgres 表（脱敏字典、密级路由配置、脱敏审计日志），不直接写 Milvus / MinIO；下游拿到的是已回填原文，原文从不以脱敏态进入权威源或派生索引，故不引入"脱敏态 / 原文态"双形态一致性问题；与 c03 的耦合仅为"共用 Postgres 词条来源"，词条更新由 D-8 的 AC 副本切换保证网关侧即时一致，Milvus 派生索引的精确召回字段由 c03 在入库时据同一词条抽取（必要时可从 Postgres + MinIO 全量重建，锚定 D5 / 架构注记）。

## Migration Plan

本 change 为新增能力，无既有行为迁移；按"先识别、再处置、后审计"的依赖顺序分步上线，每步可独立验证、可回退：

1. **脱敏库 + Postgres 表落地**（D-8）：建脱敏字典 / 密级路由配置 / 脱敏审计日志表，开放 RBAC 受限的脱敏库维护 API。→ 验证：管理员可预导入 / 增删词条并按类型落库；非授权角色被 RBAC 拒绝。
2. **Go 进程内识别（正则 + 字典 AC）+ 占位 / 回填**（D-2 / D-3）：先上线不依赖 NER 的两层与回填。→ 验证：文号 / 金额 / 字典词条命中并类型化占位、本地回填、指代一致。
3. **拦截点接入 c01 公网分支 + 密级路由**（D-1 / D-7）：装饰 c01 出站抽象，按密级决定路径。→ 验证：公网调用必经网关；涉密强制私有、敏感脱敏后公网、非密直走。
4. **外置 NER 兜底 + 熔断 + fail-closed 状态机**（D-2 第三层 / D-5 / D-6）：接入 NER 微服务与熔断、落地处置优先级。→ 验证：NER 超时 / 5xx / 格式异常均触发 fail-closed 处置；涉密永不降级；熔断打开期不逐请求干等。
5. **流式尾缓冲**（D-4）：SSE 路径接入尾缓冲回填。→ 验证：占位符跨 chunk 截断不泄露、流结束刷新缓冲。
6. **出公网审计留痕**（D-9）：接齐 diff + 命中项 + 处置事件留痕。→ 验证：脱敏出公网与各处置事件均可被审计员只读查询。

**回退策略**：拦截点为装饰链，可整体下线回退到 c01 原始公网调用——但这会失去脱敏保护，故回退仅限非生产 / 无真实公文环境；生产环境的安全回退路径是把对应密级降级开关关闭（回到阻断 fail-closed），而非摘除网关。外置 NER 故障的"回退"由熔断 + D-5 状态机在运行时自动处理，无需人工下线。

## Open Questions

- **密级策略（哪些密级 / 类型允许脱敏后上公网、哪些强制私有化）**：客户合规 / 保密部门政策决定（PRD Open Questions、ADR-0001 待决项），工程侧按其策略配置降级开关与路由表，不阻塞 MVP 开工；客户须明确承接脱敏漏识别的残余风险。
- **外置 NER 选型（HanLP v2.1.x / PP-UIE / GLiNER）**：公文域召回实测后定（锚定 D7），本 change 只锁定接口契约（收文本片段、回 span、无状态）；选型结果不改变本 design 的状态机与边界。
- **熔断 / NER 超时阈值默认值**：依公文域真实调用量与私有部署负载在实施期定档（D-6），先给保守默认、上线后据审计与监控数据调参。
- **占位符边界符的最终字符集**：`〖…〗` 为初定，须在 PoC 用真实公文样本回归确认与正文 / 模型输出的碰撞率，必要时调整（与 D7 提到的 `sugarme/tokenizer` 多字节 offset 回归一并验证 offset 切割正确性）。
