## ADDED Requirements

### Requirement: 客户可维护脱敏库管理

系统应当（SHALL）提供客户可维护脱敏库，支持单位 / 人员名册、项目代号、涉密关键词黑名单的预导入与增删管理；词条落 PostgreSQL，作为脱敏网关字典识别层与（c03）文号 / 单位名结构化精确召回的统一权威词条来源。词条应当区分类型（单位 / 人员 / 项目代号 / 关键词黑名单等），以便字典层据类型生成对应类型化占位符。脱敏库维护入口必须受 RBAC 约束，仅授权角色（系统管理员）可增删词条；RBAC 校验消费 c04 `rbac-authorization` 的授权决策接口（`dict.manage` 权限点），本能力不自建 RBAC 判定。

#### Scenario: 预导入词条
- **WHEN** 系统管理员批量预导入一份单位 / 人员名册
- **THEN** 词条必须按类型落 PostgreSQL，并可被脱敏网关字典层用于 Aho-Corasick 匹配

#### Scenario: 增删词条即时生效
- **WHEN** 系统管理员新增或删除一条字典词条
- **THEN** 脱敏网关后续识别必须按更新后的词条集合生效

#### Scenario: 非授权角色不可维护
- **WHEN** 非授权角色尝试调用脱敏库维护入口
- **THEN** 系统必须依据 RBAC 拒绝该操作

### Requirement: 词条作为统一权威来源

脱敏库词条必须（MUST）作为脱敏识别（字典层 Aho-Corasick 匹配）与下游结构化精确召回（c03 复用文号 / 单位名抽取）的同一套权威来源，不得在两处各自维护造成口径漂移。词条来源应当一致，使脱敏命中与精确召回字段抽取使用相同的单位 / 项目代号定义。

#### Scenario: 脱敏与精确召回复用同一词条
- **WHEN** 一个单位名既需在脱敏时被识别替换、又需在 c03 入库时作为精确召回标量字段抽取
- **THEN** 两处必须引用脱敏库中的同一词条来源，不得各自独立维护

#### Scenario: 涉密关键词黑名单驱动识别
- **WHEN** 文本命中脱敏库中的涉密关键词黑名单词条
- **THEN** 该命中必须进入网关命中项集合并参与处置决策（黑名单命中作最高优先级命中项参与脱敏处置；密级仍以上游传入为准，路由不由黑名单改判密级）
