## Context

GovScribe 公网线靠"范文知识库 + RAG + 提示工程"提质而非微调（PRD「公网提质路线」、ADR-0001 D4）。要让初稿"像公文、可直接采纳"，c05 / c07 生成编排侧（c05 必选、c07 可选）在起草时必须能从范文库稳定召回同类优质样例喂给模型、其生成最终经 c01 窄抽象接口出站。这同时要求两类召回：

- **语义召回**：找"内容相近"的范文（写作意图 → 同类范文）。
- **精确召回**：按文号（如 `〔2024〕X号`）、发文单位名命中特定公文——这类标识在中文分词器下会被切碎，无法靠 BM25 / `TEXT_MATCH` 稳定命中。

当前项目尚无任何检索能力：没有范文入库管线、没有向量库、没有 embedding / rerank 接入。前序 c01 已具备模型接入与 SSE 流式生成的窄抽象（唯一模型调用入口），c02 已具备脱敏网关与词条抽取能力（正则 + AC 字典 + 外置 NER，fail-closed）、可对范文脱敏入库并提供 `desensitization-dictionary` 作为文号 / 单位名抽取的统一权威词条来源。本 change 在 c01 与 c02 之间补齐"建库 + 检索"，让 RAG 提质路线跑通——否则"库空 → 检索不到样例 → 退回通用模型不像公文"，PRD 的价值假设无从验证（PRD Open Questions「范文库数据不足」风险）。

约束（均锚定 ADR-0001，不另立口径）：

- 向量库锁 Milvus 2.6.x（server 如 2.6.18，Apache-2.0），Go 客户端锁 `milvus/client/v2@v2.6.5`（import `milvusclient`），与 server 大版本对齐，不上 3.0-beta（ADR-0001 §3、D4）。该 client module 的 `GoVersion` 为 1.25.8，故仓库 Go 基线为 1.25.8；`v2.6.0` 虽为 Go 1.24.4，但其配套依赖在 Windows 下编译失败，暂不采用。
- Milvus 无法嵌进 Go 单二进制（Lite 仅 Python / FLAT-only / 不可生产），始终是独立进程 + etcd + 内置 MinIO 后端；"单二进制"仅对 Go 应用层成立（ADR-0001 D4）。
- 三库切分、跨库最终一致、密级防越权三条架构口径全部来自 ADR-0001 §2；本 change 实现它们，不重新定义。
- 权威 ACL 判定属 c04（Postgres RBAC + 密级），本 change 只消费其判定结果做预过滤；脱敏识别算法本身属 c02，本 change 只消费其抽取产物；模型生成正文属 c05。

利益相关方：文秘 / 办公室综合人员（建库、补充、采纳回流的操作者）、c05 / c07 生成编排侧（c05 必选、c07 可选；检索能力的调用方，其生成最终经 c01 窄抽象接口出站）、运维（索引重建与对账）、客户保密 / 合规部门（密级与脱敏策略来源）。

## Goals / Non-Goals

**Goals:**

- 建立**范文知识库写入侧**：客户存量公文初次批量建库（可断点续传、幂等、单篇失败不阻断整批）、日常补充入库、采纳稿回流入库；范文经 c02 脱敏后落地三库。
- 建立**范文知识库读取侧**：基于 Milvus 2.6.x 原生 BM25 + 稠密向量混合检索（`HybridSearch` + `RRF`），对外提供"给定写作意图召回同类范文"的能力，供 c05 / c07 生成编排侧（c05 必选、c07 可选）取样例做 RAG 提示拼装、其生成最终经 c01 窄抽象接口出站；语义召回与文号 / 单位名精确召回并行，候选结果可经 rerank 精排。
- 把文号 / 单位名抽取为 Milvus **不切词标量 VARCHAR 字段** + Postgres 列，做精确等值召回，规避中文 analyzer 切碎标识的问题。
- 将 embedding / rerank **服务化接入**为窄抽象（OpenAI 兼容 embedding 锁 `EncodingFormat="float"`；自写 `/v1/rerank` 薄 client），为信创阶段切换国产推理（如 mis-tei）预留替换点。
- 落实**三库分工**（Postgres 权威源 + ACL / Milvus 可重建派生索引 / MinIO 双桶）、**跨库最终一致**（outbox + 软删 + 对账）、**密级防越权**（Postgres 预过滤为权威 ACL，Milvus partition / 标量过滤为纵深）三条架构口径。

**Non-Goals:**

- 不实现脱敏识别算法本身（复用 c02 网关与 `desensitization-dictionary` 词条来源）。
- 不实现权威 ACL / RBAC / 密级判定逻辑（属 c04 Postgres）；本 change 只消费其判定结果注入预过滤条件。
- 不实现模型生成正文与 RAG 提示模板设计（属 c05 / c07 生成编排侧，c05 必选、c07 可选；其生成最终经 c01 窄抽象接口出站）；本 change 只把范文样例 + 元数据交付给调用方。
- 不上 Milvus 3.0-beta 的 jieba 文件式大词典（`FileResource`）；海量单位名录不靠大词典分词，走结构化标量字段（ADR-0001 D4）。
- 不做向量库微调 / 重排序模型训练；公网线不微调（PRD）。
- 不做信创 / 私有化部署形态本身（后置）；本 change 只保证替换点存在、托管与私有化可平滑切换。

## Decisions

### D-1 向量库与混合检索：Milvus 2.6.x 原生 `HybridSearch` + `RRF`（锚 ADR-0001 D4）

一个引擎覆盖"语义召回 + 精确词召回"。检索以 `HybridSearch` 下发两路 `NewAnnRequest`：

- 稠密路：`entity.FloatVector`，向量由 vector-embedding-service 生成，召回内容相近的范文。
- 稀疏路：BM25 走 `entity.Text`，由 Milvus 内置 `chinese` analyzer 承担分词，召回关键词相关的范文。

两路以 `NewRRFReranker()`（RRF）融合排序起步。单路命中为空时仍以 RRF 返回另一路结果，不因单路空而整体返回空。

- **为何 Milvus 而非自建双索引（如 Postgres pgvector + Elasticsearch）**：Milvus 原生混合检索一个引擎覆盖两路并自带 RRF，避免维护两套索引与跨引擎融合逻辑，符合"简单优先"。pgvector 不具备生产级稀疏 + 稠密融合；ES 引入第二套全文引擎与运维负担。
- **为何锁 2.6.x 不上 3.0-beta**：3.0 仍 beta，且其唯一对本 change 有吸引力的特性（jieba 文件式大词典 `FileResource`）已被结构化标量字段方案替代（见 D-3），不构成升级理由。Go 客户端锁 `milvus/client/v2`（import `milvusclient` / `entity` / `index`），旧 `milvus-sdk-go` 已废弃（ADR-0001 §3、§6）。
- **代价**：Milvus 独立进程 + etcd + 内置 MinIO 后端（2.6 用内置 Woodpecker 去掉 Pulsar / Kafka），无法进 Go 单二进制，运维多一组件。接受——这是混合检索能力的必要成本。

### D-2 embedding / rerank 服务化窄抽象（锚 ADR-0001 D6、D1）

重 ML 算子不进 Go 进程，一律服务化外置。embedding 经 OpenAI 兼容端点调用（优先中文模型，BGE 系），rerank 走 Cohere / Jina 风格 `/v1/rerank`。Go 侧封装为 `vector-embedding-service` 窄抽象，上层（corpus-ingestion / hybrid-retrieval）只面向接口编程。

- **embedding 显式设 `EncodingFormat="float"`**：`github.com/openai/openai-go/v3`（导入路径必须带 `/v3`）对 base64 响应反序列化有未修复缺陷（#241）；显式设 float 规避（ADR-0001 D6）。
- **rerank 自写约 20 行薄 client**：OpenAI 规范无 rerank，无法复用官方 SDK；自写 client 接收查询 + 候选列表、返回带相关性得分的重排序结果，统一超时控制，超时 / 连接失败 / 5xx / 响应格式异常一律判为"不可用"，由调用方降级（ADR-0001 D6）。
- **建库侧与查询侧必须同 embedding 模型 / 维度**：否则查询向量与库内向量不在同一空间，召回失真。模型名 / 维度 / 端点 / 鉴权经配置注入，不硬编码。
- **替换点**：信创阶段昇腾走 mis-tei（rerank 不走 vLLM-Ascend）、鲲鹏 CPU 走 llama.cpp GGUF 兜底（ADR-0001 D6、§4）；切换时仅改配置，上层业务代码不动。

### D-3 文号 / 单位名精确召回走结构化标量字段，不靠分词（锚 ADR-0001 §2、D4、§6）

入库时复用 c02 脱敏网关的正则 + AC 字典能力，从范文正文抽取文号与发文单位名，整体（不切词）写入 Milvus 标量 VARCHAR（keyword）字段与 Postgres 对应列，检索时做等值匹配。

- **为何不用 BM25 / `TEXT_MATCH` / jieba**：Milvus `TEXT_MATCH` 与 jieba 同 analyzer，`〔2024〕X号` 会被切碎导致漏召（ADR-0001 §6 被核验纠正的认知）。精确召回必须靠不经 analyzer 的结构化标量字段。
- **为何不引 jieba 文件式大词典装单位名录**：该特性 Milvus 3.0-beta 才有，2.5 / 2.6 只能内联进 schema，海量单位名录不可行（ADR-0001 D4）；标量字段方案在 2.6.x 即可落地。
- **词条来源**：复用 c02 `desensitization-dictionary`（单位 / 人员名册、项目代号），抽取产物即标量字段值；本 change 不实现识别算法，只消费产物。开工前需用真实公文跑 Milvus `run_analyzer` 验证（ADR-0001 §4 PoC）。

**脱敏边界：正文敏感实体脱敏入库 vs 标识字段明文精确召回（消除自相矛盾，锚 ADR-0001 §2）**

"逐篇脱敏后落地"与"文号 / 单位名明文写 Milvus 标量做精确等值召回"看似矛盾，实为两类不同对象、两种不同处置：

- **范文正文中的人名 / 证件号 / 金额等敏感实体**：脱敏（占位替换）后入库，不以明文承载。
- **文号、发文单位名等"标识性精确召回字段"**：在客户密级策略下通常为公开信息 → 默认以明文写入 Milvus 标量 VARCHAR 字段做精确等值召回（对齐 ADR-0001 §2"不靠 jieba 切分、用标量精确等值"）；其在 Postgres 权威源亦以明文存储。
- **"脱敏"针对的是"检索样例出公网喂模型前"这一出域动作（走 c02），而非本地入库本身**：私有部署的本地存储不属"出域"，因此标识字段明文落本地库与脱敏边界不冲突。
- **客户密级策略覆盖时的降级**：若客户密级策略判定文号 / 单位名亦敏感，则降级为"明文仅落 Postgres 权威源（受 ACL）、Milvus 标量存确定性哈希 / 占位"，精确召回先在 Postgres 命中再回 `chunk_id` 取 Milvus 派生数据。
- **残余风险与依赖**：标识字段是否可明文取决于客户密级策略，此为对外部策略的依赖项；策略收紧时按上条降级，相应增加一次 Postgres 回表的检索开销。

### D-4 三库分工与可全量重建（锚 ADR-0001 §2、D5）

- **Postgres = 权威源 + ACL**：文档元数据、全局 `chunk_id` 主键、密级 / 文种 / 文号 / 单位名字段、采纳稿回流记录、outbox 表、软删 `is_deleted` 与审计双时间戳。密级 / 文种 / 文号 / 单位名列与 c04 权威 ACL 表对接，本 change 不重定义 ACL 判定。
- **Milvus = 可重建的派生混合索引**：密级 = partition key、文种 = 标量过滤字段、文号 / 单位名 = 不切词标量字段、稠密向量 + BM25 稀疏。必须可在丢失后从 Postgres + MinIO 全量重建，重建结果与权威源一致。
- **MinIO = 双桶**：范文原件落业务范文桶，与 Milvus 自身后端桶**物理隔离**，不得共用同一桶。
- **为何 Milvus 设为派生而非权威**：向量与索引是可由原文 + 元数据再生的产物，设为派生使其可重建、可降密重整、可随 embedding 模型升级重灌，避免把不可再生数据锁死在向量库。

### D-5 跨库最终一致：outbox + 软删 + 对账（锚 ADR-0001 §2、D5）

无跨库事务，故"关系库先落地 + outbox 异步管线建索引 + 软删 + 定期对账 job"。

- 写入先在 Postgres 落地并写 outbox 事件，再由异步管线消费驱动 Milvus 建索引；建索引失败事件可重试不丢失。
- 删除走软删（`is_deleted=true`）并经 outbox 同步 Milvus，不物理直删导致索引漂移；查询层强制 `is_deleted==false` 兜底覆盖删除可见性延迟。
- 定期对账 job 比对 Postgres 与 Milvus 的 `chunk_id` 集合（含密级 / 文种 / 软删状态），检出缺失 / 多余 / 不一致项并重新触发建索引或告警。
- **为何不追求强一致**：跨库 2PC 复杂且脆弱；范文检索对短暂索引可见性延迟容忍度高（补充入库"立即可命中"以 outbox 延迟为约束即可），故选最终一致 + 对账兜底，符合"简单优先"。`chunk_id` 全局主键是两库对账与重建的锚。

### D-6 密级防越权：Postgres 预过滤为权威 ACL，Milvus 为纵深（锚 ADR-0001 §2，保密红线相关）

- **Postgres 预过滤为唯一权威 ACL**：检索结果以 c04 在 Postgres 的密级 / 权限判定预过滤，确保不返回当前用户不可见的范文；即便 Milvus partition 过滤被绕过也不返回越级范文。
- **Milvus partition（密级）+ 标量过滤（文种）仅作性能与纵深**，不得作为唯一访问控制依据。
- **过滤条件由后端注入，前端不可传任意过滤表达式**：携带前端自定义表达式的请求一律忽略，仅用后端按用户密级 / 权限注入的条件。
- **partition key 不可变 → 降密走 delete + reinsert**，不就地改密级。
- **缺失密级拒绝入库**：建库 / 补充 / 回流任一入口缺密级标签的范文一律拒绝，不得按非密默认放行；回流稿未能继承或确认密级时阻断（与 c02"涉密永不降级"红线同向）。
- **为何双层而非只信 Milvus 过滤**：向量库过滤是性能优化，一旦表达式注入或 partition 配置出错即越权；把权威判定留在关系库 + 后端注入，是保密红线下的纵深防御。

### D-7 对 c02 / c04 / c05 / c07 的依赖与契约边界

- **复用 c02**：脱敏入库（涉密恒走私有、永不降级，fail-closed——单篇脱敏未完成则跳过该篇不写任何库，记入失败清单）、文号 / 单位名抽取（正则 + AC 字典 + `desensitization-dictionary` 词条）。本 change 是 c02 抽取产物的消费方。
- **复用 c04（两层正交叠加、各自落地，不可相互替代）**：本 change 对 c04 的复用分两层独立落地——
  - **数据层（密级数据 ACL 预过滤）**：检索召回以 c04 在 Postgres 的密级 / 权限判定做权威 ACL 预过滤（见 D-6），裁决"能召回 / 能外发哪些范文"，本 change 不重定义 ACL 判定。
  - **操作层（RBAC 操作权限点）**：范文检索入口消费 c04 `rbac-authorization`（`template.search` 权限点）、范文入库（初次建库 / 补充）入口消费 c04 `rbac-authorization`（`template.ingest` 权限点）裁决"能否发起该操作"，未授予即拒绝（fail-closed），本 change 不自建 RBAC 判定。
  - 二者正交叠加（操作层 RBAC ⊥ 数据层密级 ACL）、不可相互替代，任一层拒绝即整体拒绝；采纳稿回流走 c08 约定 outbox / 事件表单点交付、不经检索 / 入库人工入口，其操作权限边界由 c08 侧持有。
- **被 c05 / c07 生成编排侧依赖**：c05 / c07 生成编排侧（c05 必选、c07 可选）调用本 change 的混合检索 API 取范文样例做 RAG 提示拼装，其生成最终经 c01 窄抽象接口出站；本 change 只交付样例 + 元数据，不做提示模板与正文生成。
- **采纳稿回流**：采纳决策由 c08 持有，c08 对计入采纳（采纳结论为"直接用 / 小改"）的成稿产出回流信号，写入 c08 与本 change 约定的同一 outbox / 事件表（不直调本 change API，避免双写）。回流信号字段固定为 { 成稿引用（MinIO 对象键 / chunk 标识）、采纳结论（直接用 / 小改 / 大改 / 弃用；"直接用 + 小改"计入采纳）、密级、文种 }。本 change 单点消费该 outbox / 事件表 → 经同一脱敏 + 抽取 + 三库管线沉淀为新范文 + 对账，Postgres 记录回流来源以便审计对账。回流信号必带密级（由 c08 信号提供，来源为继承原生成任务密级或采纳 / 核稿时由文秘显式确认），缺失则阻断不入库，不再悬空。本 change 不提供"c05 / c07 生成编排侧调用回流 API"的入口。

## Risks / Trade-offs

- **范文库数据不足 / 未按时到位 → RAG 检索不到优质样例，初稿退回"不像公文"，UAT 说不清是模型还是缺数据问题**（PRD Open Questions 高风险）→ 缓解：开工前按文种盘点客户存量范文（对照 <100 条稀缺线，每文种宜 ≥100 篇）；`hybrid-retrieval` 向消费方（c05 / c07）返回"目标文种样例不足"标识，样例不足时仍带结构契约生成并标记由 c05 处置，是否走 c07 通用兜底以 c06 路由判定为单一权威，c03 不做文种降级决策、不承诺"深做"；约定数据到位时间与清洗 / 脱敏入库责任方。
- **密级越权召回（保密红线）**→ 缓解：D-6 双层防御（Postgres 预过滤权威 ACL + Milvus partition / 标量纵深 + 后端注入过滤 + `is_deleted==false` 兜底 + 降密 delete + reinsert）；残余风险落在 c04 ACL 判定正确性，本 change 以预过滤强制兜底。
- **脱敏漏识别导致敏感原文入库**→ 缓解：复用 c02 fail-closed 管线，脱敏未完成则该篇不入任何库；涉密恒走私有不入公网检索路径；残余风险（NER 漏识别）由 c02 持有，本 change 不降低其门槛（缺密级一律拒绝、不默认放行）。
- **embedding 模型 / 维度在建库与查询间不一致 → 召回失真**→ 缓解：两侧强制同模型同维度并经配置统一注入；升级 embedding 模型须全量重灌 Milvus 派生索引（D-4 可重建保证此操作可行）。
- **跨库最终一致带来短暂索引可见性延迟（补充 / 删除不即时生效）**→ 缓解：查询层 `is_deleted==false` 兜底防"软删未同步即被召回"；对账 job 定期修复漂移；接受"补充入库立即可命中"以 outbox 延迟为界，不承诺零延迟。
- **Milvus 派生索引丢失 / 损坏**→ 缓解：D-4 全量重建（Postgres 元数据 + MinIO 原件 → 重生成 chunk 向量 + 标量字段写回），重建后与权威源一致；MinIO 双桶物理隔离避免业务原件与 Milvus 后端互相污染。
- **`milvus/client/v2` 与 server 版本漂移 / openai-go 版本陷阱**→ 缓解：Go 客户端与 server 大版本对齐 2.6.x、锁版本；openai-go 锁 `/v3` 导入路径并显式 `EncodingFormat="float"`（D-2）；落地前以 `go list -m -versions` 与官方 releases 实时复核（ADR-0001 §3）。
- **信创阶段国产推理（昇腾 mis-tei / 鲲鹏 GGUF）吞吐 / 兼容不达标**→ 缓解：D-2 窄抽象使切换仅改配置；ARM64 / 龙芯 Milvus 与 BM25 中文分词须按 ADR-0001 §4 PoC 清单先验证（龙芯高风险、勿向客户承诺），不在本 change 落地，只保留替换点。

## Migration Plan

新建系统、无存量数据迁移，按依赖顺序分步交付（每步可独立验证、失败可回退到上一步）：

1. **基础设施就位**：部署 Milvus 2.6.x（独立进程 + etcd + 内置 MinIO 后端）、MinIO 业务范文桶（与 Milvus 后端桶物理隔离）、外置 embedding（中文优先）与 rerank 服务（OpenAI 兼容 / `/v1/rerank` 端点）→ 验证：`run_analyzer` 跑真实公文确认 BM25 中文分词与文号结构化字段兜底；embedding 返回 float 向量、rerank 返回得分。
2. **Postgres schema + vector-embedding-service**：建范文元数据 / `chunk_id` / 密级 / 文种 / 文号 / 单位名 / 回流记录 / outbox / 软删 + 审计双时间戳表；落 embedding（`EncodingFormat="float"`）与 rerank 窄抽象 → 验证：单元测试覆盖 float 编码、rerank 超时判定、同模型同维度约束。
3. **corpus-ingestion 写入侧**：初次批量建库（断点续传 / 幂等 / 单篇失败不阻断 / 缺密级拒绝）+ 补充入库 + 采纳稿回流（单点消费 c08 outbox / 事件表入库 + 对账，回流信号缺密级则阻断）；接 c02 脱敏与抽取；outbox 异步建索引 + 软删 + 对账 job → 验证：脱敏失败单篇跳过、三库落地、对账检出并修复漂移、派生索引可从权威源全量重建。
4. **hybrid-retrieval 读取侧**：`HybridSearch` 两路 + RRF、文号 / 单位名精确等值召回、文种过滤、rerank 精排（不可用降级）、Postgres 预过滤权威 ACL + `is_deleted==false` 兜底 → 验证：越级范文被拦截、前端过滤表达式被忽略、单路空仍融合、rerank 不可用降级返回 RRF。
5. **对接 c05 / c07 生成编排侧**（c05 必选、c07 可选；其生成最终经 c01 窄抽象接口出站）：暴露范文检索 API 供 RAG 提示拼装；向消费方（c05 / c07）返回"样例不足"标识，是否走 c07 通用兜底以 c06 路由判定为单一权威，本 change 不做文种降级决策。

**回退**：Milvus 为派生索引，任一步出问题可清空 Milvus 重建而不丢权威数据（Postgres + MinIO 不动）；检索侧故障时由 c05 / c07 生成编排侧据 c06 路由判定决定是否降级走通用兜底（PRD 5 档能力阶梯第 5 档），不阻断 c05 / c07 生成编排主流程。

**部署路径**：MVP 公网快速验证（向量库可临时用境内数据驻留的托管服务）→ 信创 / 私有化阶段 Milvus / embedding / rerank / MinIO 全本地离线、脱敏映射表不出域（ADR-0001 §2、D4、§4 PoC）。

## Open Questions

- **RRF 之外是否需要加权或自定义融合**：MVP 以 `NewRRFReranker()` 起步（ADR-0001 D4），稠密 / BM25 两路权重是否需按文种或意图调参，待真实公文检索效果评测后定。
- **rerank 在 MVP 默认开 / 关**：rerank 为可选环节（不可用即降级），是否默认启用取决于其对召回质量的边际收益与延迟成本，待评测。
- **采纳稿回流的密级来源**：密级由 c08 回流信号提供（继承原生成任务密级或采纳 / 核稿时文秘显式确认），具体默认策略由 c08 与客户保密策略对齐；本 change 侧只校验"必带密级、缺失则阻断"的底线（不变）。
- **embedding 模型选型与 chunk 切分粒度**：中文 embedding 具体模型（BGE 系候选）与公文 chunk 切分粒度（整篇 / 段落 / 滑窗）影响召回质量与向量维度，待 PoC 摸底（ADR-0001 §4）。
- **文种"可用阈值 / 稀缺线"取值**："样例不足"判定阈值（PRD 提示 <100 条稀缺线、每文种宜 ≥100 篇）的工程取值需结合客户实际存量与 UAT 标准确定。
- **对账 job 频率与告警阈值**：定期对账的周期与漂移告警阈值待按库规模与写入频率定。
