## ADDED Requirements

### Requirement: 客户存量公文初次批量建库

系统必须（MUST）提供范文知识库的初次批量建库入口，接收客户提供的存量公文（按文种成批导入），逐篇经 c02 脱敏网关脱敏后落地，并完成三库写入（Postgres 元数据、MinIO 原件、Milvus 派生索引）。批量导入必须可断点续传、可幂等重试，单篇失败不阻断整批，并产出逐篇结果清单（成功 / 脱敏失败 / 入库失败及原因）。每篇范文必须携带密级与文种标签；缺失密级的范文必须被拒绝入库（不得按非密默认放行）。

#### Scenario: 整批导入成功并三库落地
- **WHEN** 文秘上传一批带密级与文种标签的存量公文并触发初次建库
- **THEN** 系统对每篇先经 c02 脱敏网关脱敏，再将文档元数据与 `chunk_id` 写入 Postgres、原件写入 MinIO 范文桶，并经 outbox 异步在 Milvus 建立稠密向量 + BM25 索引，最终返回逐篇成功清单

#### Scenario: 单篇脱敏失败不阻断整批
- **WHEN** 批量导入中某一篇范文在 c02 脱敏网关返回"脱敏未完成"（超时 / 5xx / 格式异常等）
- **THEN** 系统必须跳过该篇且不将其原文写入任何库，将该篇标记为脱敏失败并记入结果清单，继续处理批内其余范文

#### Scenario: 缺失密级标签的范文被拒
- **WHEN** 待导入范文未携带密级标签
- **THEN** 系统必须拒绝该篇入库并在结果清单中标注"缺少密级"，不得以非密默认值放行

### Requirement: 范文日常补充入库

系统必须（MUST）提供单篇或小批量的范文补充入库入口，供文秘在初次建库后持续扩充范文库。补充入库必须复用与初次建库相同的脱敏、抽取与三库写入管线，且必须校验密级与文种标签齐备。补充入库的写入路径必须与初次建库一致，使得新增范文在检索侧立即可被命中（受 outbox 索引可见性延迟约束）。

#### Scenario: 文秘补充单篇范文
- **WHEN** 文秘通过补充入口提交一篇带密级与文种的新范文
- **THEN** 系统经 c02 脱敏后写入 Postgres / MinIO 并经 outbox 在 Milvus 建索引，索引建立完成后该范文可被混合检索召回

### Requirement: 采纳稿回流单点消费入库

采纳决策由 c08 持有。c08 必须（MUST）对计入采纳（采纳结论为"直接用"或"小改"）的成稿产出回流信号，写入 c08 与本 change 约定的同一 outbox / 事件表（不直调本 change API，避免双写）。回流信号字段固定为 { 成稿引用（MinIO 对象键 / chunk 标识）、采纳结论（直接用 / 小改 / 大改 / 弃用；"直接用 + 小改"计入采纳）、密级、文种 }。本 change 必须（MUST）单点消费该 outbox / 事件表，将采纳稿经与范文相同的脱敏与抽取管线沉淀为新的范文样例，并在 Postgres 记录回流来源（关联原生成任务 / 采纳动作）以便审计与对账。回流信号必须携带密级（由 c08 信号提供，来源为继承原生成任务密级或采纳 / 核稿时由文秘显式确认）；缺失密级时本 change 必须阻断入库，不得静默以非密放行。本 change 不提供供 c05 / c07 生成编排侧直调的回流入库 API。

#### Scenario: 单点消费 c08 回流信号入库
- **WHEN** c08 对一篇计入采纳的成稿向约定 outbox / 事件表写入回流信号（含成稿引用、采纳结论、密级、文种）
- **THEN** 本 change 单点消费该信号，将采纳稿经脱敏与抽取管线处理后作为新范文写入三库，并在 Postgres 回流记录中关联原生成任务，使其后续可被检索召回与对账

#### Scenario: 回流信号缺密级时阻断
- **WHEN** 消费到的回流信号未携带密级
- **THEN** 本 change 必须阻断该信号入库并标注"缺少密级"，不得静默以非密入库

#### Scenario: 未计入采纳不回流
- **WHEN** c08 对一篇采纳结论为"大改"或"弃用"的成稿做处置
- **THEN** 该成稿不产出回流信号、不进入本 change 入库管线，仅"直接用 / 小改"计入采纳并回流

### Requirement: 文号与单位名结构化标量抽取

入库时系统必须（MUST）复用 c02 脱敏网关的正则与 AC 字典能力，从范文正文抽取文号（如 `〔2024〕X号`）与发文单位名，写入 Milvus 的标量 VARCHAR（不切词 / keyword）字段与 Postgres 对应列，用于后续精确等值召回。系统不得对文号 / 单位名使用 jieba、BM25 或 `TEXT_MATCH` 等经 analyzer 切分的字段承载精确召回（同一 analyzer 会把 `〔2024〕X号` 切碎）。本 change 不实现脱敏识别算法本身，仅消费 c02 的词条来源与抽取产物。

脱敏边界：范文正文中的人名 / 证件号 / 金额等敏感实体必须（MUST）脱敏（占位替换）后入库；文号、发文单位名等标识性精确召回字段在客户密级策略下通常为公开信息，默认以明文写入 Milvus 标量字段与 Postgres 列做精确等值召回。脱敏针对的是"检索样例出公网喂模型前"的出域动作（走 c02），本地入库不属出域，故标识字段明文落本地库与脱敏边界不冲突。若客户密级策略判定文号 / 单位名亦敏感，则降级为：明文仅落 Postgres 权威源（受 ACL），Milvus 标量存确定性哈希 / 占位，精确召回先在 Postgres 命中再回 `chunk_id`。

#### Scenario: 文号抽取为不切词标量字段
- **WHEN** 一篇范文正文含文号 `〔2024〕12号`
- **THEN** 系统经 c02 正则 + AC 字典抽取后，将该文号整体写入 Milvus 标量 VARCHAR 字段与 Postgres 列，且不经 jieba / BM25 切分

#### Scenario: 单位名进入精确召回字段
- **WHEN** 一篇范文含发文单位名"XX市人民政府办公厅"
- **THEN** 系统将该单位名整体作为不切词标量字段写入，供精确等值匹配，而非作为 BM25 分词文本

#### Scenario: 正文敏感实体脱敏后入库
- **WHEN** 一篇范文正文含人名 / 证件号 / 金额等敏感实体
- **THEN** 系统经 c02 脱敏网关将这些敏感实体占位替换后再入库，敏感实体不以明文落入任一库，而文号 / 单位名等标识字段按默认明文写入精确召回字段

#### Scenario: 客户策略判定文号敏感时降级回表
- **WHEN** 客户密级策略判定文号 / 单位名亦为敏感、不可明文落 Milvus
- **THEN** 系统将文号 / 单位名明文仅落 Postgres 权威源（受 ACL）、Milvus 标量改存确定性哈希 / 占位，精确召回先在 Postgres 命中再回 `chunk_id` 取派生数据

### Requirement: 三库分工写入与可全量重建

系统必须（MUST）按三库分工写入范文：Postgres 为权威源（文档元数据、`chunk_id` 主键、密级 / 文种 / 文号 / 单位名字段、采纳稿回流记录、软删与审计双时间戳），MinIO 范文桶存原件且与 Milvus 后端桶物理隔离，Milvus 为可重建的派生混合索引（密级 = partition key、文种 = 标量过滤字段）。Milvus 索引必须可在丢失后从 Postgres + MinIO 全量重建，重建结果必须与权威源一致。

#### Scenario: 派生索引从权威源重建
- **WHEN** 运维触发 Milvus 派生索引全量重建
- **THEN** 系统以 Postgres 元数据与 MinIO 原件为权威源重新生成全部 chunk 向量与标量字段并写回 Milvus，重建后每个 `chunk_id` 的密级 partition、文种过滤字段与权威源一致

#### Scenario: 范文桶与 Milvus 后端桶物理隔离
- **WHEN** 系统写入范文原件
- **THEN** 原件必须落入 MinIO 业务范文桶，与 Milvus 自身后端使用的桶物理隔离，二者不得共用同一桶

### Requirement: 跨库最终一致性与软删对账

由于不存在跨库事务，系统必须（MUST）以"关系库先落地 + outbox 异步管线建索引 + 软删 + 定期对账 job"保障三库最终一致。写入必须先在 Postgres 落地并记录 outbox 事件，再由异步管线驱动 Milvus 建索引；删除必须走软删（`is_deleted=true`），不得物理直删导致索引漂移；定期对账 job 必须比对 Postgres 与 Milvus 的 `chunk_id` 集合，发现缺失 / 多余 / 状态不一致项并修复或告警。`chunk_id` 必须为全局主键。

#### Scenario: outbox 驱动异步建索引
- **WHEN** 一篇范文在 Postgres 落地并写入 outbox 事件
- **THEN** 异步管线消费该事件在 Milvus 建立索引，建索引失败时事件可重试且不丢失，最终该 `chunk_id` 在 Milvus 可见

#### Scenario: 对账 job 发现并修复漂移
- **WHEN** 定期对账 job 检出某 `chunk_id` 在 Postgres 标记为有效但在 Milvus 缺失
- **THEN** 系统必须重新触发该 chunk 的索引建立或告警，使两库 `chunk_id` 集合在密级 / 文种 / 软删状态上重新一致

#### Scenario: 删除走软删而非物理直删
- **WHEN** 需删除一篇范文
- **THEN** 系统在 Postgres 置 `is_deleted=true` 并经 outbox 同步 Milvus，查询层凭 `is_deleted==false` 兜底过滤，不依赖物理直删的即时生效

### Requirement: 范文入库操作层 RBAC 鉴权

发起范文入库前系统必须（MUST）消费 c04 `rbac-authorization` 的授权决策（`template.ingest` 权限点），未授予即拒绝（fail-closed），本 change 不自建 RBAC 判定。初次批量建库与日常补充入库的人工入口均受此约束；采纳稿回流由 c08 经约定 outbox / 事件表单点交付、不经此人工入库入口，其权限边界由 c08 侧持有。该操作层 RBAC 与既有密级数据 ACL 预过滤（Postgres 预过滤为权威 ACL）两层正交叠加、不可相互替代（操作层 RBAC ⊥ 数据层密级 ACL）：RBAC 判定"能否发起入库"，密级 ACL 判定"数据可见 / 可外发"，任一层拒绝即整体拒绝。

#### Scenario: 未授权角色不可发起范文入库
- **WHEN** 某角色未被授予 `template.ingest` 权限点而通过初次建库或补充入库入口发起范文入库
- **THEN** 系统经 c04 `rbac-authorization` 判定拒绝该入库请求（fail-closed），不进入脱敏 / 抽取 / 三库写入管线，本 change 不自建 RBAC 判定

#### Scenario: 已授权入库仍受密级缺失拒绝约束
- **WHEN** 某角色已被授予 `template.ingest` 权限点并提交缺失密级标签的范文
- **THEN** 系统放行入库操作权限，但仍因缺失密级拒绝该篇入库并标注"缺少密级"，操作层 RBAC 与数据层密级处置两层正交叠加、不可相互替代
