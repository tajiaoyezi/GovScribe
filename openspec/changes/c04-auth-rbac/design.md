## Context

GovScribe 的每一项业务能力（c01 模型接入配置 / 脱敏库维护 / 审计查看、c03 范文检索与入库、c05–c08 起草与在线核稿 / 采纳）在执行前都要先回答"这个人是谁"与"他能做什么 / 能看什么"两个问题。在统一基座出现之前，各 change 只能各自硬编码权限判断，既无法贯彻职责分离（审计员只读、业务兼职用户权限收窄），也无法保证"密级防越权"红线被一致执行。

本 change 提供贯穿全系统的鉴权 / 授权基座，落 MVP 单租户私有部署形态。权威锚点为 ADR-0001 **D10**（单租户、RBAC 落 Postgres、本地账号密码、可插拔认证适配、RBAC 与密级正交叠加）与 **D5**（PostgreSQL 为权威源 + 访问控制判定）；可插拔适配思路复用 **D1**（供应商窄抽象、切换实现业务代码不动）的范式。

当前状态与约束：

- 单二进制 Go 后端（D1），认证 / 授权以中间件 + 判定接口形式内嵌，不引入独立鉴权微服务，契合气隙 / 信创单机交付。
- PostgreSQL 已被定位为"权威源 + 访问控制判定"（D5、D10）；本 change 的用户 / 角色 / 权限映射与「数据可见 / 可外发」密级 ACL 同库共存，密级 ACL 的权威实现即 **PostgreSQL 预过滤**（ADR-0001 §2 / D5 / D10，后端注入过滤条件）。
- 密级两层须区分：①「数据可见 / 可外发什么数据」的数据 ACL —— 权威 = PostgreSQL 预过滤，归本 change 叠加判定的「密级臂」对接，c02 不提供该 ACL 判定；②「出公网走哪条模型路径」的出站密级路由与脱敏 —— 归 c02，本 change 不触及。本判定点不自建第二套密级判定。
- 红线继承：访问控制判定遵循与脱敏网关一致的 **fail-closed** 原则——判定信息不完整时默认拒绝，绝不静默放行。
- 三个 capability：`local-identity-auth`（认证与会话）、`rbac-authorization`（角色与操作授权 + 用户管理）、`access-control-composition`（两层正交叠加与统一决策接口）。

利益相关方：系统管理员（账号 / 角色 / 模型接入 / 脱敏库 / 审计的管理者）、文秘（核心使用者）、业务兼职用户（受限起草者）、审计员（只读监督者）、以及作为消费方的 c01–c08 各 change。

## Goals / Non-Goals

**Goals:**

- 提供统一身份模型与会话机制：本地账号密码登录、JWT 会话签发 / 校验、登录失败限制与停用即时失效，后端各接口据 JWT 识别 principal（用户标识 + 角色）。
- 把认证抽象成可插拔 Provider：MVP 落本地账号密码 Provider，为 CAS / OIDC / LDAP / 国产 IAM 预留接入位；切换 Provider 时业务 handler、RBAC 判定与密级判定代码不动，不同 Provider 产出的 principal 收敛为同一内部身份模型。
- RBAC 落 Postgres 为唯一权威源：固化系统管理员 / 文秘 / 业务兼职用户 / 审计员四角色、离散操作权限点与角色—权限映射；运行期判定以库内映射为准，未授予即拒绝。
- 提供统一访问控制判定点：将 RBAC（能做什么操作）与密级数据 ACL（能看 / 能外发什么数据，对接 PostgreSQL 预过滤为权威 ACL，ADR-0001 §2 / D10）两层正交叠加，对外暴露一致的"是否允许"决策接口供 c01–c08 调用，任一层拒绝即整体拒绝。
- 提供系统管理员侧用户 / 角色管理：创建 / 停用账号、分配 / 变更角色、重置密码，变更全程审计留痕。

**Non-Goals:**

- 不实现任何具体公文业务操作（属 c03 / c05–c08）；本 change 只是它们访问控制的统一前置依赖。
- 不实现脱敏识别 / 出站密级路由出域判定 / 脱敏审计（属 c02）；本 change 的「密级臂」对接 PostgreSQL 预过滤为权威 ACL（ADR-0001 §2 / D10）做数据可见 / 可外发判定，不自建第二套密级判定。
- 不引入多租户、组织树 SaaS、自助注册 / 找回密码等流程（D10 单租户私有部署，账号由系统管理员维护）。
- MVP 不真正接入外部 IAM（CAS / OIDC / LDAP / 国产 IAM）；只交付抽象接口与接入位，外部 Provider 实现属信创 / 政企终局阶段。
- 不在前端做权威授权判定；前端按角色收敛功能入口可见性属体验层，权威判定恒在后端。
- 本 change 自有「账号安全审计表」（承载账号 / 角色 / 认证 / 授权 / 登录失败事件，落 PostgreSQL，纳入相应密级 ACL），由审计员 / 系统管理员经 `audit.read` 权限点只读；脱敏出公网审计（脱敏前后 diff + 命中项 + 处置事件）的「脱敏审计表」属 c02，本 change 不承载、不归 c01。本 change 不实现脱敏审计表，也不把账号安全审计的承载外包给其它 change。

## Decisions

### D-1 单二进制内嵌鉴权，不拆独立鉴权服务

认证中间件、RBAC 判定服务、访问控制叠加判定点均以 Go 包 + HTTP 中间件 + 进程内接口形式内嵌在单二进制后端中（呼应 ADR-0001 D1 的单二进制 / 气隙目标）。

- 理由：单租户私有 / 信创单机部署下，独立鉴权服务只增加部署面与跨进程延迟，无收益；判定逻辑轻量，进程内调用即可。
- 备选：独立 AuthN/AuthZ 微服务（如 Keycloak）。否决——与单二进制气隙交付方向冲突，且引入额外运维与许可面；其能力在终局可由"外部 IAM Provider"接入位覆盖。

### D-2 会话凭证用 JWT，无状态校验 + 停用即时失效的折中

登录成功后签发 JWT，claims 含用户标识与角色信息，供下游 RBAC 与叠加判定直接消费；设有效期，过期 / 签名失败 / 缺失一律按未认证拒绝且不执行业务逻辑。

- 关键张力：JWT 无状态 → 停用账号后既有未过期凭证仍可能被接受，而 spec 要求"停用后既有会话失效"。
- 决策：采用"短有效期 JWT + 服务端校验时查 Postgres 账号启用状态 / 角色版本"的折中——每次受保护请求在解析 JWT 后，以其 user_id 反查 Postgres 当前 `is_active` 与角色，停用或角色变更即时反映。即 JWT 承载身份声明，账号启用态与角色映射的权威仍在 Postgres（D5 / D10）。这同时满足"角色变更后判定即时生效"。
- 备选一：纯无状态 JWT 不回查 → 停用 / 角色变更需等凭证自然过期，违反 spec，否决。
- 备选二：完全有状态服务端 session（每请求查 session 表）→ 失去 JWT 与 OnlyOffice JWT 回调（D8）一致的无状态优势，且未来对接 OIDC 时形态不匹配，不采用。
- 与 D8 关系：本 change 的会话 JWT 与 OnlyOffice 文档回调 JWT（D8）是两套独立用途的令牌，签名密钥与受众分离，不复用同一 token，避免把会话凭证暴露给编辑器集成面。
- 依赖登记：会话 JWT 库的选型与版本不在 ADR-0001 §3 锚点表之列，属本 change 在 design / 实现阶段新定版的依赖，按 ADR 体例登记其版本 / 许可（或建议补入 ADR-0001 §3 后再引用），不声称引用既有锚点表条目。

### D-3 认证可插拔 Provider 抽象（窄接口，复用 D1 范式）

定义统一认证 Provider 接口：输入凭证或外部回调，输出已认证的内部 principal（用户标识 + 角色映射所需信息）。MVP 实现 `LocalPasswordProvider`；为 CAS / OIDC / LDAP / 国产 IAM 预留同接口的实现位。

- 关键设计：外部 Provider 产出的外部身份必须在适配层映射为系统统一身份模型——本地用户表保留一行作为内部权威 principal（含角色绑定），外部 IAM 只负责"认证通过 + 提供外部用户标识 / 属性"，角色仍由本系统 RBAC 绑定（账号与角色的权威留在 Postgres）。下游 handler / RBAC / 密级判定不感知 Provider 来源。
- 理由：与 D1"供应商窄抽象、切换实现业务代码不动"同思路；把易变的对接细节关在适配层，业务与授权逻辑稳定。
- 备选：每接一种 IAM 改一遍业务鉴权链路 → 与 D10"业务代码不动"矛盾，否决。
- 边界说明：角色来源采用"内部绑定"而非"信任外部 IAM 下发的角色声明"，因为四类角色是 GovScribe 公文域的职责定义，外部 IAM 通常不具备对应语义；外部组 → 内部角色的映射规则留作终局阶段 Open Question。

### D-4 RBAC 模型：角色 + 离散操作权限点 + 映射，全部落 Postgres 为权威

数据模型（PostgreSQL，权威源，D5 / D10）：

- `users`（用户标识、用户名、加盐哈希密码、`is_active`、所属部门、创建 / 停用时间等）。
- `roles`（四类固定角色：系统管理员 / 文秘 / 业务兼职用户 / 审计员）。
- `permissions`（离散操作权限点，由各业务 change 声明、本 change 作为「权限点登记中心」收口登记；命名与归属对各消费方 change 固定）。本 change 登记的离散操作权限点至少含：
  - `document.open` / `document.edit` / `document.export`（文档打开 / 编辑 / 导出，供 c08）；
  - `review.online`（在线核稿，供 c08）；
  - `adopt.decide`（采纳决策，供 c08）；
  - `audit.read`（审计只读，覆盖本 change「账号安全审计表」与 c02「脱敏审计表」）；
  - `dict.manage`（脱敏库维护，供 c02）；
  - `model.config`（模型配置，供 c01）；
  - `template.search` / `template.ingest`（范文检索 / 入库，供 c03）；
  - `draft.create`（起草，供 c05 / c07）；
  - `user.manage`（用户 / 角色管理，本 change 自有）。
- `role_permissions`（角色—权限映射，运行期判定的唯一依据）。默认角色—权限映射（供 c08 / c02 / c01 等消费方对齐）：
  - **系统管理员**：`model.config`、`dict.manage`、`audit.read`、`user.manage`；
  - **文秘（核心）**：`document.open`、`document.edit`、`document.export`、`review.online`、`adopt.decide`、`template.search`、`template.ingest`、`draft.create`；
  - **业务兼职用户**：`document.open`、`document.edit`、`document.export`、`draft.create`、`template.search`（收窄面，不含 `model.config` / `dict.manage` / `user.manage` / `audit.read` / `review.online` / `adopt.decide` / `template.ingest`）；
  - **审计员（可选）**：仅 `audit.read`（只读，无任何写 / 业务起草权限点）。
- `user_roles`（用户—角色关联；MVP 可单角色，结构上允许多角色为终局留余地，但 MVP 不暴露多角色 UI）。

判定语义：以 principal 当前角色 + `role_permissions` 判断操作权限点；映射中未授予一律拒绝（fail-closed），对未显式声明的新操作同样默认拒绝。RBAC 只回答"能否做该操作"，不替代密级层。各消费方 change 在执行受控操作前依赖上述已登记权限点经统一判定点判定，不绕过、不硬编码副本（该约束属对消费方的接口契约约束，验收落在各消费方 change）。

- 理由：离散权限点 + 映射使新业务 change 只需"声明新权限点 + 配映射"即可纳入控制，无需改判定内核；以 Postgres 为权威避免硬编码副本与库不一致（spec 明确禁止硬编码副本）。
- 备选一：把角色—操作硬编码进 Go 常量 → 违反"以 Postgres 为权威、不得使用不一致硬编码副本"，否决。但允许在代码侧维护"操作权限点的声明清单（常量）"用于注册与编译期引用——声明 ≠ 授权判定，授权判定仍查库。
- 备选二：引入 ABAC / 策略引擎（如 OPA）→ 当前四角色 + 离散权限点的判定不需要策略语言的表达力，违反简单优先，否决。
- 备选三：把密级也建模为 RBAC 权限点 → 违反 D10"两层正交、不可相互替代"，否决。

### D-5 访问控制叠加判定点：单一决策入口，两层 AND 合取

提供唯一对外决策接口（概念签名）：`Decide(principal, permission, dataSecurityContext) -> Allow | Deny`。内部按顺序：

1. principal 缺失 / 未认证 → Deny（fail-closed）。
2. 操作权限点未声明 → Deny（fail-closed）。
3. RBAC 判定 Deny → Deny（短路，不再问密级）。
4. 密级臂判定：依据数据密级上下文查 PostgreSQL 预过滤为权威 ACL（ADR-0001 §2 / D5 / D10，由后端注入过滤条件）裁决数据可见 / 可外发；ACL 判定信息不完整或不可获取 → Deny（fail-closed）。
5. 两层均 Allow → Allow。

- 理由：单入口保证 c01–c08 不各自硬编码，且 fail-closed 与脱敏网关红线一致；AND 合取直接落实 D10"任一层拒绝即整体拒绝、缺任一层判定按拒绝处理"。
- RBAC 先于密级求值：RBAC 判定纯查本库、廉价、与密级臂同库（PostgreSQL），先短路可减少 ACL 预过滤的求值；二者均为必要条件，求值顺序不改变最终合取结果。
- 密级臂边界：本判定点的「密级臂」对接 PostgreSQL 预过滤为权威 ACL（数据可见 / 可外发），不自行重写脱敏 / 出站密级路由的出域逻辑（出域路由归 c02，避免出现第二套密级口径，违反 D10 与红线）；c02 不提供本 change 所需的数据 ACL 判定。
- 备选：让各 change 各自先查 RBAC 再查密级 → 回到硬编码、易漏查、口径漂移，正是本 change 要消除的问题，否决。

### D-6 密码存储与登录安全

- 密码以加盐哈希（如 bcrypt / argon2 类自适应算法，具体算法与参数在实现阶段定）存储于 Postgres；密码哈希算法版本不在 ADR-0001 §3 锚点表之列，属本 change 在 design / 实现阶段新定版的依赖，按 ADR 体例登记其版本 / 许可（或建议补入 ADR-0001 §3 后再引用），不声称引用既有锚点表条目。任何接口与日志不回显密码原文或完整哈希，重置密码的审计留痕不含新密码原文 / 哈希。
- 登录失败统一返回不区分"用户名不存在 / 密码错误"的提示（防账号枚举）；同账号限定窗口内连续失败达阈值后临时拒绝（防暴力破解），失败触发在审计留痕（承载见 rbac-authorization 用户管理）。
- 理由：直接落实 spec 的安全场景；自适应哈希 + 失败限流是私有部署下抗离线 / 在线爆破的最低成本组合。

### D-7 初始系统管理员引导（单租户私有部署）

不引入自助注册，故需明确"第一个管理员从哪来"：MVP 采用首次部署引导——通过部署期初始化（启动配置 / 一次性初始化命令）创建初始系统管理员账号，强制首次登录改密；初始凭证不硬编码进二进制、不写入日志。

- 理由：单租户私有部署无外部账号源，必须有冷启动入口；强制改密 + 不硬编码避免出厂默认口令风险。
- 备选：固定默认管理员口令 → 私有交付场景高危，否决。

## Risks / Trade-offs

- [JWT 无状态与"停用即时失效"的张力] → 采用 D-2：短有效期 JWT + 每次受保护请求回查 Postgres 账号启用态 / 角色，权衡掉部分"纯无状态零查询"性能换取红线（停用 / 角色变更即时生效）；回查走主键单点查询、可加进程内短 TTL 缓存但缓存 TTL 必须远小于停用容忍窗口。
- [外部 IAM 角色语义缺失] → c04 角色为公文域职责定义，外部 IAM 多无对应；MVP 决定角色由内部 RBAC 绑定（D-3），外部 Provider 只管认证；外部组 → 内部角色映射作为终局 Open Question，避免现在为不确定需求过度设计。
- [对脱敏网关的影响] → 本 change 不改脱敏网关（脱敏与出站密级路由归 c02），但叠加判定的 fail-closed 与脱敏网关 fail-closed 同源：密级臂的 PostgreSQL 预过滤 ACL 判定信息不完整 / 不可获取时整体拒绝，确保"授权维度"不成为绕过密级 / 脱敏的旁路。受控操作必须先过统一判定点，杜绝"先取数据后判密级"的越权窗口。
- [密级防越权的影响] → RBAC 通过不等于数据可见 / 可外发；统一判定点强制 AND 合取且密级以 Postgres 预过滤为权威（D5 / 架构 §密级防越权），本 change 不向前端开放任意过滤表达式，过滤条件由后端注入的格局不变。风险：调用方绕过统一判定点直连数据 → 缓解：设计上要求受控操作一律经统一接口，并由 code review / 集成约定兜底（缺授权依据即视为越权路径）。
- [三库（Postgres / Milvus / MinIO）一致性的影响] → 本 change 的权威数据（用户 / 角色 / 权限映射 / 账号启用态）全部只落 Postgres，不进 Milvus / MinIO，故不新增跨库一致性负担；Milvus 仍是可重建派生索引（D5 / 架构），其密级 partition / 标量过滤的权威过滤条件由后端依据 Postgres ACL 注入。风险：角色 / 停用变更后，已发起的 Milvus 检索若用旧 ACL → 缓解：ACL 过滤条件在每次检索请求时由后端按当前 Postgres 状态注入，不缓存到 Milvus 侧；本 change 保证 principal / 角色取值即时，c03 检索按当次注入执行。
- [角色粒度过粗] → 仅四角色 + 离散权限点，某些细分场景（如同为文秘但跨部门数据隔离）靠密级 / 部门字段而非更多角色解决；权衡为不引入复杂角色层级，保持简单优先；若后续确需更细操作粒度，按"新增权限点 + 调映射"扩展而非加角色，判定内核不动。
- [审计承载归属] → 账号 / 角色 / 认证 / 授权 / 登录失败事件由本 change 自有的「账号安全审计表」承载（落 PostgreSQL、纳入相应密级 ACL），不归 c01；脱敏出公网审计另由 c02「脱敏审计表」承载。审计事件含操作者 / 被操作账号 / 变更内容 / 时间，不含密码原文 / 哈希；查看权限受 `audit.read` 权限点约束（仅系统管理员 / 审计员），审计员经该权限点只读「账号安全审计表」与 c02「脱敏审计表」两类。

## Migration Plan

本 change 是新增基座，`openspec/specs/` 下无既有 capability，不修改任何已有需求，主要是数据库与中间件的引入。

1. 数据库迁移：在 Postgres 建 `users` / `roles` / `permissions` / `role_permissions` / `user_roles` 表，以及本 change 自有「账号安全审计表」；初始化四个固定角色，由权限点登记中心收口登记 D-4 列出的离散操作权限点种子（`document.open/edit/export`、`review.online`、`adopt.decide`、`audit.read`、`dict.manage`、`model.config`、`template.search/ingest`、`draft.create`、`user.manage`）+ D-4 默认角色—权限映射 → 验证：迁移脚本可重复执行（幂等），种子数据与 D10 角色边界、D-4 默认映射一致。
2. 初始管理员引导：部署期初始化创建系统管理员并强制首次改密（D-7）→ 验证：未改密前限制访问，初始凭证不出现在日志 / 二进制。
3. 后端接入：上线认证中间件（解析 JWT + 回查账号启用态 / 角色）与统一访问控制判定点；本 change 自身的登录接口与用户 / 角色管理接口先行接入并自校验 → 验证：缺失 / 过期 / 篡改 JWT 被拒；停用账号既有凭证被拒；非管理员无法执行用户管理。
4. 消费方接入：c01–c08 改为调用统一决策接口替换各自硬编码（分 change 推进，不在本 change 内完成）→ 验证：受控操作未过统一判定即被视为越权路径。
5. 密级臂对接：叠加判定的密级臂接 PostgreSQL 预过滤为权威 ACL（ADR-0001 §2 / D10，由后端注入过滤条件）裁决数据可见 / 可外发 → 验证：ACL 判定信息不完整 / 不可获取时整体拒绝（fail-closed）。

回滚策略：

- 本 change 为基座，一旦被 c01–c08 消费即不可简单回退；回滚仅适用于"上线尚未被消费"阶段——回退中间件挂载、回滚数据库迁移（drop 上述新表，因数据不进 Milvus / MinIO，回滚不触发跨库重建）。
- 已被消费后若发现授权缺陷，按"修判定逻辑 / 调 Postgres 映射"前滚修复，不整体回退，避免业务 change 失去访问控制依赖而暴露越权面（fail-closed 缺省下，宁可误拒不可误放）。

## Open Questions

- 外部 IAM（CAS / OIDC / LDAP / 国产 IAM）的"外部组 / 属性 → c04 内部四角色"映射规则如何配置？（终局阶段解决，MVP 角色内部绑定。）
- JWT 有效期与失败限流阈值、回查缓存 TTL 的具体取值？（实现阶段按私有部署运维口径定，需远小于停用容忍窗口。）
- 加盐哈希算法与参数选型（bcrypt / argon2 及代价因子）？（实现阶段定；密码哈希算法版本不在 ADR-0001 §3 锚点表，属本 change 新定版依赖，按 ADR 体例登记或建议补入 §3 后再引用，不声称引用既有锚点条目。）
- 会话 JWT 库选型与版本？（实现阶段定；不在 ADR-0001 §3 锚点表，属本 change 新定版依赖，按 ADR 体例登记或建议补入 §3 后再引用。）
- 业务兼职用户与文秘的部门级数据隔离，是完全交由密级 / 部门字段（PostgreSQL 预过滤数据 ACL）承载，还是需要在 RBAC 侧补部门维度的操作权限点？（倾向前者，保持两层正交，待各业务 change 落地后复核。）
- 审计员"只读审计与脱敏留痕"的具体只读范围边界？（账号安全审计表由本 change 承载，脱敏审计表由 c02 承载；审计员经 `audit.read` 只读两类，与 c02 联调脱敏审计表的只读视图对齐点。）
