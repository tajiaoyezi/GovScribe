## Why

GovScribe 的所有业务能力（模型接入配置、脱敏库维护、范文检索与入库、初稿起草、在线核稿与采纳、审计查看）都需要先回答两个问题：**这个人是谁**、**他能做什么 / 能看什么**。在没有统一身份与授权基座之前，各 change 只能各自硬编码权限判断，既无法贯彻"职责分离"（如审计员只读、业务兼职用户权限收窄），也无法保证密级防越权红线被一致执行。

本 change 提供贯穿全系统的鉴权 / 授权基座：MVP 阶段以本地账号密码起步、单租户私有部署，权限用 RBAC 落 Postgres，并把认证抽象成可插拔适配层，使信创 / 政企终局对接客户统一身份认证时业务代码不动（口径见 ADR-0001 D10）。它本身不实现任何公文业务操作，而是被 c01–c08 各 change 统一消费。

## What Changes

- 新增**本地账号密码认证**：用户登录后签发会话凭证（JWT），后端各接口据此识别 principal；密码加盐哈希存储，不引入注册自助流程（账号由系统管理员维护，契合单租户私有部署）。
- 新增**认证可插拔适配层**：定义统一的认证 Provider 抽象（输入凭证 / 回调 → 输出已认证 principal），MVP 实现本地账号密码 Provider；为信创 / 政企终局预留 CAS / OIDC / LDAP / 国产 IAM 适配实现的接入位，切换 Provider 时业务代码与 RBAC / 密级判定不动（与 D1 模型接入窄抽象同思路）。
- 新增 **RBAC 授权模型并落 Postgres**：固化四类角色——**系统管理员**（配模型接入 / 维护脱敏库 / 看审计 / 管用户）、**文秘（核心）**（起草 / 检索范文 / 在线核稿 / 决定采纳 / 补范文入库）、**业务兼职用户**（文种引导 + 通用兜底起草，权限较窄）、**审计员**（只读审计与脱敏留痕，职责分离，可选）。角色—权限（操作）映射以 Postgres 为权威源。
- 新增**统一访问控制判定点**：把 **RBAC（管"能做什么操作"）与密级（管"能看 / 能外发什么数据"，以 PostgreSQL 预过滤为权威 ACL，口径见 ADR-0001 §2 / D10，后端注入过滤条件）两层正交叠加**判定，对外提供一致的"是否允许"接口供其它 change 调用；两层不可相互替代，任一层拒绝即拒绝。
- 新增**用户 / 角色管理能力**（系统管理员视角）：创建 / 停用账号、分配角色、重置密码；变更进审计留痕。
- 明确**单租户边界**：MVP 不引入多租户复杂度，数据按部门 / 用户 / 密级隔离，每客户独立部署一套。

> 范围声明：本 change 不触及脱敏识别 / 出站密级路由出域判定（属 c02）；其访问控制叠加判定的「密级臂」对接 PostgreSQL 预过滤为权威 ACL（ADR-0001 §2 / D10）完成正交叠加，不自建第二套密级判定；不实现任何具体公文业务操作（属 c03/c05–c08）。本 change 落在 MVP 范围内，是贯穿基座；它直接服务于保密红线——RBAC 与密级两层正交叠加是密级防越权在"授权"维度的承载，但脱敏与出站密级路由本身不在此 change 实现。

## Capabilities

### New Capabilities

- `local-identity-auth`：身份认证与会话。本地账号密码登录、JWT 会话签发与校验、登录失败处理；并定义可插拔认证 Provider 抽象，MVP 落本地实现，为 CAS / OIDC / LDAP / 国产 IAM 预留接入位而不改业务代码。
- `rbac-authorization`：基于角色的操作授权。在 Postgres 落地角色（系统管理员 / 文秘 / 业务兼职用户 / 审计员）、操作权限点与角色—权限映射，回答"某 principal 是否可执行某操作"；并含系统管理员侧的用户 / 角色管理。
- `access-control-composition`：访问控制叠加判定。把 RBAC 的操作授权与「密级数据可见 / 可外发」判定（对接 PostgreSQL 预过滤为权威 ACL，ADR-0001 §2 / D10）两层正交叠加，对外提供供 c01–c08 统一调用的鉴权决策接口，确保任一层拒绝即整体拒绝。

### Modified Capabilities

（无。`openspec/specs/` 下当前无既有 capability，本 change 为新增基座，不修改任何已有 spec 的需求。）

## Impact

- **数据库（PostgreSQL）**：新增用户、角色、操作权限点、角色—权限映射、用户—角色关联等表，以及本 change 自有的「账号安全审计表」（账号 / 角色 / 认证 / 授权 / 登录事件）；作为认证、RBAC 与数据 ACL 的权威源——密级数据可见 / 可外发的权威实现即 PostgreSQL 预过滤 ACL（ADR-0001 §2 / D10），由后端注入过滤条件。
- **后端（Go）**：新增认证 Provider 抽象与本地实现、会话（JWT）签发 / 校验中间件、RBAC 判定服务、访问控制叠加判定入口；以中间件 / 判定接口形式被各业务 handler 统一调用。
- **被消费方（其它 change）**：c01 模型配置（`model.config`）、c02 脱敏库维护（`dict.manage`）等管理端操作受 RBAC 约束；审计只读（`audit.read`）覆盖本 change 自有的「账号安全审计表」与 c02 的「脱敏审计表」两类；c03 范文检索 / 入库、c05–c08 起草与在线核稿 / 采纳均需先过认证再过 RBAC + 密级叠加判定。本 change 是它们访问控制的统一前置依赖。
- **依赖关系**：密级臂对接 PostgreSQL 预过滤为权威 ACL（ADR-0001 §2 / D5 / D10，后端注入过滤条件）完成正交叠加，本 change 不自建第二套密级判定。会话 JWT 库与密码加盐哈希算法不在 ADR-0001 §3 锚点表之列，属本 change 在 design / 实现阶段新定版的依赖，按 ADR 体例登记其版本 / 许可（或建议补入 ADR-0001 §3 后再引用）；其余既有依赖仍引用 ADR-0001。
- **前端（React + TypeScript + Ant Design）**：新增登录页与会话态管理；管理端新增用户 / 角色管理界面；按角色对功能入口做可见性收敛（前端为体验层，权威判定仍在后端）。
- **运维 / 部署**：单租户私有部署，初始系统管理员账号的创建 / 引导方式需在 design 中明确（不引入多租户与外部注册）。
