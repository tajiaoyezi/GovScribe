## Context

GovScribe 的核心价值假设是"让通用大模型写出像公文、可直接采纳的初稿"（PRD「Problem」「Hypothesis」）。c01（可配置模型层 + 统一 SSE 流式透传）与 c03（范文 RAG 混合检索）已分别就位：c01 提供窄抽象生成接口与三类归一 SSE 事件，c03 提供按文种过滤的范文召回并能在样例不足时返回"样例不足"标识。但这两层只解决了"用什么模型、喂什么样例"，没有回答"按什么公文规范把样例组织成规范正文"。本 change 落地 Milestone 2，在 c01/c03 之上补齐两件事：把 9 个高频文种（通知 / 请示 / 报告 / 函 / 会议纪要 / 通报 / 批复 / 讲话稿 / 方案）的行文规范沉淀为可读取的结构契约，并以一条固定编排管线把"检索 → 拼装 → 生成 → 流式输出"串起来。

约束来自三处权威口径，本设计不另立第二套：

- PRD「文种覆盖矩阵」：5 档能力阶梯（① 模板生成 → ② 模板辅助 → ③ 框架写 → ④ 待计划训练 → ⑤ 通用兜底）；标黄子类 = 训练数据 <100 条。文种 / 子类落在哪一档、是否标黄稀缺、是否降级的判定属 c06 文种能力档分级表（c06 权威），本 change 仅处理 c06 已判为目标 capability==c05（即"① 模板生成"深做档）的请求，不自建该分级判定。
- PRD 验收 rubric 草案：文种规范 / 结构完整 / 行文关系 / 机关口径四个维度，叠加采纳率口径作为 UAT 判定。
- ADR-0001 决策 D1–D10：Go 单二进制应用层、SSE 流式、LiteLLM→Go 直连切换、Milvus 混合检索、Postgres 权威源、脱敏网关 fail-closed、RBAC 与密级两层正交。

利益相关方：文秘（核心，对文种规范 / 行文关系要求最高）、业务兼职用户（最怕选错文种、写得不像）、系统管理员（维护分级配置）、客户合规部门（密级口径与范文盘点）。

当前状态：c05 尚无后端实现，仅有 proposal 与本 change 的两份 delta spec（`doctype-prompt-templating`、`draft-generation-streaming`）。本设计为这两份 spec 的实现路径给出技术取舍。

## Goals / Non-Goals

**Goals:**

- 把 9 个高频文种的行文规范（标题 / 称谓 / 主送 / 正文结构 / 落款 / 行文关系口吻 / 机关口径红线）沉淀为数据驱动、可按文种读取的结构契约，使生成编排代码不硬编码任何单一文种的规则。
- 定义一条确定的初稿生成编排管线：消费 c06 结构化场景上下文契约（目标 capability==c05）→ 经 c03 检索同类范文 → 按结构契约与 few-shot 编排规则拼装提示 → 仅经 c01 窄抽象接口发起生成 → 经统一 SSE 事件流式下发正文。
- 只处理 c06 判定目标 capability==c05 的请求，不自建文种分级判定、不复算是否标黄稀缺、不自行移交 c07（文种分流单一权威 = c06）。
- 模型调用仅经 c01 窄抽象接口发起，编排层不绕过 c01 窄抽象直连任何 SDK；脱敏与出站密级路由由 c01 公网分支上的 c02 强制拦截点承接，c05 不直接面对公网、不自行实现或断言脱敏与出站密级路由环节。
- 消费 c06 结构化场景上下文契约中的内容密级（取值：非密 / 敏感 / 涉密），并在经 c01 窄抽象接口发起生成时将该内容密级作为出站请求密级随调用透传供 c01 公网分支上的 c02 路由（仅透传该密级输入值、不自建密级判定，与 c07 同口径，不改写 / 不降级）。
- 给出范文不足、流中断、底层失败三类边界的可观察处置，避免"功能就绪、库空"导致 UAT 说不清是模型问题还是缺数据（PRD Risks）。

**Non-Goals:**

- 不做文种路由 / 文种判别 / 分流 / 标黄稀缺降级判定（c06）：本 change 消费 c06 输出的结构化场景上下文契约，仅处理目标 capability==c05 的请求，不复算文种 / 子类 / 是否稀缺。
- 不做通用兜底生成本身（c07），也不在 c05 内做"移交 c07"判定：标黄稀缺降级移交只发生一次、在 c06 路由阶段；落到 c05 的请求即 c06 已判为 capability==c05。
- 不做在线编辑 / 核稿界面（c08）：只交付生成接口与流式契约；前端落地范围以 c08 为准。
- 不做 GB/T 9704 版式 / 红头 / 字体字号排版（后置，PRD Post-MVP）：出参限于规范正文内容。
- 不实现脱敏与出站密级路由判定逻辑，也不持有该环节：脱敏与密级路由由 c01 公网分支上的 c02 强制拦截点承接，c05 不直接面对公网，只保证模型调用经 c01 窄抽象接口、不绕过其直连 SDK。
- 不改 c01 / c03 对外契约，不微调模型（PRD「公网提质路线」明确不微调）。
- 不新增 Milvus collection：范文检索复用 c03 既有混合索引。

## Decisions

### D-1 结构契约用数据配置而非代码硬编码

**决策**：把每个文种的结构要素（标题构成、称谓、主送机关规则、正文段落结构与必备要素、落款、行文关系口吻、机关口径红线）建模为 Postgres 中的结构化配置 + MinIO 业务桶中的提示模板文本，由生成编排在运行时按文种标识读取拼装；生成编排代码内不出现"if 文种==请示 then …"的分支。文种 / 子类的能力档分级与是否标黄稀缺不在本表内，属 c06 文种能力档分级表（c06 权威），本 change 不承载。

**理由**：9 个文种 + 大量子类、且 PRD 明示"开发中可适当补充 / 调整"，规则会随客户 UAT 反馈频繁迭代；落数据让管理员可维护，符合简单优先与 ADR-0001 D5"Postgres 作权威源 + 文档元数据"的定位。规则若散在 Go 代码里，每次调措辞都要改代码、重编单二进制，违背可维护性。

**取舍**：相比"全部硬编码进 Go struct"（启动快、强类型校验，但改一条措辞要发版）与"全部塞进单条超长 system prompt"（最灵活，但无法结构化校验必备要素覆盖），选择"结构化字段（用于校验）+ 模板文本（用于提示拼装）"的分层：行文关系 / 必备要素清单进 Postgres 标量字段，可被 SQL 查询与一致性校验；冗长的口吻指令与 few-shot 框架文本进 MinIO 业务桶按版本存放。对齐 ADR-0001 D5 三库定位，不引入新组件。

### D-2 编排管线固定为"消费 c06 上下文 → 检索 → 拼装 → 生成 → 流式"，不在本 change 复算文种分级

**决策**：一次初稿生成按固定顺序执行：① 消费 c06 `doctype-classification` 输出的结构化场景上下文契约，仅当目标 capability==c05 时进入本管线（c05 不自建文种分级判定、不复算是否稀缺 / 标黄、不自行移交 c07）；② 调 c03 按上下文中的文种过滤检索范文样例；③ 按 `doctype-prompt-templating` 契约与 few-shot 规则拼装提示；④ 仅经 c01 `model-provider-abstraction` 窄抽象接口发起生成；⑤ 经 c01 统一 SSE 事件流式下发。

**理由**：文种判别 / 分流 / 标黄稀缺降级判定的单一权威是 c06——c06 在路由阶段查其文种能力档分级表，一次性判定目标 capability（c05 或 c07），降级移交判定只发生一次且在 c06。c05 处于其下游，若再查一份分级配置自行判定标黄、自行决定是否移交 c07，会产生与 c06 双口径漂移的风险，违背"文种分流单一权威"。因此本 change 不持有分级配置、不做二次判定，只对 c06 已判为 capability==c05 的请求施加深做结构契约。c03 的"文种过滤召回 + 样例不足标识"已就位（c03 hybrid-retrieval spec），本管线直接消费，不重复实现检索。

**取舍**：把分级判定完全交给 c06，代价是 c05 强依赖 c06 上下文契约的正确性；换来与 c06 / c07 口径一致、无双判定漂移。模型调用坚持只经 c01 窄抽象接口（编排模块导入图中不得出现 `openai-go` / `anthropic-sdk-go` / LiteLLM 专有类型，对齐 c01 spec 与本 change spec「生成仅经 c01 窄抽象接口」），换来 ADR-0001 D3 的 LiteLLM→Go 直连切换时本 change 业务代码零改动。

### D-3 复用 c01 统一 SSE 事件，不为本 change 另造流式协议

**决策**：本 change 的"高频文种初稿生成"接口直接复用 c01 `model-streaming-passthrough` 的三类归一事件（文本增量 / 正常结束 / 异常结束），在其上仅追加本 change 语义所需的元数据（所消费的 c06 上下文中的文种 / 子类等标识），不定义第二套 SSE 帧格式。

**理由**：c01 已把 LiteLLM 后端与 Go 直连 SDK 后端的流式差异归一（对齐 ADR-0001 D2、D3），并定义了"已透传增量不回滚""异常结束事件可区分内容完整 vs 中断不完整"。本 change 的中断 / 失败语义（用户取消释放资源、底层失败发错误事件不伪装完成）正是 c01 异常结束事件的应用，没有理由重造。

**取舍**：相比自定义流式协议（可塞入更丰富的公文结构标注），复用 c01 事件让前端只需对接一套 SSE；代价是本 change 的结构化元数据只能在流首 / 流尾事件携带，无法逐片段标注段落角色——但版式与结构标注后置（Non-Goals），当前不需要逐片段语义，可接受。ADR-0001 D3 已核验 Anthropic 经 LiteLLM 的字段映射损耗，c01 spec 以宽松 JSON 吸收，本 change 不再处理该坑。

### D-4 范文样例只来自 c03，脱敏文本不可改写，few-shot 条数设上限

**决策**：拼装提示时的范文样例仅取 c03 检索接口返回的结果，编排层不从任何其它来源臆造或改写范文；c03 返回的样例已是脱敏后文本，本 change 不得还原 / 修改其内容；few-shot 注入条数设契约上限（如 TopK，具体值待实测校准），样例与待写场景要素分区呈现。

**理由**：范文是真实公文经脱敏入库，ADR-0001 D7 铁律要求映射表 / 原文不出 Go 网关；本 change 处于网关下游，只应消费 c03 已脱敏的样例文本，任何"补全 / 润色范文"都可能把脱敏占位符还原成敏感内容或臆造出真实公文不存在的表述，越过 c02 红线。设条数上限是为控制提示长度与公网 token 成本，并降低"长上下文里样例淹没用户场景要素"的风险。

**取舍**：限制样例来源牺牲了"模型自由发挥补充范文"的灵活性，换来可审计性与脱敏红线不被旁路。

### D-5 缺要素留占位、不臆造，初稿恒为待核稿态

**决策**：机关口径红线作为每个文种契约的内置约束注入提示：政治性表述准确、不臆造事实 / 数据 / 文号 / 人名单位；用户未提供的关键要素（文号、具体数据、单位全称等）以占位或明确待补提示呈现，不编造取值；产出恒定性为"待人工核稿的初稿"，接口与提示中不含任何自动发文 / 自动提交语义。

**理由**：直接对齐 PRD「人工最终把关，绝不自动发文」与 Risks「政治性 / 合规内容风险」。公文里虚构的文号 / 单位名会让初稿"不能直接用甚至有合规风险"，留占位反而提升采纳率（文秘补一个空比删一段编造内容快）。

**取舍**：留占位的初稿"完成度观感"略低于全部填满的版本，但后者以臆造为代价，与产品红线冲突，不可取。

### D-6 文种分级 / 标黄 / 降级以 c06 为单一权威，本 change 不持有分级判定

**决策**：文种 / 子类 → 能力档 → 是否标黄稀缺 → 是否降级的判定，全部以 c06 `doctype-classification` 在路由阶段查"文种能力档分级表"（owner=c06）一次性完成为唯一权威；c05 只消费 c06 输出的结构化场景上下文契约中的目标 capability，仅对 capability==c05 的请求施加深做结构契约，不在本 change 维护任何分级配置、不复算是否标黄、不自行决定是否移交 c07。标黄稀缺降级移交判定只发生一次，在 c06。

**理由**：PRD Risks「数据稀缺文种生成质量不足」与「'不像公文'是主观判断，验收标准难量化」要求对外承诺与实际行为一致；文种分流单一权威是 c06，若 c05 再自建一份分级判定，会与 c06 产生双口径漂移、且把"降级移交 c07"这一只应发生一次的判定重复执行。把分级完全收归 c06 是避免漂移的最简手段。

**取舍**：c05 不再持有分级配置，强依赖 c06 上下文契约的目标 capability 正确；换来与 c06 / c07 口径严格一致、降级判定单点不重复。

## Risks / Trade-offs

- [范文库数据不足 / 未按时到位，深做文种检索不到优质样例，初稿退回"不像公文"，无法判断是模型问题还是缺数据] → 编排层消费 c03 的"样例不足"标识，深做文种即便零样例仍施加结构契约约束并在生成元数据标记"范文样例不足、质量可能下降"（不静默退回无契约通用生成）；开工前按 9 个文种盘点客户存量范文（对照 <100 条稀缺线，每文种宜 ≥100 篇，ADR-0001 §5 待确认项），不足文种降级不承诺深做。

- [编排层可能绕过 c01 窄抽象直连某 SDK，使敏感数据出域时不经 c01 公网分支上的 c02 拦截点] → 全部模型调用只经 c01 窄抽象接口；对编排模块做依赖扫描断言导入图中无 `openai-go` / `anthropic-sdk-go` / LiteLLM 专有类型。脱敏、出站密级路由、涉密恒走私有永不降级均由 c01 公网分支上的 c02 强制拦截点承接，c05 不直接面对公网、不自行透传或断言该环节，仅以"不绕过 c01 窄抽象直连 SDK"作为本 change 内可验证的红线。

- [密级防越权：范文样例可能携带当前用户密级不可见的内容进入提示] → 密级 ACL 由 c03 以 Postgres 预过滤为权威（c03 hybrid-retrieval spec「密级防越权召回」）落地，本 change 处于其下游，只消费已过滤的样例，且按 ADR-0001 D10 RBAC 与密级两层正交：能否发起生成（操作）由 RBAC 判定，能取到哪些范文（数据）由密级判定。残余风险是 c03 上游过滤缺陷，归属 c03，不在本 change 兜底范围，但本 change 不得自行放宽 c03 注入的过滤条件。

- [三库一致性：文种结构契约配置（Postgres）、范文样例（Milvus 派生索引 + MinIO 原文）可能不一致] → 结构契约 / 模板配置以 Postgres 为权威源（ADR-0001 D5）；范文检索复用 c03 的 Milvus 派生索引（密级 partition + 文种标量过滤），Milvus 可从 Postgres + MinIO 全量重建（ADR-0001 §2），本 change 不新增 collection、不引入新的跨库写。采纳稿回流由 c08 产信号、c03 单点入库，c05 不承担，故本 change 不引入回流相关跨库写。

- [标黄稀缺子类被误当深做处理，对客户做出无法兑现的质量承诺] → 标黄稀缺降级移交判定只发生一次、在 c06 路由阶段（c06 查其文种能力档分级表），落到 c05 的请求即 c06 已判为 capability==c05；c05 不二次判定、不维护分级配置，从源头消除"c05 误判标黄子类为深做"的可能。残余风险归 c06 上下文契约正确性，不在本 change 兜底。

- [LiteLLM 经 LiteLLM 的 Anthropic 流式字段映射损耗 / 多 worker 热切换传播延迟] → 已由 ADR-0001 D3 核验、c01 spec 以宽松 JSON 吸收并归一 SSE 事件，本 change 复用 c01 流式（D-3），不重复处理；重度依赖 thinking 的文种若出现损耗，按 c01 既有路径（走原生 `/v1/messages` 或直连 anthropic-sdk-go）解决，本 change 业务代码不动。

- [提示注入：用户场景要素中夹带指令试图绕过机关口径红线 / 套出脱敏前内容] → 场景要素与契约指令、few-shot 样例分区呈现（D-4），机关口径红线作为契约级约束（D-5）置于用户输入之外；脱敏由 c02 前置承接，本 change 不持有脱敏前原文与映射表。

## Migration Plan

本 change 为新增能力，无存量数据迁移。部署步骤：

1. 建文种结构契约配置表：在 Postgres 落"文种 → 行文关系 → 必备要素清单"等结构契约标量配置（9 个高频文种），提示模板文本入 MinIO 业务桶并记版本。文种 / 子类的能力档分级与标黄标记属 c06 权威，不在本步初始化。
2. 上线生成编排服务与"高频文种初稿生成"SSE 接口（Go 单二进制应用层，ADR-0001 D1），依赖 c01 / c03 / c06 已部署可用；脱敏与出站密级路由由 c01 公网分支上的 c02 强制拦截点承接（ADR-0001 D7），本 change 不直接面对公网。
3. 灰度：先开放少数样本充足的深做文种（如通知 / 请示 / 报告）给试点文秘，按验收 rubric 四维 + 采纳率打标，校准 few-shot 条数上限与契约措辞，再逐步放开其余文种。
4. 回滚：本 change 与上游解耦，回滚只需下线本生成接口；分级 / 模板配置为新增表与新增 MinIO 对象，不影响 c01 / c03 既有数据。配置级调整（措辞 / 标黄 / 能力档）走改 Postgres + MinIO 版本对象、无需重发版（D-1 的收益）。
