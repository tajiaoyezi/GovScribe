## ADDED Requirements

### Requirement: 9 个高频文种的结构契约库

系统必须（MUST）为 9 个高频文种——通知、请示、报告、函、会议纪要、通报、批复、讲话稿、方案——分别定义一份公文结构契约（对齐 PRD「文种覆盖矩阵 A」与 Decisions「高频文种清单」）。每份契约必须显式声明该文种的规范要素及其约束：标题构成（如"关于 + 事由 + 文种"）、称谓、主送机关、正文段落结构与必备要素、落款（发文单位 + 成文日期）。契约必须可被 `draft-generation-streaming` 在拼装提示时读取，作为该文种"正文应当长成什么样"的权威依据；不得在生成编排代码中硬编码各文种的结构规则。

#### Scenario: 按文种取到完整结构契约

- **WHEN** 上层以某高频文种标识（如"请示"）请求其结构契约
- **THEN** 系统必须返回该文种的标题、称谓、主送机关、正文段落结构、落款各要素的规范定义
- **AND** 返回的要素集合必须覆盖该文种的全部必备要素，不得缺漏其中任一项

#### Scenario: 非高频文种不返回深做契约

- **WHEN** 上层以不属于上述 9 个高频文种的文种标识请求结构契约（正常路由下不应出现，因 c06 仅将这 9 个深做文种判为目标 capability==c05）
- **THEN** 系统必须明确返回"无深做契约"而不得返回某个高频文种的契约
- **AND** 必须不在本 change 内自行判定移交 c07（文种分流与降级移交单一权威 = c06），仅以"无深做契约"防御性拒绝

### Requirement: 行文关系决定口吻与措辞

每个文种契约必须（MUST）标注其行文关系（上行文 / 下行文 / 平行文），并据此约束生成正文的口吻与措辞（对齐验收 rubric「行文关系」）。请示、报告为上行文，须用呈请、报告口吻并含恰当的结束语（如"妥否，请批示"）；通知、通报、批复为下行文，须用部署、告知口吻；函为平行文（不相隶属机关之间），须用商洽、询问口吻。系统必须保证同一事由在不同文种下因行文关系不同而采用不同口吻，不得套用统一模板腔。

#### Scenario: 上行文采用呈请口吻

- **WHEN** 取用请示或报告文种的契约拼装提示
- **THEN** 契约必须要求正文采用对上级的呈请 / 报告口吻
- **AND** 必须包含与上行文相称的结束语约束（如请示须有请示事项的结束语）

#### Scenario: 同事由不同文种口吻有别

- **WHEN** 同一写作事由分别按请示（上行）与通知（下行）取用契约
- **THEN** 两份契约施加的口吻与措辞约束必须不同
- **AND** 不得令两个文种共用同一套口吻规则

### Requirement: 范文样例编排进提示的契约

系统必须（MUST）定义"如何将 `corpus-rag-retrieval`（c03）检索到的同类范文样例编排进文种提示"的契约（对齐 PRD「公网提质路线」few-shot 思路）。契约必须规定：样例以 few-shot 形式注入、与待写场景要素分区呈现、注入条数上限、以及样例与目标文种 / 子类一致性的取用规则。系统必须只编排由 c03 检索接口返回的样例，不得自行从其它来源臆造范文，也不得改变 c03 返回样例的脱敏后文本内容。

#### Scenario: 检索到样例时按 few-shot 注入

- **WHEN** c03 针对当前文种与场景返回若干条同类范文样例
- **THEN** 系统必须按契约将样例以 few-shot 形式编排进提示，并与待写场景要素分区
- **AND** 注入的样例条数不得超过契约规定的上限

#### Scenario: 不臆造范文样例

- **WHEN** 拼装提示需要范文样例
- **THEN** 系统必须仅使用 c03 检索接口返回的样例文本
- **AND** 不得凭空生成或改写范文样例充当 few-shot 输入

### Requirement: 范文不足时的降级口径

当落到本 change（目标 capability==c05）的某文种经 c03 检索到的可用样例不足以支撑"深做"质量时，系统必须（MUST）按既定降级口径处理（对齐 PRD「Open Questions 范文数据」与 Risks「范文库数据不足」）：仍按该文种结构契约拼装提示生成，并在生成元数据标记"范文样例不足、质量可能下降"，不得静默退回无结构约束的通用生成而不作标记。此处"范文不足"指 c03 运行时检索样例数量不足，与文种是否标黄稀缺的分级判定无关——后者属 c06 文种能力档分级表，本 change 不判定、不复算。

#### Scenario: 样例不足仍带结构契约生成并标记

- **WHEN** 深做文种检索到的范文样例数量低于契约设定的充足阈值
- **THEN** 系统必须仍依据该文种结构契约拼装提示，而非退回无契约的通用生成
- **AND** 必须在生成元数据中标记"范文样例不足、质量可能下降"

#### Scenario: 不静默丢弃结构约束

- **WHEN** 任一深做文种检索不到任何范文样例
- **THEN** 系统必须仍施加该文种的结构契约约束
- **AND** 不得在无任何标记的情况下退回到不含文种结构约束的输出

### Requirement: 机关口径红线约束

每个文种契约必须（MUST）内置机关口径红线约束（对齐 PRD 验收 rubric「机关口径」与 Risks「政治性 / 合规内容风险」）：政治性表述必须准确、不得臆造事实 / 数据 / 文号 / 人名单位，不得输出敏感或不合规表述。契约必须要求模型对用户未提供的关键要素以占位或留空提示，而非编造填充。本 change 产出仅为初稿，必须不承担自动发文职责，由人工核稿决定采纳（对齐 PRD「人工最终把关」）。

#### Scenario: 缺要素时不臆造而留占位

- **WHEN** 用户场景要素中缺少文号、具体数据或单位全称等关键信息
- **THEN** 契约必须要求模型以占位或明确的待补提示呈现，而不得编造具体取值
- **AND** 生成的初稿不得出现未经用户提供的虚构文号或单位名

#### Scenario: 初稿不承诺自动发文

- **WHEN** 任一文种完成初稿生成
- **THEN** 系统产出必须被定性为待人工核稿的初稿
- **AND** 不得包含任何自动发文 / 自动提交的语义或动作
