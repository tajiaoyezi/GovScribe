## ADDED Requirements

### Requirement: 高频文种初稿生成请求 / 响应契约

系统必须（MUST）提供一个"高频文种初稿生成"接口，其入参为 c06 `doctype-classification` 输出的结构化场景上下文契约（含目标 capability、目标文种、代表子类、行文方向、置信度、用户原始场景描述、已补齐要素、缺失 / 未确认要素标记、内容密级（取值：非密 / 敏感 / 涉密）），本 change 仅处理目标 capability==c05 的请求，不在接口内重新发起文种判别、不复算分级、不自建密级判定，仅将上下文中的内容密级作为出站请求密级随 c01 窄抽象调用透传供 c02 路由。出参为符合该文种结构契约的规范正文（标题 / 称谓 / 主送 / 正文结构 / 落款）。出参限于正文内容，版式排版后置、不在本 change 输出（对齐 PRD Scope「输出规范正文内容，版式后置」与边界）。接口必须随响应回传所消费的文种 / 子类等上下文标识，供 c08 在线编辑消费。

#### Scenario: 提交 c06 场景上下文返回规范正文

- **WHEN** 调用方提交 c06 输出的、目标 capability==c05 的结构化场景上下文发起生成
- **THEN** 系统必须返回符合该文种结构契约的规范正文（含标题 / 称谓 / 主送 / 正文结构 / 落款各要素）
- **AND** 必须回传所消费的文种 / 子类等上下文标识，且不在本接口内重新判别文种或复算分级

#### Scenario: 输出不含版式排版

- **WHEN** 任一文种完成生成
- **THEN** 输出必须为规范正文内容，不得包含 GB/T 9704 版式 / 红头 / 字体字号等排版结果
- **AND** 版式相关处理必须留待后置环节，不在本接口出参内

#### Scenario: 未授权角色不可发起起草

- **WHEN** 某角色对本接口发起高频文种初稿起草
- **THEN** 系统必须先消费 c04 `rbac-authorization` 的授权决策（`draft.create` 权限点），未授予即拒绝（fail-closed），本 change 不自建 RBAC 判定
- **AND** 该操作层 RBAC 与 c03 落地的密级数据 ACL 预过滤两层正交叠加、不可相互替代，任一不通过即拒绝

### Requirement: 生成编排顺序组合 c06 上下文、c03 检索与 c01 模型调用

系统必须（MUST）按既定顺序编排一次初稿生成：① 消费 c06 `doctype-classification` 输出的结构化场景上下文契约，仅当目标 capability==c05 时进入本管线，不在本 change 重新判别文种、不复算是否标黄稀缺、不自行移交 c07；② 据上下文中的文种 / 子类与场景要素调用 `corpus-rag-retrieval`（c03）检索同类范文样例；③ 依 `doctype-prompt-templating` 的结构契约与样例编排规则拼装提示；④ 经 `model-provider-abstraction`（c01）窄抽象接口发起生成。系统必须只经 c01 窄抽象接口调用模型，不得直接 import 任何具体供应商 SDK 或 LiteLLM 客户端类型（对齐 c01 spec「上层包不出现具体 SDK 导入」）；对 c01 / c03 / c06 为消费关系，不得改变其对外契约。

#### Scenario: 消费 c06 上下文且检索先于生成

- **WHEN** 一次目标 capability==c05 的高频文种初稿生成被触发
- **THEN** 系统必须以 c06 上下文中的文种 / 子类为准，先调用 c03 完成范文样例检索，再调用 c01 发起生成
- **AND** 拼装提示必须使用 `doctype-prompt-templating` 的结构契约与样例编排规则，且不在本 change 内重新判别文种或复算分级

#### Scenario: 生成仅经 c01 窄抽象接口

- **WHEN** 对本 change 的生成编排模块执行依赖扫描
- **THEN** 其导入图中不得出现 `openai-go`、`anthropic-sdk-go` 或任何 LiteLLM 专有客户端类型
- **AND** 全部模型调用必须经 c01 的窄抽象接口发起

### Requirement: SSE 流式输出初稿

系统必须（MUST）经 SSE 将生成的初稿边生成边流式返回，以改善体感时延（对齐 PRD「速度」非功能需求与 ADR-0001 D2「初稿流式输出走 SSE」）。流式输出必须保持文本顺序、增量下发正文片段，并在正常结束时发出明确的完成事件；调用方据流式片段拼接得到的最终正文，必须与一次性生成的完整正文在内容上等价。

#### Scenario: 初稿边生成边流式下发

- **WHEN** 调用方以流式方式发起高频文种初稿生成
- **THEN** 系统必须经 SSE 按顺序增量下发正文片段，并在生成完成时发出完成事件
- **AND** 调用方拼接所有片段得到的正文必须与同参数一次性生成的正文内容等价

### Requirement: 生成中断与失败语义

系统必须（MUST）为流式生成定义明确的中断与失败语义。调用方主动中断（如用户取消、连接断开）时，系统必须及时停止该次生成并释放资源，已下发片段视为未完成初稿、不得被当作完成结果。底层模型调用失败或超时时，系统必须经流发出明确的错误事件并终止该流，不得静默挂起或返回半截正文却伪装为完成。

#### Scenario: 调用方主动中断

- **WHEN** 流式生成进行中调用方主动取消或连接断开
- **THEN** 系统必须停止该次生成并释放相关资源
- **AND** 已下发的片段必须被标识为未完成，不得标记为完成结果

#### Scenario: 底层调用失败发错误事件

- **WHEN** 生成过程中 c01 模型调用返回失败或超时
- **THEN** 系统必须经流发出明确的错误事件并终止该流
- **AND** 不得把已下发的半截正文标记为正常完成

### Requirement: 模型调用仅经 c01 窄抽象接口、脱敏与密级路由由 c02 拦截点承接

本 change 的模型调用必须（MUST）仅经 c01 `model-provider-abstraction` 窄抽象接口发起；脱敏与出站密级路由由 c01 公网分支上的 c02 强制拦截点承接，本 change 不直接面对公网、不自行实现或断言该脱敏 / 密级路由环节，仅保证不绕过 c01 窄抽象直连任何具体供应商 SDK 或 LiteLLM 客户端（对齐 ADR-0001 D7、c02 为脱敏 / 出站密级路由权威、c01 为唯一模型调用入口）。本 change 必须（MUST）消费 c06 结构化场景上下文契约中的内容密级（取值：非密 / 敏感 / 涉密），并在经 c01 窄抽象接口发起生成时将该内容密级作为出站请求密级随调用传递，供 c01 公网分支上的 c02 据以做出站密级路由判定；本 change 仅透传该密级输入值、不自建密级判定，不在本 change 复算或改判密级。

#### Scenario: 不绕过 c01 窄抽象直连 SDK

- **WHEN** 一次高频文种生成需发起模型调用
- **THEN** 该调用必须经 c01 窄抽象接口发起，由 c01 公网分支上的 c02 拦截点承接脱敏与出站密级路由
- **AND** 生成编排中不得存在绕过 c01 窄抽象直连具体供应商 SDK 或 LiteLLM 客户端的旁路

#### Scenario: 透传 c06 内容密级作为出站请求密级

- **WHEN** 一次高频文种生成消费 c06 结构化场景上下文契约并经 c01 窄抽象接口发起生成
- **THEN** 系统必须将上下文中的内容密级（取值：非密 / 敏感 / 涉密）作为出站请求密级随调用传递给 c01，供 c01 公网分支上的 c02 据以做出站密级路由判定
- **AND** 本 change 仅透传该密级输入值、不自建密级判定，不在本 change 复算或改判密级
