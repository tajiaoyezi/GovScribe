## Context

GovScribe 的核心用户是文秘与业务部门兼职写作者，后者往往不熟悉「同一件事用请示还是报告、用函还是通知」的文种区分，而文种一旦选错，称谓、主送机关、行文关系与口吻全篇皆错，必须返工。PRD Milestone 3「文种选择引导」要求：用户用自然语言描述写作场景，即被正确路由到对应文种 / 细分子类，减少选错文种。

当前状态：前序 change 已确立可被本 change 直接复用的能力——c01 提供统一模型调用入口（LiteLLM Proxy，窄抽象屏蔽供应商）；c02 提供脱敏网关 desensitization-gateway 与出站密级路由 security-classification-routing（判别调用经 c01 入口透传至 c02 处置）；c05 提供 A 表 9 个深做文种的高频生成；c07 提供通用兜底生成。本 change 介于「用户输入」与「正文生成」之间，是入口判别层，不生成正文。

权威约束来源：
- PRD「文种覆盖矩阵」——5 档能力阶梯（① 模板生成 / 深做 → ② 模版辅助写 → ③ 框架写 → ④ 待计划训练 → ⑤ 通用兜底）、A 表 9 个深做文种及代表子类、B 表其余文种按能力档归类、标黄子类（<100 条）不承诺深做。
- ADR-0001 决策：D1（Go 应用层 + 窄抽象，重 ML 算子外置）、D3（MVP 走 LiteLLM Proxy，信创切 Go 直连官方 SDK，业务代码不动）、D5（PostgreSQL 权威源 + 访问控制判定）、D7（脱敏网关 fail-closed，涉密永不降级）、D10（单租户 + RBAC 落 Postgres）。
- proposal.md 与本 change 两个 spec（doctype-classification、slot-clarification）已固化的需求边界。

相关方：业务兼职用户（主受益者，文种引导降低选错门槛）、文秘（核心场景）、c01（统一模型调用入口）、c02（脱敏网关 + 出站密级路由，对判别调用做处置）、c05 / c07（下游契约消费方，需就场景上下文字段对齐）、平台管理员（维护判别提示词、文种/子类映射、文种能力档分级表、必需要素清单等配置）。

## Goals / Non-Goals

**Goals:**

- 让用户仅凭自然语言场景描述发起写作，由 Agent 完成文种 / 代表子类判别并给出归一化置信度，判别口径严格对齐 PRD「文种覆盖矩阵」5 档阶梯。
- 低置信或多义时返回按置信度降序的 Top-N 候选供一键确认，绝不静默猜测；用户改选可覆盖模型判别。
- 按能力档分流：A 表 9 个深做文种路由到 c05，B 表其余文种 / 标黄稀缺子类 / 无法归类一律路由到 c07，全程无「无法处理」死路。
- 依目标文种的必需要素清单校验完整度，缺关键要素时发起有限轮次、针对性的澄清式追问；要素齐备、达轮次上限或用户显式跳过即放行。
- 产出结构化场景上下文契约（文种 / 子类 / 行文方向 / 置信度 / 已补齐要素 / 缺失标记 / 原始描述），交给 c05 / c07 直接消费，下游不重复判别。

**Non-Goals:**

- 不生成任何文种正文，不触发正文的 SSE 流式输出（属 c05 / c07）；判别与澄清交互按请求-响应处理，不要求流式。
- 不检索范文库、不调用 RAG（属 c05 / c07 的生成阶段）。
- 不另立保密口径：判别若需调模型，经 c01 统一模型调用入口透传至 c02 的脱敏网关 + 出站密级路由；本 change 不维护第二套脱敏 / 密级逻辑。
- 不新增权限模型：沿用 D10 的 RBAC，不引入新的角色或资源类型。
- 不实现 c05 / c07 的生成逻辑，只定义并交付供其消费的契约。
- 不覆盖 Post-MVP 项（GB/T 9704 版式、OA 集成、智能核稿、私有化训练平台）。

## Decisions

### D-06-1：判别用 LLM 分类而非规则 / 关键词匹配，调用收口到 c01 窄抽象

文种判别本质是「自然语言场景 → 文种 / 子类」的语义映射，关键词规则对「把某事处理情况向上级讲清楚」这类既可判报告又可判请示的多义场景无能为力。决策以 LLM 意图分类为主：在 Go 应用层组装一段判别提示词（提供文种覆盖矩阵作为受限标签集 + 要求输出文种 / 代表子类 / 行文方向 / 归一化置信度的结构化 JSON），经 c01 的统一模型接入层调用。

依 ADR-0001 D3，MVP 该调用走 LiteLLM Proxy，信创阶段切 Go 直连官方 SDK，业务代码不动——本 change 只依赖 c01 暴露的窄抽象接口（D1），不感知底层供应商。判别属重 ML 推理之外的轻编排，留在 Go 进程内（D1 边界：embedding / rerank / NER 才外置）。

- 备选 1：纯关键词 / 正则规则表。否决——无法处理多义与长尾表述，且文种覆盖矩阵子类繁多，规则维护成本高于收益。
- 备选 2：训练专用分类小模型。否决——MVP 公网阶段靠提示工程 + 受限标签集即可（与 PRD「公网提质走 RAG + 提示工程、不微调」一致），专用模型属 Post-MVP 训练平台范畴。
- 取舍：LLM 判别有误判与不稳定风险，由 D-06-2 的 Top-N 候选 + 用户确认兜住，不追求一次判准。

### D-06-2：低置信 / 多义返回 Top-N 候选，用户选择优先级高于模型

设定置信度阈值与多义判定（Top-1 与 Top-2 置信度差小于既定间距视为多义）。命中其一即返回按置信度降序的 Top-N 候选（文种 + 代表子类 + 置信度），等待用户一键确认，绝不静默选定单一文种直接路由（doctype-classification spec 强约束）。用户确认或改选后，以用户最终选择覆盖模型判别结果继续路由——人是最终把关者，符合产品「辅助起草、人工最终把关」定位。

- 备选：始终取 Top-1 直接路由。否决——多义场景静默猜测会把选错文种的风险从用户转嫁为 Agent 的隐性错误，违背本 change「降低选错」初衷。
- 取舍：多一次确认交互换取路由准确性；高置信单义场景可直接放行不打断用户，阈值具体取值列入 Open Questions 待实测定档（与 PRD「指标阈值待实测定档」口径一致）。

### D-06-3：按能力档分流，A 表→c05 / 其余一律→c07，无死路

路由表 = c06 拥有的「文种能力档分级表」（owner=c06，作为只读共享配置，来源 PRD 文种覆盖矩阵）：判别命中 A 表 9 个深做文种（通知 / 请示 / 报告 / 函 / 会议纪要 / 通报 / 批复 / 讲话稿 / 方案）→ 目标 capability 标 c05；命中 B 表其余文种（模版辅助写 / 框架写 / 待计划训练档）、A 表内的标黄稀缺子类（<100 条，如方案-调研 / 方案-会议）、或无法稳定归类的任意场景 → 目标 capability 标 c07，不承诺深做质量。标黄稀缺的降级移交判定由 c06 在路由阶段查分级表**一次性**完成、只发生在 c06（c05 / c07 不再复算是否命中深做）。路由结果是结构化标签（目标 capability、文种、子类、行文方向），不是直接调用 c05 / c07，调用由上层编排在放行后发起。

「无法归类也兜底而非报错」是硬规则：无法归类时降级标 c07，绝不向用户返回「无法处理」（对齐 PRD Milestone 4「全程不出现无法处理的死路」）。

- 备选：把稀缺子类也送 c05 深做。否决——违反 PRD「标黄文种不纳入 MVP 深做、不承诺深做质量」决策，会让 UAT 难以区分是模型问题还是缺数据问题。
- 取舍：B 表与稀缺子类的初稿质量低于 A 表深做，这是 MVP 已确认的能力边界，由 c07 兜底质量而非本 change 解决。

### D-06-4：行文方向由文种 + 机关关系规则推断，作为契约一等字段

行文方向（上行 / 下行 / 平行）影响下游正文的称谓与口吻，必须在判别阶段确定并写入契约。采用「文种默认方向 + 机关关系修正」的轻规则：请示 / 报告默认上行，批复 / （下行）通知默认下行，函默认平行；再依场景中的机关关系线索（「向上级请求」→上行、「对下级答复批复」→下行、「与同级商洽」→平行）修正。规则与 LLM 输出的方向取并，冲突时以规则为准并降低置信度。

- 备选：完全交由 LLM 输出方向。否决——方向是强结构化、强领域规则的判断，规则更稳定可控；LLM 输出仅作线索。
- 取舍：规则需随文种维护，但条目有限（首期仅 9 深做文种 + B 表代表文种），可接受。

### D-06-5：必需要素清单可配置，按文种 + 行文方向维护

必需要素清单（发文单位 / 主送机关 / 事由-关键事项 / 关键时间地点等）随文种与行文方向可配置维护（slot-clarification spec 要求），存为可维护配置而非硬编码——便于客户按实际公文习惯调整、便于随文种扩展。校验分两步：先从用户原始场景描述抽取已知要素（复用 D-06-1 同一次 LLM 判别的结构化输出或追加一次轻抽取），再比对清单识别缺失关键要素，作为是否发起澄清的依据。

- 备选：要素清单硬编码进代码。否决——不同客户的请示主送层级、发文单位口径差异大，硬编码不可维护。
- 取舍：配置需初始化一份默认清单（首期覆盖 9 深做文种），随交付迭代。

### D-06-6：澄清有限轮次，针对缺失要素具体提问，三条件任一放行

澄清必须针对具体缺失要素提问（「这份请示主送哪个上级单位？」），而非泛泛「补充更多信息」。设澄清轮次上限，达上限即停止追问、进入放行判定，避免无限循环。放行触发条件三者任一：要素齐备 / 达轮次上限 / 用户显式跳过。放行时若仍有要素未补齐，必须在场景上下文中标记为「缺失 / 未确认」，提示下游 c05 / c07 谨慎处理、不凭空臆造关键信息（slot-clarification spec 强约束）。

- 备选：要素不齐就阻断、不放行。否决——会卡死用户，违背「辅助起草」体验；标记缺失 + 下游谨慎处理是更优解。
- 取舍：放行带缺失要素可能让初稿留有占位，但优于强制用户填满或卡死；下游据缺失标记规避臆造。

### D-06-7：结构化场景上下文作为唯一下游契约，下游不重复判别

判别 + 澄清的全部产出收敛为单一结构化场景上下文（字段固定为 9 项：目标 capability、目标文种、代表子类、行文方向、置信度、用户原始场景描述、已补齐要素、缺失/未确认要素标记、内容密级），作为交给 c05 / c07 的契约。c05 / c07 必须以契约中的文种 / 子类 / 行文方向为准继续生成，不再重新发起文种判别（doctype-classification spec 强约束）。三方需就字段对齐，契约定义归本 change 维护。

内容密级（非密 / 敏感 / 涉密）来源：由写作发起方在发起时确定——用户 / 文秘显式选择，或按可配置的部门默认派生；具体定级权限与策略归客户密级策略（对齐 PRD Open Question「密级策略由客户设定」），本 change 只落"字段 + 强制传递 + fail-closed"，不替客户判定密级口径。该字段随契约交给 c05 / c07，由其经 c01 窄抽象发起生成时作为出站请求密级随调用传递，供 c01 公网分支上的 c02 出站密级路由判定。内容密级缺失或未知时按涉密 fail-closed 由 c02 兜底（强制转私有，无可用私有则阻断，涉密永不降级并入审计），本 change 不以缺省"非密"放行。

- 备选：c05 / c07 各自再判一次以防错。否决——重复判别浪费模型调用、可能与本 change 结论不一致，且违反契约单一来源原则。
- 备选：内容密级由 c06 / c05 / c07 自行判定。否决——定级权限与策略归客户密级策略，本组 change 只负责承载与强制传递该字段，不另立密级判定口径。

## Risks / Trade-offs

- **[LLM 文种误判 / 子类张冠李戴]** → Top-N 候选 + 用户一键确认兜底（D-06-2）；高置信单义才直接放行，阈值实测定档；用户改选始终优先于模型。
- **[置信度阈值取值不当：过高频繁打断、过低静默误判]** → MVP 不预设硬阈值，与 PRD「指标阈值待实测定档」一致，首版实测后与客户共同校准（见 Open Questions）。
- **[判别调模型时敏感 / 涉密场景描述出域]** → 不另立口径，判别调用经 c01 统一模型调用入口透传至 c02 的脱敏网关 desensitization-gateway + 出站密级路由 security-classification-routing 处置（D-06-1、proposal）：默认 fail-closed，外置 NER 不可用时绝不静默发原文；涉密恒走私有线、永不降级（ADR-0001 D7）。本 change 仅处理用户输入的场景文本与要素，不调用公网模型生成正文、不检索范文库，出域面小于生成阶段，但仍须经同一网关，不得旁路。
- **[密级防越权]** → 场景描述与已补齐要素可能含敏感信息，两层密级控制各有归属、本 change 均消费而不自建：① 出站密级路由（出公网走哪条模型路径）归 c02 的 security-classification-routing；② 数据可见 / 可外发（能看 / 能外发什么数据）归 c04 的 access-control-composition 密级臂叠加 RBAC，权威实现为 Postgres 预过滤 ACL（ADR-0001 §2 密级防越权，后端注入过滤条件）。D10 在本 change 仅用于 RBAC / 认证 / 单租户边界，不承载密级路由。本 change 不绕过、不新增密级例外；澄清回填的要素同样纳入上述判定，不因「只是澄清」而降级处理。
- **[三库一致性影响]** → 本 change 以判别 / 路由 / 澄清为主，写入面窄：① PostgreSQL——若需落判别与澄清的会话状态 / 审计（出公网脱敏与密级路由降级 / 阻断的审计留痕由 c02 脱敏网关审计承载，本 change 不自建审计表），按 D5 以 Postgres 为权威源，单库事务内完成，无跨库一致性问题；② Milvus——本 change 不读写向量库（不检索范文），零影响；③ MinIO——本 change 不产出正文文件、不读写范文 / 生成稿桶，零影响。结构化场景上下文作为内存 / 请求态契约移交下游，不引入新的跨库写入，规避了三库不一致风险。
- **[无法归类被迫兜底导致 c07 承载过载 / 初稿质量低]** → 这是 MVP 已确认的能力边界（PRD：稀缺与长尾走通用兜底、不承诺深做）；本 change 保证「不报错」，质量由 c07 负责，并通过判别置信度与路由标签为后续扩深做文种提供数据。
- **[必需要素清单 / 路由表 / 判别提示词配置漂移]** → 三者均为可维护配置（D-06-3 / D-06-5），与文种覆盖矩阵同源；初始化一份覆盖 9 深做文种的默认配置，变更走配置管理，避免散落硬编码。
- **[澄清无限循环或体验卡死]** → 轮次上限 + 三条件任一放行 + 缺失标记移交（D-06-6），保证有界且不卡死用户。

## Migration Plan

本 change 为新增能力，无既有数据 / schema 迁移，部署以「配置就位 + 契约对齐 + 灰度」为主：

1. 初始化判别配置 → 验证：文种覆盖矩阵映射表（A 表 9 文种 + 代表子类、B 表能力档归类、标黄稀缺子类标记）、判别提示词、行文方向规则表、各文种必需要素清单落入可维护配置，与 PRD 矩阵逐项核对一致。
2. 对齐下游契约 → 验证：与 c05 / c07 就结构化场景上下文字段（capability / 文种 / 子类 / 行文方向 / 置信度 / 原始描述 / 已补齐要素 / 缺失标记 / 内容密级）签字段定义，c05 / c07 能按契约消费且不重复判别，并约定把内容密级作为出站请求密级随 c01 调用透传供 c02 路由。
3. 接入 c01 模型层与 c02 脱敏 / 密级路由 → 验证：判别调用经 c01 窄抽象发起、透传至 c02 的 desensitization-gateway + security-classification-routing，敏感 / 涉密场景描述确实经脱敏网关 + 出站密级路由（fail-closed、涉密走私有），降级 / 阻断由 c02 脱敏网关审计留痕。
4. 前端入口接入 → 验证：写作工作台场景描述入口、Top-N 候选确认、澄清追问交互可用；空 / 过短描述被拒绝并提示而非凭空判别。
5. 端到端灰度 → 验证：覆盖典型场景（请示-上行经费、通知-下行会议、多义报告/请示、稀缺方案-调研降级 c07、无法归类兜底 c07、要素缺失澄清后回填、用户跳过带缺失标记放行），路由与契约符合各 spec Scenario。

回滚策略：本 change 不改既有数据结构，回滚即关闭场景描述入口、恢复用户手选文种的旧路径（若已有），或停用判别服务由前端直接让用户选文种后送 c05 / c07；配置与契约定义保留，下次重新启用无数据损失。

## Open Questions

- 置信度阈值与「多义」判定的间距具体取值——MVP 先经验取值、首版实测后与客户共同定档（对齐 PRD「指标阈值待实测定档」）。
- Top-N 的 N 取值（候选展示个数）与澄清轮次上限的具体数值，待结合实测打断率 / 补齐率确定。
