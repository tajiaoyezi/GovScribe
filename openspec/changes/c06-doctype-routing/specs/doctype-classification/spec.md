## ADDED Requirements

### Requirement: 自然语言场景描述入口

系统必须（MUST）允许用户以自然语言陈述写作意图（如"想给上级单位申请一笔活动经费""通知各科室周五开年度民主生活会"）发起写作，而**不要求**用户预先从 9 个深做文种或大量细分子类中手动选定文种。系统应当接收该场景描述文本作为文种判别的唯一必填入口，文种 / 子类的确定由后续判别能力完成。

#### Scenario: 用户仅凭场景描述发起写作

- **WHEN** 业务兼职用户在写作工作台输入自然语言场景描述且未选择任何文种
- **THEN** 系统接受该输入并触发文种判别流程，不向用户强制弹出文种选择列表作为前置必选项

#### Scenario: 场景描述为空或过短无法判别

- **WHEN** 用户提交的场景描述为空或短到不含任何可判别的事由 / 行文意图线索
- **THEN** 系统必须拒绝直接路由，并提示用户补充"想写什么、给谁、为了什么事"，不得凭空猜测文种

### Requirement: 基于 c01 模型层的文种与子类判别

系统必须（MUST）复用 c01 的统一模型接入能力对场景描述做意图分类，依 PRD「文种覆盖矩阵」将场景映射到**文种**与**代表子类**，并输出归一化的置信度。文种判别口径必须对齐矩阵的 5 档能力阶梯：A 表 9 个深做文种（通知 / 请示 / 报告 / 函 / 会议纪要 / 通报 / 批复 / 讲话稿 / 方案）走深做档；B 表其余文种按其所属能力档（模版辅助写 / 框架写 / 待计划训练）标注。判别调用涉及模型时必须经 c01 统一模型调用入口透传至 c02 的脱敏网关 desensitization-gateway 与出站密级路由 security-classification-routing 处置，本能力不得另立保密口径，也不得调用公网模型生成正文或检索范文库。

#### Scenario: 命中 A 表深做文种并给出子类

- **WHEN** 用户描述"区政府想成立一个节能监测中心、需向上级请求批准"
- **THEN** 系统判别文种为"请示"、代表子类为"组织成立"，并返回该判别的置信度

#### Scenario: 命中 B 表非深做文种并标注能力档

- **WHEN** 用户描述"要发布一则任免某同志职务的命令"
- **THEN** 系统判别文种为"命令"，标注其能力档为"模版辅助写"（非 A 表深做），以便路由能力据此分流

#### Scenario: 判别调用经由 c02 脱敏网关与出站密级路由

- **WHEN** 场景描述含敏感实体且需调用模型完成意图分类
- **THEN** 系统必须经 c01 统一模型调用入口透传至 c02 的脱敏网关 + 出站密级路由处置该调用，涉密内容恒走私有线、绝不静默发送原文

### Requirement: 低置信与多义时返回 Top-N 候选

当判别置信度低于既定阈值，或场景同时强匹配多个文种（多义）时，系统必须（MUST）返回按置信度降序排列的 Top-N 候选文种 / 子类供用户一键确认，而**不得**静默选定单一文种直接路由。用户确认或改选后，系统应当以用户最终选择覆盖模型判别结果继续路由。

#### Scenario: 多义场景返回候选供确认

- **WHEN** 用户描述"把某事的处理情况向上级讲清楚"，既可判为"报告"又可判为"请示"
- **THEN** 系统返回 Top-N 候选（含"报告""请示"及其代表子类与置信度），等待用户一键确认而非自行选定

#### Scenario: 用户改选覆盖模型判别

- **WHEN** 模型置信度最高项为"报告"，但用户在候选中确认选择"请示"
- **THEN** 系统以用户选择的"请示"作为最终文种继续后续路由与要素校验

### Requirement: 按能力档分流路由至 c05 或 c07

系统必须（MUST）依判别结果将场景路由到下游生成能力，**全程不得出现"无法处理"的死路**。命中 A 表 9 个深做文种时必须路由到 c05 高频文种生成；落在 B 表其余文种、标黄稀缺子类（<100 条数据）或无法归类的任意场景时，必须路由到 c07 通用兜底。路由结果必须为结构化标签，含目标 capability（c05 / c07）、文种、子类、行文方向，并随场景上下文一并移交。

#### Scenario: 深做文种路由到 c05

- **WHEN** 文种判别为 A 表的"通知"且子类为"召开会议"
- **THEN** 系统将目标 capability 标注为 c05（高频文种深做生成）并移交场景上下文

#### Scenario: 稀缺子类降级路由到 c07

- **WHEN** 文种命中"方案"但子类为标黄稀缺的"调研方案"（<100 条数据）
- **THEN** 系统将目标 capability 标注为 c07 通用兜底，不承诺"深做"质量

#### Scenario: 无法归类仍兜底而非报错

- **WHEN** 用户场景无法稳定归入任一已规划文种
- **THEN** 系统必须路由到 c07 通用兜底，绝不向用户返回"无法处理"

### Requirement: 输出结构化场景上下文契约

判别完成后，系统必须（MUST）产出结构化场景上下文作为交给下游 c05 / c07 的契约，字段固定为 9 项：目标 capability（c05 或 c07）、目标文种、代表子类、行文方向（上行 / 下行 / 平行）、置信度、用户原始场景描述、已补齐要素、缺失 / 未确认要素标记、内容密级（非密 / 敏感 / 涉密）。下游 c05 / c07 应当直接消费该上下文，不再重复做文种判别。行文方向必须依文种与机关关系推断（如"向上级请求"为上行、"对下级答复批复"为下行、"与同级商洽"为平行）。

内容密级由写作发起方在发起时确定——用户 / 文秘显式选择，或按可配置的部门默认派生；具体定级权限与策略归客户密级策略，本能力只负责承载该字段并强制随契约传递，不替客户判定密级口径。系统必须（MUST）将内容密级随场景上下文交给 c05 / c07；c05 / c07 经 c01 窄抽象发起生成时，须把内容密级作为出站请求密级随调用传递，供 c01 公网分支上的 c02 路由判定。当内容密级缺失或未知时，下游网关必须按涉密 fail-closed 处置，本能力不得以缺省"非密"放行。

#### Scenario: 产出含行文方向的结构化上下文

- **WHEN** 文种判别为"请示"、子类"资金费用申请"、场景为向上级单位申请经费
- **THEN** 系统产出结构化场景上下文，目标 capability 标为 c05，行文方向标注为"上行"，并随附目标文种、代表子类、置信度、原始描述、已补齐要素、缺失 / 未确认要素标记与内容密级（非密 / 敏感 / 涉密，共 9 项字段）

#### Scenario: 下游不重复判别

- **WHEN** c05 或 c07 接收到本能力产出的结构化场景上下文
- **THEN** 下游必须以上下文中的文种 / 子类 / 行文方向为准继续生成，不再重新发起文种判别

#### Scenario: 内容密级随契约传递供出站路由判定

- **WHEN** 写作发起方在发起时确定内容密级（用户 / 文秘显式选择或按部门默认派生）并随场景描述提交
- **THEN** 系统将内容密级（非密 / 敏感 / 涉密）写入结构化场景上下文随契约交给 c05 / c07，由 c05 / c07 经 c01 窄抽象发起生成时把它作为出站请求密级随调用传递，供 c02 路由判定

#### Scenario: 内容密级缺失按涉密 fail-closed 不缺省非密

- **WHEN** 移交下游的场景上下文中内容密级缺失或未知
- **THEN** 系统不得以缺省"非密"放行公网，下游网关必须按涉密 fail-closed 处置（强制转私有，无可用私有则阻断，涉密永不降级并入审计）
