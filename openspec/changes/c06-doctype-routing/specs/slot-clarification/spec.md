## ADDED Requirements

### Requirement: 依目标文种校验必需要素完整度

文种判别完成后、放行至下游生成前，系统必须（MUST）依**目标文种的必需要素清单**校验已知场景信息的完整度。必需要素清单应当随文种 / 行文方向可配置维护，典型必需要素包括：发文单位、主送机关、事由 / 关键事项、关键时间地点。系统必须从用户原始场景描述中抽取已知要素、识别缺失的关键要素，校验结果作为是否发起澄清的依据。

#### Scenario: 要素齐备直接放行

- **WHEN** 用户场景已含发文单位、主送机关、事由与关键事项，目标文种为"请示"
- **THEN** 系统判定必需要素齐备，不发起澄清追问，直接放行至下游 c05 生成

#### Scenario: 识别出缺失关键要素

- **WHEN** 用户描述"想申请一笔活动经费"但未给出主送机关与具体金额 / 事由细节
- **THEN** 系统按"请示"必需要素清单识别出"主送机关""关键事项"缺失，标记为需澄清

### Requirement: 有限轮次澄清式追问补齐要素

当关键要素缺失时，系统必须（MUST）发起**有限轮次**的澄清式追问以补齐要素，追问必须（MUST）针对缺失要素提出具体问题，而非泛泛要求"补充更多信息"。系统必须设定澄清轮次上限，达到上限后即便要素仍不全也必须停止追问、避免无限循环。澄清交互不要求流式输出（正文生成的 SSE 流属 c05 / c07）。

#### Scenario: 针对缺失要素发起具体追问

- **WHEN** 系统识别"请示"场景缺失"主送机关"
- **THEN** 系统发起澄清提问"这份请示主送哪个上级单位？"，而非笼统要求补充信息

#### Scenario: 达到轮次上限停止追问

- **WHEN** 澄清追问已达到设定的轮次上限但部分要素仍未补齐
- **THEN** 系统必须停止继续追问，进入放行判定，不得无限循环提问

#### Scenario: 用户补齐后回填场景上下文

- **WHEN** 用户在澄清追问中补充了主送机关与事由
- **THEN** 系统必须将补齐的要素回填到结构化场景上下文，供下游 c05 / c07 消费

### Requirement: 用户显式跳过澄清后放行

系统必须（MUST）允许用户在澄清过程中**显式跳过**剩余追问。要素齐备、达到轮次上限、或用户显式跳过三者任一成立时，系统应当放行至下游生成。放行时若仍有未补齐的要素，系统必须在移交的场景上下文中标记这些要素为"缺失 / 未确认"，以便下游 c05 / c07 在生成时谨慎处理、不凭空臆造关键信息。

#### Scenario: 用户跳过澄清直接放行

- **WHEN** 用户在澄清追问中选择"跳过、先生成初稿"
- **THEN** 系统停止追问并放行至下游生成，将未补齐要素标记为"缺失 / 未确认"随上下文移交

#### Scenario: 放行时标记未确认要素

- **WHEN** 达到轮次上限后"关键时间地点"仍未补齐即放行
- **THEN** 系统移交的场景上下文必须将"关键时间地点"标注为缺失，提示下游不得臆造该要素
