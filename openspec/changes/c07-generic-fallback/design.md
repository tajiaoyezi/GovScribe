## Context

GovScribe 的核心价值假设（PRD Hypothesis）要求"覆盖高频文种**且对任意文种有通用兜底**"，PRD Milestone 4 把"任意非高频场景均可走通用公文生成，全程不出现'无法处理'"列为独立交付目标。文种覆盖矩阵的能力阶梯第 ⑤ 档「通用兜底」明确：任意「其它 XX」或上述档位均无法满足的场景，一律走通用公文生成。

当前态（前序 change 已确立）：

- c01 已提供模型供应商窄抽象接口与 `model-streaming-passthrough` 统一 SSE 流式透传（传输层）；公文生成的模型调用链路唯此一条。
- c02 已提供脱敏网关（fail-closed、涉密恒私有、映射表不出域）与密级路由。
- c03（可选）提供 Milvus 范文库混合检索（BM25 + 稠密向量），按密级 partition 与文种标量过滤召回样例。
- c04 提供 RBAC + 密级两层正交判定与审计 / PostgreSQL 元数据落库机制。
- c05 深做 9 个文种（通知 / 请示 / 报告 / 函 / 会议纪要 / 通报 / 批复 / 讲话稿 / 方案），其 `draft-generation-streaming` 是生成编排管线（检索 → 拼装 → 调 c01 → SSE）的载体。
- c06 文种路由：唯一的文种判别 / 分流 / 降级判定源，把用户自然语言场景归类后产出结构化场景上下文契约，并落定目标 capability（c05 或 c07）；稀缺降级与无法归类直接标 c07，低置信 / 多义由 c06 走强制 Top-N 用户确认而非静默路由。

约束：c05 + c06 只覆盖矩阵的一部分；矩阵 B 档的「模版辅助写 / 框架写 / 待计划训练」诸文种、标黄数据稀缺文种（<100 条）、以及 c06 无法可靠归类的描述，若直接返回"无法处理"，会在试点中制造死路，使价值假设当场崩塌。本 change 是这条兜底底线的落地：让产品对任意合法公文请求都产出一份"形似公文"的初稿，同时诚实标注为通用兜底、不承诺深做质量。本 change 落在 MVP 范围内（Milestone 4）。

干系人：试点客户文秘（核心用户，主要消费兜底稿并人工把关）、业务兼职用户（D10 角色，权限较窄，文种引导 + 通用兜底起草是其主路径）、产品 / 数据侧（依赖兜底标记区分采纳率统计口径）。

## Goals / Non-Goals

**Goals:**

- 建立通用兜底生成路径，作为与 c05 共用的生成编排管线上的一个 prompt 组装分支：消费 c06 结构化场景上下文契约的目标 capability，在目标 capability == c07 时进入兜底，不在 c07 内复算分流降级判定。
- 提供统一的通用公文要素骨架（标题 / 主送机关 / 开头缘由 / 主体事项 / 结尾要求 / 落款），套用于任意场景，保证结构完整性，缺信息处给占位提示而非编造。
- 在 c03 可用时注入语义相近范文样例作 few-shot 提质；不可用 / 无命中时退回纯骨架 + 通用提示工程，绝不阻断。
- 落地"无死路"不变量：任意合法公文请求必落到「深做 / 已规划档 / 通用兜底」之一并产出初稿，仅在明确界定的系统级失败边界下报错。
- 兜底透明：初稿携带「通用公文兜底」标记，前端展示预期口径（未经深做、需人工重点把关），生成元数据可区分兜底稿与深做稿，兜底触发 / 降级留可观测与审计。

**Non-Goals:**

- 不深做任何文种（属 c05），不提升兜底稿到"可直接交付"质量；兜底显式不承诺深做质量。
- 不做文种分类 / 路由（属 c06）；本 change 只消费路由结论，不自行重做文种判定。
- 不修改 c01 生成、c02 脱敏 / 密级路由、c03 检索、c04 RBAC / 审计的既有需求级行为；对它们是复用 / 可选依赖 / 边界衔接关系。
- 不放宽任何安全约定：不为"制造"一份初稿而绕过脱敏或密级路由。
- 不涉及 GB/T 9704 版式、OFD、私有化 / 信创部署（均 MVP 后置，口径见 ADR-0001 D8 / D9 与 PRD Out of scope）。

## Decisions

### D-A 兜底是与 c05 共用的生成编排管线上的一个 prompt 组装分支，不是新链路

**决策**：通用兜底不新建模型调用链路，而是在公文生成服务内表现为一个"prompt 组装策略"分支——与 c05 的 `draft-generation-streaming` 共用同一条生成编排管线"检索 → 拼装提示 → 调 c01 窄抽象（其公网分支由 c02 强制拦截点承接脱敏 / 密级路由）→ 经 `model-streaming-passthrough` SSE 透传"，仅在"用哪套 prompt 模板 / 是否注入专属 few-shot"这一步分叉。c01 在此为传输层（窄抽象 + 流式透传），不持有前端生成端点。模型接入版本 / 许可口径完全沿用 **ADR-0001 D3**（MVP LiteLLM Proxy 统一 OpenAI / Anthropic / 私有兼容端点，信创切 Go 直连官方 SDK，业务代码不动），本 change 不另立。

c07 经 c01 窄抽象发起生成时，必须消费 c06 契约中的「内容密级（非密 / 敏感 / 涉密）」并将其作为出站请求密级随调用透传供 c01 公网分支上的 c02 做出站密级路由，与 c05 同口径、不自建密级判定、不改写 / 不降级；字段名 / 取值与 c06 产出一致。

**理由**：兜底与深做的差异只在"知识与模板"（深做有专属文种模板 + 充足范文，兜底只有通用骨架 + 可得样例），不在"如何调模型"。复用同一下游管线天然继承 c02 fail-closed / 密级路由与 c01 SSE 契约，是落地"兜底不放宽安全约定"和"统一 SSE 流式"两条需求的最省力做法。

**备选**：(1) 为兜底单独起一条模型调用链路——被否，会复制 c01 / c02 逻辑、产生第二套需维护的安全边界，违反 ADR-0001 D3 的窄抽象单链路初衷。(2) 把兜底做成 c06 路由的一个输出分支并在路由层直接生成——被否，越界到 c06，且把生成耦合进路由。

### D-B 触发判定纯消费 c06 目标 capability，本 change 不复算分流降级

**决策**：兜底触发判定的输入只有 c06 结构化场景上下文契约的「目标 capability」字段。判定为纯函数式映射：目标 capability == c07 即进兜底，== c05 即交 c05。c07 **不**复算"是否命中 9 个深做文种""是否命中 B 档且无专属模板"，也**不**依赖任何「无明确文种」信号字段（c06 契约无此字段）；稀缺降级移交在 c06 路由阶段查「文种能力档分级表」一次性落定为 c07，低置信 / 多义由 c06 走强制 Top-N 用户确认而非自动进 c07。c07 仅在 c06 最终落定目标 capability == c07 时进兜底。

**理由**：c06 是唯一的文种判别 / 分流 / 降级判定源，避免与 c06 重复维护两套分类 / 降级逻辑（与 ADR-0001 "窄抽象、不另立第二套口径"的总原则一致）；触发判定退化为对单一契约字段的纯函数映射，可测、可审计。9 个深做文种集合的归属与稀缺分级表均在 c06，不在本 change 内复制清单语义或档级判定。

**备选**：(1) 兜底侧自带一个轻量文种识别做"二次确认"——被否，越界且制造分类口径分歧。(2) c07 内对低置信请求自动兜底——被否，低置信归 c06 强制用户确认，自动兜底会绕过用户确认且与 c06 单一判定源矛盾。

### D-C 通用公文要素骨架 + 行文方向自适应，缺信息占位不编造

**决策**：骨架固定六要素——标题、主送机关、开头缘由、主体事项、结尾要求、落款（发文机关署名 + 成文日期）。当 c06 给出行文方向（上行 / 平行 / 下行）或子类线索时，骨架的称谓与结尾用语据此适配（如上行用"请示性"结尾、平行用"商洽 / 函复"语气）；无线索退回中性通用表述。无明确主送机关时在主送位给"需人工补全"占位提示，日期缺失同理给占位提示——**绝不虚构受文机关或编造成文日期**。

**理由**：弥补"无专属文种模板时不像公文"是兜底的核心价值；占位而非编造，既保证结构完整又守住"不编造机关口径"的公文红线，且与 PRD"输出规范正文内容（标题 / 称谓 / 主送 / 正文 / 落款符合行文规范）"一致。行文方向自适应让兜底在有部分路由线索时不至于完全中性、提升像公文程度，但不依赖具体文种深做。

**备选**：(1) 每个 B 档文种各做一套轻骨架——被否，逼近 c05 深做、超出兜底范围与本里程碑。(2) 主送 / 日期缺失时由模型自行补全一个合理值——被否，违反"不编造受文机关"红线。

### D-D RAG 样例为"可选增强"，遵循 c02/c04 密级与访问控制，无样例不阻断

**决策**：c03 可用时检索与场景语义相近的范文样例，作 few-shot 注入并在元数据标记「命中 RAG 样例」；c03 不可用 / 无命中 / 相关度不足时退回纯骨架 + 通用提示工程，标记「未命中 RAG 样例」，不阻断不失败。样例检索与取用走 **c03 既有混合检索**（ADR-0001 D4 Milvus 2.6.x BM25 + 稠密混合检索）并**强制经 c04 RBAC + c02 密级路由过滤**——密级防越权以 **ADR-0001 D5 / §2 架构 PostgreSQL 预过滤为权威 ACL**，Milvus 标量 / partition 过滤仅作性能纵深，过滤条件由后端注入、前端不可传任意表达式。

**理由**："有则提质、无则保命"是兜底相对深做的本质差异——深做以充足范文 + 专属模板为前提，兜底必须在范文库空 / 未到位时仍能产出初稿（呼应 PRD 风险"库空导致 UAT 说不清"的缓解）。样例取用复用 c03 / c04 既有判定而非另开取数口径，保证兜底不成为越权读范文的旁路。

**备选**：把 RAG 设为兜底的硬依赖——被否，违反"无样例退回骨架不阻断"需求，且与可选依赖 c03 的边界矛盾。

### D-E "无死路"不变量与系统级失败边界

**决策**：任意进入起草的合法公文请求必落到「深做 / 已规划档 / 通用兜底」之一并产出初稿。仅三类系统级失败允许不产出初稿并明确报错：(1) c02 脱敏网关 fail-closed 阻断（外置 NER 不可用且按密级不允许降级、又无私有模型可用，按 **ADR-0001 D7**）；(2) c01 模型层不可用；(3) 请求本身非合法公文写作请求。这些边界下必须返回明确可读的错误原因，不得静默失败，**不得通过放宽安全约定来"制造"初稿**。

**理由**：这是本 change 的底线不变量，直接交付 Milestone 4。把"无死路"与"系统级失败"清晰切分，避免把安全阻断误当"产品拒绝"——脱敏 fail-closed 阻断是 ADR-0001 D7 的既定红线，属"明确报错"而非"死路"，二者用户可感知的文案需区分（前者是合规阻断、后者本就不应出现）。

**备选**：把脱敏阻断也兜底为"降级生成一份初稿"——被否，直接违反 ADR-0001 D7"涉密恒走私有、永不降级；fail-open 永远是客户为某密级显式开启的结果，不是默认值"。

### D-F 兜底标记与预期管理为显式契约，落 c04 既有审计 / PostgreSQL

**决策**：起草生成 API / SSE 流式响应携带「兜底模式」标识字段；生成结果元数据含「是否兜底」「是否命中 RAG 样例」两维度，落 **ADR-0001 D5 PostgreSQL**（权威源，文档元数据 / 采纳稿回流同表）。c05 深做稿不携带兜底标记，二者元数据层可区分。前端对兜底稿展示统一口径"本稿为通用兜底、未经该文种深做、需人工重点把关"，文案不得暗示已达"可直接采纳"深做质量。关联的 c06 结构化场景上下文契约（目标 capability == c07 及目标文种 / 子类 / 行文方向 / 置信度）、是否命中样例入审计，复用 **c04 既有审计机制**，审计内容遵循 **c02 既有脱敏留痕约定、不落未脱敏原文**。

**理由**：兜底稿与深做稿在 PRD Success Metrics 的采纳率口径需区分统计，元数据维度是统计前提；预期管理避免用户拿兜底稿当深做稿用，是兜底"诚实"的产品契约。复用 c04 审计 / D5 PostgreSQL 落库而非另起一套，符合 ADR-0001 不另立第二套口径的总原则。

**备选**：兜底标记仅前端临时态、不落库——被否，无法支撑采纳稿回流与采纳率区分统计，也无审计可观测。

## Risks / Trade-offs

- **[兜底稿被误当深做稿采纳，质量风险外溢]** → 标记 + 预期口径作显式契约（D-F）；元数据「是否兜底」可在采纳稿回流时区分统计，对齐 PRD Success Metrics；前端文案禁用"可直接采纳"暗示语。残余风险靠人工最终把关总原则（PRD 核心原则，绝不自动发文）兜住。
- **[范文库未到位 / 检索无优质样例，兜底退回纯骨架后"不够像公文"]** → 这是 PRD 已识别风险（库空导致 UAT 说不清）。本 change 的缓解是退回纯骨架仍产出结构完整初稿、绝不阻断，并以元数据「未命中 RAG 样例」标出，使 UAT 能区分"缺数据"与"模型差"。提升像公文程度的进一步空间靠 c03 范文库按文种盘点到位（PRD 数据责任约定），不在本 change 内解决。
- **[c06 路由置信阈值 / 用户确认口径设置不当，过多请求落兜底或误把可深做的请求兜底]** → 触发判定以 c06 落定的目标 capability 为唯一输入（D-B），阈值与低置信用户确认口径均属 c06，本 change 不自行调、不在 c07 内对低置信自动兜底；通过审计记录「关联 c06 场景上下文契约 + 置信度」为阈值调优提供数据，但不在本 change 内重做分类或降级。
- **[兜底为产出初稿而绕过脱敏 / 密级路由，触碰保密红线]** → 复用与 c05 共用的生成编排管线、经 c01 窄抽象（D-A），完整经 c02 脱敏 + 密级路由；系统级失败边界显式禁止放宽安全约定"制造"初稿（D-E）。脱敏 fail-closed 阻断按 ADR-0001 D7 走"明确报错"而非降级生成。
- **[RAG 样例取用成为越权读范文的旁路]** → 样例检索强制经 c04 RBAC + c02 密级路由过滤，密级防越权以 ADR-0001 D5 / §2 PostgreSQL 预过滤为权威，Milvus 过滤仅性能纵深、条件后端注入（D-D）。兜底不新增任何取数口径。
- **[Anthropic 经 LiteLLM 的 OpenAI 格式流式有字段映射损耗 / 丢失 bug]** → 沿用 ADR-0001 D3 既有缓解（重度依赖时走原生 `/v1/messages` 或直连 anthropic-sdk-go，Go 端宽松 JSON 接非标字段）；兜底复用 c01 `model-streaming-passthrough` 既有 SSE 契约，不引入新的流式问题，异常下可观察行为沿用 c01。

### 对脱敏网关 / 密级防越权 / 三库一致性的显式影响

- **脱敏网关（c02 / ADR-0001 D7）**：零放宽。兜底所有模型调用经同一 c02 网关与密级路由（涉密恒私有、fail-closed 不静默发原文）。脱敏阻断在本 change 的语义是"系统级失败明确报错"，与"无死路"不矛盾——本 change 显式不把脱敏阻断兜底为降级生成。
- **密级防越权（c04 / ADR-0001 D5 / §2）**：零放宽。RAG 样例取用与生成稿落库均经 RBAC + 密级两层正交判定；PostgreSQL 预过滤为权威 ACL，Milvus 标量 / partition 过滤仅纵深。兜底不引入任何可绕过密级过滤的取数 / 写入路径。
- **三库一致性（ADR-0001 §2）**：本 change 对三库的新增仅是 **PostgreSQL 生成元数据上的两个维度字段**（是否兜底 / 是否命中 RAG 样例）及审计记录，落 D5 既有 PostgreSQL 权威源。**不新增 Milvus 集合 / 字段**（仅以读者身份消费 c03 既有混合索引，密级 = partition key、文种 = 标量过滤不变），**不新增 MinIO 桶**（兜底生成稿沿用既有生成稿桶，业务桶与 Milvus 桶物理隔离不变）。因此本 change 不引入新的跨库写一致性面——既有 outbox 异步管线 + 软删 + 对账 job 足以覆盖；兜底元数据与生成稿的关联通过既有全局 `chunk_id` / 文档主键维系。

## Migration Plan

本 change 为 MVP 范围内新增能力，无存量数据迁移；按以下步骤随生成服务一同上线：

1. PostgreSQL 生成元数据表增加「是否兜底」「是否命中 RAG 样例」两维度（可空，默认非兜底）→ 验证：存量深做稿元数据读取不受影响、新字段对老记录取默认值。
2. 生成服务接入兜底 prompt 组装分支与通用公文骨架模板，消费 c06 结构化场景上下文契约的目标 capability 作触发判定 → 验证：spec 场景"目标 capability 为 c07 走兜底""目标 capability 为 c05 不走兜底"全部通过。
3. 接入可得 RAG 样例注入（c03 可用时）与无样例退回骨架的降级 → 验证：c03 不可用 / 无命中时仍产出结构完整初稿且标「未命中 RAG 样例」。
4. SSE 响应与生成结果元数据携带兜底标识，前端展示「通用兜底」标注与预期口径提示 → 验证：兜底稿响应含兜底标识、深做稿不含、二者元数据可区分；前端文案不含"可直接采纳"暗示。
5. 兜底触发 / 降级接入 c04 既有审计 → 验证：审计含触发原因 + 是否命中样例 + 关联路由结论且不落未脱敏原文。

**回滚**：兜底分支为生成服务内的策略分叉，可经特性开关关闭兜底分支回退到"仅深做 + 路由"行为（此时未命中场景恢复为路由侧既有处置）；新增 PostgreSQL 维度字段可空、回滚后保留不影响存量读写，无破坏性 schema 变更。
