## Why

GovScribe 的高频文种深做（c05，9 个文种）与文种路由（c06）只能覆盖客户《支持公文类型》矩阵中的一部分；剩下的"其它 XX"场景、标黄数据稀缺文种（<100 条）、以及路由无法可靠归类的描述，如果直接返回"无法处理"，会在试点中制造"死路"——用户被产品当面拒绝，价值假设当场崩塌。PRD 的核心假设（Hypothesis）明确要求"覆盖高频文种**且对任意文种有通用兜底**"，Milestone 4 把"任意非高频场景均可走通用公文生成，全程不出现'无法处理'"列为独立交付目标。本 change 落地这条兜底底线：让产品对任意公文写作请求都给出一份"形似公文"的初稿，同时诚实地把它标注为通用兜底、不承诺"深做"质量，避免用户拿兜底稿当深做稿用。

## What Changes

- 新增**通用兜底生成路径**：当 c06 落定结构化场景上下文契约的「目标 capability」为 c07（包括其它已规划档外文种、标黄稀缺文种、以及 c06 无法可靠归类的描述）时，一律进入通用公文生成，产出标题 / 称谓 / 主送 / 正文 / 落款齐备的规范正文初稿，复用 c01 的窄抽象与 `model-streaming-passthrough`、并与 c05 共用同一条生成编排管线；c07 经 c01 窄抽象发起生成时，把 c06 契约中的「内容密级（非密 / 敏感 / 涉密）」作为出站请求密级随调用透传供 c02 路由，与 c05 同口径、不自建密级判定、不改写 / 不降级。
- 新增**通用公文结构骨架**：以"标题 + 主送机关 + 开头缘由 + 主体事项 + 结尾要求 + 落款"为通用公文要素模板，套用于任意场景，保证结构完整性，弥补无专属文种模板时的"不像公文"。
- 接入**可得 RAG 样例（可选依赖 c03）**：若范文库中检索得到与场景语义相近的样例，则作为 few-shot 提示注入提升"像公文"程度；库内无可用样例时退回纯结构骨架 + 通用提示工程，不阻断生成。
- 新增**兜底模式显式标注与预期管理**：对走通用兜底的初稿在生成结果上明确打"通用公文兜底"标记，并向用户传达"本稿为通用兜底、未经该文种深做、需人工重点把关"的预期口径，避免误用。
- 建立**"无死路"不变量**：任何进入起草的合法公文请求都必须落到"深做 / 已规划档 / 通用兜底"三者之一并产出初稿；定义无可用兜底时的明确报错边界（如脱敏 fail-closed 阻断、模型层不可用等系统级失败），并对兜底触发与降级留审计可观测。

本 change 不触及保密红线 / 密级路由 / 脱敏的既有口径：通用兜底只是生成路径的兜底分支，仍完整复用 c02 脱敏网关与密级路由（涉密恒走私有、fail-closed 不静默发原文），不放宽任何安全约定。本 change 落在 **MVP 范围内**（Milestone 4）。

## Capabilities

### New Capabilities

- `generic-document-fallback`：通用公文兜底生成能力。定义"何时触发兜底"（c06 结构化场景上下文契约的目标 capability == c07）、"兜底如何生成"（通用公文结构骨架 + 可得 RAG 样例 + 通用提示工程，复用 c01 窄抽象与 `model-streaming-passthrough`、与 c05 共用生成编排管线）、以及"无死路"不变量（任意合法公文请求必产出初稿，仅在系统级失败时按既有 fail-closed 边界明确报错）。
- `fallback-transparency`：兜底透明度与预期管理能力。定义初稿的"通用公文兜底"标注契约、对用户的预期口径（不承诺深做质量、需人工重点把关）、以及兜底触发 / 降级的可观测与审计要求，确保兜底稿不被误当深做稿。

### Modified Capabilities

（无。本 change 不修改其它 change 已确立的需求级行为；与 c05 深做文种、c06 路由通过"未命中即兜底"的边界衔接，对 c01 生成与 c03 检索为复用 / 可选依赖关系，不改其需求。）

## Impact

- **受影响代码 / 模块**：起草生成服务新增兜底 prompt 组装分支与通用结构骨架模板，与 c05 共用生成编排管线（检索 → 拼装 → 调 c01 → SSE）；与 c06 的衔接点为"消费结构化场景上下文契约的目标 capability == c07 即进兜底"，本 change 不复算分流降级判定。
- **接口**：起草生成 API / SSE 流式响应需携带"兜底模式"标识字段，供前端展示"通用兜底"标注与预期提示；生成结果元数据增加"是否兜底 / 是否命中 RAG 样例"以供采纳稿回流与审计区分。
- **依赖（前序 / 可选）**：依赖 c01 模型供应商窄抽象与 `model-streaming-passthrough`（必需，传输层）；与 c05 的 `draft-generation-streaming` 共用同一条生成编排管线（检索 → 拼装 → 调 c01 → SSE），仅在 prompt 模板 / few-shot 注入处分叉；消费 c06 结构化场景上下文契约的目标 capability 作为触发输入；依赖 c04 `rbac-authorization` 授权决策（`draft.create` 权限点，用于兜底生成端点入口，区别于现仅 RAG 样例取数的密级 / 访问控制），发起兜底起草前消费其授权决策、未授予即拒绝（fail-closed），本 change 不自建 RBAC 判定，与既有密级数据 ACL 预过滤两层正交叠加、不可相互替代（操作层 RBAC ⊥ 数据层密级 ACL）；可选依赖 c03 范文库 RAG 检索（有样例则用、无则退骨架，不阻断）。
- **数据 / 存储**：兜底生成稿仍按既有 MinIO 生成稿桶存储、PostgreSQL 落生成元数据；新增"兜底标记"维度用于后续区分兜底稿与深做稿的采纳率统计（PRD Success Metrics 口径区分）。
- **系统约束**：完整沿用 c02 脱敏网关 + 密级路由（fail-closed、涉密恒私有、映射不出域）与 c04 RBAC / 密级两层正交判定；兜底不引入任何安全降级。版本号 / 许可 / 信创 PoC 口径一律以 ADR-0001 为准，本 change 不另立。
