## ADDED Requirements

### Requirement: 兜底模式标注契约

凡经通用兜底路径产出的初稿，系统必须（MUST）在生成结果上携带明确的「通用公文兜底」标记。起草生成 API / SSE 流式响应必须包含「兜底模式」标识字段，前端据此展示「通用兜底」标注；生成结果元数据必须包含「是否兜底」与「是否命中 RAG 样例」两个维度。深做文种（c05）产出的初稿不得携带兜底标记，二者在元数据层面必须可区分。

#### Scenario: 兜底初稿响应携带兜底标识

- **WHEN** 一份初稿经通用兜底路径产出并通过 SSE 流式返回前端
- **THEN** 响应必须包含「兜底模式」标识字段，且生成结果元数据中「是否兜底」为真

#### Scenario: 深做稿与兜底稿元数据可区分

- **WHEN** 同一用户先后获得一份 c05 深做初稿与一份通用兜底初稿
- **THEN** 两份初稿的元数据「是否兜底」维度取值必须不同，系统据此可在采纳稿回流与统计中区分深做稿与兜底稿

### Requirement: 用户预期管理口径

对走通用兜底的初稿，系统必须（MUST）向用户传达一致的预期口径：本稿为通用兜底、未经该文种深做、需人工重点把关。前端必须在兜底初稿上呈现该口径提示，且提示用语不得暗示该稿已达「可直接采纳」的深做质量，以避免用户将兜底稿误当深做稿使用。该提示不得替代「人工最终把关、绝不自动发文」的总原则，而是在其之上额外强调兜底稿的把关重点。

#### Scenario: 兜底初稿呈现预期提示

- **WHEN** 前端展示一份带「通用公文兜底」标记的初稿
- **THEN** 界面必须呈现「本稿为通用兜底、未经该文种深做、需人工重点把关」的预期口径提示

#### Scenario: 提示不承诺深做质量

- **WHEN** 系统对兜底初稿生成预期提示文案
- **THEN** 文案不得包含暗示该稿已达「可直接交付 / 可直接采纳」深做质量的表述

### Requirement: 兜底可观测与审计

系统必须（MUST）对兜底触发与降级保留可观测与审计记录。每次进入通用兜底路径，必须记录关联的 c06 结构化场景上下文契约（目标 capability == c07 及目标文种 / 子类 / 行文方向 / 置信度）、是否命中 RAG 样例，供后续区分兜底稿与深做稿的采纳率统计（对齐 PRD Success Metrics 区分口径）。该审计记录复用 c04 既有审计与 PostgreSQL 元数据落库机制，不另起一套；记录内容不得包含未脱敏的敏感原文，遵循 c02 既有审计留痕约定。

#### Scenario: 兜底触发留审计

- **WHEN** 一个请求因 c06 落定目标 capability == c07 进入通用兜底路径并产出初稿
- **THEN** 系统必须记录关联的 c06 结构化场景上下文契约（目标 capability == c07 及置信度）、是否命中 RAG 样例，供采纳率统计区分

#### Scenario: 审计不落未脱敏原文

- **WHEN** 系统写入兜底触发 / 降级的审计记录
- **THEN** 记录内容必须遵循 c02 既有脱敏留痕约定，不得包含未脱敏的敏感原文
