## ADDED Requirements

### Requirement: 兜底触发判定

系统必须（MUST）以 c06 产出的结构化场景上下文契约为唯一输入判定是否进入通用兜底：当该契约的「目标 capability」字段为 c07 时，将合法公文写作请求路由进入通用兜底生成路径；当「目标 capability」为 c05 时，必须交由 c05 深做生成路径处理，不得进入兜底。本 change 不自行重做文种分类，不在 c07 内复算「是否命中 9 个深做文种」「是否命中 B 档且无专属模板」，分流降级判定一律以 c06 落定的目标 capability 为准。本 change 消费的、由 c06 产出的结构化场景上下文契约固定为 9 项：目标 capability（c05 / c07）、目标文种、代表子类、行文方向（上行 / 下行 / 平行）、置信度、用户原始场景描述、已补齐要素、缺失 / 未确认要素标记、内容密级（取值：非密 / 敏感 / 涉密）。

#### Scenario: 目标 capability 为 c07 走兜底

- **WHEN** c06 将场景（如「条例」属待计划训练档、或稀缺子类、或无法归类）落定为目标 capability == c07 并移交结构化场景上下文契约
- **THEN** 系统应当将该请求路由进入通用兜底生成路径，而非返回「无法处理」或要求用户改选文种

#### Scenario: 目标 capability 为 c05 不走兜底

- **WHEN** c06 将场景落定为目标 capability == c05（如以足够置信度判定为 9 个深做文种之一的「通知」）
- **THEN** 系统必须将请求交由 c05 深做生成路径处理，不得进入通用兜底路径

### Requirement: 通用公文结构骨架

通用兜底生成必须（MUST）套用统一的通用公文要素骨架，使任意场景的初稿都具备完整公文结构。骨架必须至少覆盖：标题、主送机关（无明确主送对象时以占位提示由人工补全，不得编造受文机关）、开头缘由、主体事项、结尾要求、落款（发文机关署名与成文日期，日期缺失时给占位提示）。当 c06 提供了行文方向（上行 / 平行 / 下行）或子类线索时，骨架的称谓与结尾用语应当据此适配；无任何线索时退回中性通用表述。

#### Scenario: 套用通用骨架产出完整结构

- **WHEN** 一个未命中深做文种的请求进入通用兜底生成
- **THEN** 产出的初稿必须包含标题、主送机关、开头缘由、主体事项、结尾要求、落款这六类要素，缺信息处以占位提示呈现而非留空或编造

#### Scenario: 无主送对象不编造受文机关

- **WHEN** 场景信息中没有明确的主送机关
- **THEN** 系统必须在主送位置给出需人工补全的占位提示，不得虚构具体单位名称

### Requirement: 可得 RAG 样例注入

通用兜底生成应当（SHALL）在 c03 范文库可用时检索与当前场景语义相近的范文样例，并将命中的样例作为 few-shot 提示注入以提升「像公文」程度。当 c03 不可用、检索无命中、或命中样例与场景相关度不足时，系统必须退回纯结构骨架 + 通用提示工程继续生成，绝不因缺少 RAG 样例而阻断或失败。RAG 样例的检索与使用必须遵循 c02/c04 既有的密级路由与访问控制判定，不得绕过密级过滤取用越权范文。

#### Scenario: 有可用样例则注入 few-shot

- **WHEN** c03 范文库检索到与当前场景相关度达标的范文样例
- **THEN** 系统应当将该样例作为 few-shot 提示注入生成请求，并在结果元数据标记「命中 RAG 样例」

#### Scenario: 无样例退回骨架不阻断

- **WHEN** c03 不可用或检索无相关度达标的样例
- **THEN** 系统必须退回纯通用公文结构骨架 + 通用提示工程继续完成生成，并在结果元数据标记「未命中 RAG 样例」

### Requirement: 复用 c01 窄抽象与共用生成编排管线

通用兜底生成必须（MUST）复用 c01 的模型供应商窄抽象接口与 `model-streaming-passthrough` 统一 SSE 流式透传（传输层）产出初稿，不得新建独立的模型调用链路；并与 c05 的 `draft-generation-streaming` 共用同一条生成编排管线（检索 → 拼装 → 调 c01 → SSE），仅在「用哪套 prompt 模板 / 是否注入专属 few-shot」一步分叉。兜底路径的初稿必须以 SSE 增量方式向前端流式输出，流中断 / 后端字段损耗等异常下的可观察行为沿用 c01 既有契约。c07 经 c01 窄抽象接口发起生成时，必须把 c06 结构化场景上下文契约中的内容密级（非密 / 敏感 / 涉密）作为出站请求密级随调用传递，供 c01 公网分支上的 c02 据以做出站密级路由判定；c07 与 c05 同口径，不自建密级判定、不改写或降级该内容密级。

#### Scenario: 兜底初稿走统一 SSE 流式输出

- **WHEN** 通用兜底生成开始产出初稿
- **THEN** 系统必须经与 c05 共用的生成编排管线、通过 c01 的统一 SSE 事件流向前端推送初稿增量，而非一次性返回或另起调用链路

#### Scenario: 内容密级随出站请求传递供 c02 路由

- **WHEN** 通用兜底经 c01 窄抽象接口发起生成、且 c06 结构化场景上下文契约携带内容密级（非密 / 敏感 / 涉密）
- **THEN** 系统必须把该内容密级作为出站请求密级随调用传递给 c01，供其公网分支上的 c02 做出站密级路由判定，c07 不自建密级判定、不改写或降级该内容密级

### Requirement: 无死路不变量

系统必须（MUST）保证任何进入起草的合法公文写作请求都落到「深做（c05）/ 已规划档 / 通用兜底」三者之一并产出初稿，全程不出现「无法处理」的死路。发起兜底起草前必须消费 c04 `rbac-authorization` 的授权决策（`draft.create` 权限点，用于兜底生成端点入口），未授予即拒绝（fail-closed），本 change 不自建 RBAC 判定；该操作层 RBAC 与既有密级数据 ACL 预过滤两层正交叠加、不可相互替代（操作层 RBAC ⊥ 数据层密级 ACL）。仅在系统级失败时允许不产出初稿并明确报错，系统级失败边界限定为：c02 脱敏网关 fail-closed 阻断（外置 NER 不可用且按密级不允许降级、又无私有模型可用，按 ADR-0001 D7）、c01 模型层不可用、或请求本身非合法公文写作请求。这些边界下系统必须返回明确、可读的错误原因，不得静默失败，且不得通过放宽安全约定来「制造」一份初稿。

#### Scenario: 未授权角色不可发起兜底起草

- **WHEN** 一个未被授予 `draft.create` 权限点的角色尝试发起通用兜底起草
- **THEN** 系统必须经 c04 `rbac-authorization` 的 `draft.create` 判定拒绝该请求（fail-closed），不自建 RBAC 判定，且该拒绝与数据层密级 ACL 预过滤相互独立、不可替代

#### Scenario: 任意合法请求必产出初稿

- **WHEN** 一个合法公文写作请求既未命中深做文种也无专属模板，已通过 c04 `draft.create` 授权，且 c01 模型层与 c02 脱敏网关均正常
- **THEN** 系统必须经通用兜底路径产出一份结构完整的初稿，不得返回「无法处理」

#### Scenario: 系统级失败明确报错而非死路

- **WHEN** c02 脱敏网关按 ADR-0001 D7 fail-closed 阻断本次公网调用（涉密永不降级 / 无可用降级路径）
- **THEN** 系统必须返回明确的阻断原因，不得静默失败，也不得绕过脱敏或密级路由强行生成初稿

#### Scenario: 兜底不放宽安全约定

- **WHEN** 通用兜底路径需要调用模型生成初稿
- **THEN** 系统必须完整经过 c02 脱敏网关与密级路由（涉密恒走私有、fail-closed 不静默发原文），不得为产出初稿而降低任何既有安全约束
