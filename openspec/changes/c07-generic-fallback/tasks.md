## 3. 兜底触发判定

- [ ] 3.2 实现"目标 capability == c05 时必交 c05、不得进兜底"的互斥分支 → 验证：spec 场景"目标 capability 为 c05 不走兜底"通过（如高置信「通知」由 c06 标 c05）。
- [ ] 3.3 实现"目标 capability == c07 走兜底"分支 → 验证：spec 场景"目标 capability 为 c07 走兜底"通过（如「条例」待计划训练档 / 稀缺子类 / 无法归类由 c06 标 c07，进兜底而非返回「无法处理」或要求改选文种）。
- [ ] 3.4 确认低置信 / 多义不在 c07 内自动兜底，而由 c06 走强制 Top-N 用户确认后落定目标 capability，c07 仅消费落定结果 → 验证：c07 无"低置信→自动兜底"分支，低置信路径在 c06 侧确认。

## 4. 通用公文结构骨架

- [ ] 4.1 实现固定六要素通用公文骨架模板：标题、主送机关、开头缘由、主体事项、结尾要求、落款（发文机关署名 + 成文日期）（spec 通用公文结构骨架，design D-C）→ 验证：spec 场景"套用通用骨架产出完整结构"通过，初稿含全部六类要素。
- [ ] 4.2 实现"缺信息占位不编造"：无明确主送机关时在主送位给"需人工补全"占位提示，成文日期缺失时给占位提示，绝不虚构受文机关或编造日期 → 验证：spec 场景"无主送对象不编造受文机关"通过，无明确主送时输出占位提示而非具体单位名。
- [ ] 4.3 实现行文方向 / 子类自适应：c06 给出上行 / 平行 / 下行或子类线索时，骨架称谓与结尾用语据此适配（上行用请示性结尾、平行用商洽 / 函复语气）；无线索退回中性通用表述（design D-C）→ 验证：给定行文方向线索时称谓 / 结尾用语随之变化，无线索时输出中性表述。

## 7. 无死路不变量与系统级失败边界

- [ ] 7.1 实现"无死路"不变量：任意进入起草的合法公文写作请求必落到「深做 / 已规划档 / 通用兜底」之一并产出初稿，全程不出现「无法处理」（spec 无死路不变量，design D-E，PRD Milestone 4）→ 验证：spec 场景"任意合法请求必产出初稿"通过。
- [ ] 7.2 实现系统级失败边界仅限三类——c02 脱敏 fail-closed 阻断（按 ADR-0001 D7）、c01 模型层不可用、请求本身非合法公文写作请求；这些边界返回明确可读错误原因、不得静默失败 → 验证：spec 场景"系统级失败明确报错而非死路"通过，脱敏阻断返回明确阻断原因而非死路文案。
- [ ] 7.3 实现"兜底不放宽安全约定"：兜底所有模型调用完整经 c02 脱敏网关与密级路由（涉密恒走私有、fail-closed 不静默发原文），不得为产出初稿绕过脱敏或密级路由、不得降级生成（spec 无死路不变量，design D-E）→ 验证：spec 场景"兜底不放宽安全约定"通过，脱敏阻断不被兜底为降级生成。
- [ ] 7.4 区分"合规阻断"与"产品死路"两类用户可感知文案：脱敏 fail-closed 阻断为合规阻断（明确报错），「无法处理」死路本就不应出现，二者文案不混淆（design D-E）→ 验证：脱敏阻断文案与（理论上不应出现的）死路文案语义可区分。
- [ ] 7.5 在兜底生成端点入口接入 RBAC 操作权限点：发起兜底起草前消费 c04 `rbac-authorization` 的授权决策（`draft.create` 权限点，用于兜底生成端点入口，区别于现仅 RAG 样例取数的密级 / 访问控制），未授予即拒绝（fail-closed），本 change 不自建 RBAC 判定，与既有密级数据 ACL 预过滤两层正交叠加、不可相互替代（操作层 RBAC ⊥ 数据层密级 ACL）（spec 无死路不变量，proposal 依赖段）→ 验证：spec 场景"未授权角色不可发起兜底起草"通过，未授予 `draft.create` 的角色发起兜底起草被拒绝；构造 RBAC 拒绝但密级 ACL 通过（及反之）两组样例，均不放行。
