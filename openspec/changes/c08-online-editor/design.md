## Context

c05/c07 已能产出"像公文"的 AI 初稿，但初稿仍是草稿。PRD Scope 把"在线编辑/核稿界面"与"人工最终把关"列为 MVP 纳入项：文秘必须能在浏览器内逐字核稿、按机关口径修订、决定是否采纳、把成稿带走精排；Agent 不自动发文。本 change 交付这块人工把关界面，并产出主指标"初稿采纳率"的数据来源与公网提质路线里"采纳稿回流"的入口信号。

当前状态与约束：

- 在线编辑组件由 ADR-0001 D8 定为 **OnlyOffice Document Server 社区版**（免费、MVP）：docx 原生保真 + 实时协同；与 Go 后端经 **JWT 回调**集成，文档读写走 **MinIO 预签名 URL**。社区版为 **AGPL v3 + 禁止去品牌**，只要不改其源码、仅经 API/JS 集成一般不传染本应用（商用前法务确认，遵 D8）。
- OFD 由 D9 定为**暂不支持**；GB/T 9704 红头版式精排后置。本 change 只做正文内容层面的核稿与 docx 保真。
- 架构方向 §2「编辑器可替换」：在线编辑做成可替换适配层（前端组件 + 后端回调/导出接口），内容按公文语义结构化存储、与版式解耦，导出抽象 `export(docx)` 预留 `export(ofd)`。
- 三库切分（§2）：PostgreSQL 为权威源 + 访问控制判定，MinIO 双桶隔离（范文桶 / 生成稿桶），Milvus 为派生索引。跨库无事务，靠 outbox 异步管线 + 软删 + 对账 job 收敛一致性。
- 前序依赖：c05/c07（有初稿可编）、c03（采纳稿回流入库）、c04（RBAC 与密级叠加判定）。

利益相关者：核心用户专职文秘（起草/核稿/决定采纳/导出）；系统管理员（部署 OnlyOffice、配额与审计）；c03（消费回流信号）；c04（提供权限裁定）。

## Goals / Non-Goals

**Goals:**

- 让文秘在浏览器内打开 c05/c07 的 AI 初稿，以 docx 原生保真方式核稿、按机关口径逐处修订、保存为成稿版本，并支持 OnlyOffice 的多人实时协同。
- 把 OnlyOffice 接入做成稳定的可替换适配层：业务接口只暴露公文域概念（文档、成稿版本、语义结构、采纳决策），不泄露 OnlyOffice 专有回调/参数格式；信创终局换 WPS 或接 OFD 服务时业务代码不动。
- 成稿内容按公文语义结构（标题、主送、正文结构、落款等）结构化存储于 PostgreSQL、与版式表现解耦；docx 原件存 MinIO 生成稿桶，二者经成稿版本记录关联。
- 提供"采纳决策"四档登记（直接用/小改/大改/弃用），作为采纳率口径的权威数据来源，并对计入采纳的成稿产出供 c03 回流入库的决策信号。
- 提供成稿导出为 .docx 的能力，导出做成统一抽象 `export(docx)`、预留 `export(ofd)` 接口位（MVP 不实现）。

**Non-Goals:**

- 不做初稿生成（c05/c07）。
- 不做采纳稿向量入库与 RAG 回流落地（c03）；本 change 只产出成稿与回流信号，不直接入库。
- 不另立权限/密级口径；"能否打开/编辑/采纳/导出"一律调用 c04 的 RBAC + 密级叠加判定。
- 不做 GB/T 9704 红头版式精排，不实现 OFD 输出（遵 D9）。
- 不解决 OnlyOffice 商业授权/去品牌/信创官方认证与 AGPL 法务确认（后置成本，遵 D8、ADR-0001 §5/§6）。

## Decisions

### D-1：在线编辑器选型与集成方式 = OnlyOffice 社区版 + JWT 回调 + MinIO 预签名 URL（遵 ADR-0001 D8）

直接采用 D8 既定口径，不另立。集成链路：

1. 文秘打开初稿 → 后端经 c04 判定权限 → 通过后为该 docx 签发**有限有效期的 MinIO 预签名读取 URL**，组装 OnlyOffice document config（含 `document.url`=预签名 URL、`document.key`=成稿版本指纹、`editorConfig.callbackUrl`=后端回调地址），并对整个 config 签发 **JWT** 后下发前端。
2. 前端用 OnlyOffice JS API（`DocsAPI.DocEditor`）以该 config 初始化编辑器。
3. 编辑/保存时 OnlyOffice 向 `callbackUrl` 回调（携带 JWT 与 `status`）；后端**校验 JWT** → 对 `status=2`（MustSave）/`status=6`（强制保存）依回调内的文档 URL 拉取最新 docx → 经**预签名写入 URL 写回生成稿桶** → 登记新成稿版本 → 返回 `{"error":0}`。

为什么不直接用 TipTap（D8 提及的 MIT 轻量备选）做 MVP 主编辑器：公文核稿要求 docx 原生保真（称谓/主送/正文结构/落款的版式细节不能丢）与多人协同，TipTap 是富文本而非 docx 引擎、保真与协同需自建，违背"可直接采纳"目标；故 MVP 主编辑器用 OnlyOffice，TipTap 仅作未来轻量备选保留在适配层之下。

为什么不为编辑器单开一套鉴权：JWT 仅护 OnlyOffice ↔ 后端这条通道（防伪造回调篡改文档状态），业务级"谁能编"仍归 c04，二者职责不同、不重叠（见 D-4）。

合规边界（遵 D8）：仅用原版社区版镜像经 API/JS 集成、不打入对其源码的修改，编辑器界面保留 OnlyOffice 品牌标识；JWT 校验失败的回调一律拒绝且不改动任何文档状态、记审计。

### D-2：编辑器可替换适配层 = 公文域接口 + 适配器实现（遵 ADR-0001 §2）

把"编辑器相关的一切"收敛进一层适配器，业务侧只见公文域接口。建议的稳定接口（命名为示意，落地以 tasks 为准）：

- `OpenSession(docID, user) → EditorConfig`：返回前端初始化所需的不透明配置（内部封装 OnlyOffice config + JWT + 预签名 URL）。
- `OnSaved(callback) → newVersion`：消化编辑器回调、写回生成稿桶、登记成稿版本（内部封装 OnlyOffice callback 协议解析）。
- `Export(versionID, format) → object`：统一导出抽象（见 D-5）。

业务实体（成稿版本、公文语义结构、采纳决策）**不依赖 OnlyOffice 任何私有表示**。替换底层编辑器（如信创终局换 WPS）或接 OFD 导出服务时，仅重写适配器实现，业务逻辑不动。

为什么不让业务直接调 OnlyOffice JS/回调格式：那会把 OnlyOffice 的 config 字段、callback `status` 语义、JWT 细节散布到业务代码，未来换引擎需全局改动，违背 §2「编辑器可替换」与 D8「商业授权是延后非消除」的退路设计。

### D-3：公文语义结构化存储与版式解耦（PostgreSQL 权威 + MinIO 原件）

成稿采用"双表示"持久化：

- **版式表示**：docx 原件存 MinIO **生成稿桶**（与范文桶双桶隔离，遵 §2），由对象引用标识。
- **语义表示**：标题/主送/正文结构/落款等公文要素以与编辑器无关的结构化字段存 PostgreSQL，作为权威源。

二者经**成稿版本记录**（version 表）关联：每次保存生成一条新版本（含 docx 对象引用 + 语义结构快照 + 来源初稿引用 + 时间戳），版本不可变、追加式，便于追溯与采纳口径逐篇回溯。

语义结构的抽取时机：保存回调写回 docx 后，由适配层从 docx 解析公文要素落 PostgreSQL（MVP 可先存最小要素集，结构随文种规范演进）。语义结构表达**不绑定 OnlyOffice 内部格式**，保证编辑器后端可替换（D-2）且为 c03 回流提供与版式无关的内容载体。

为什么不只存 docx：只存版式原件则采纳率口径、回流信号、未来检索都得反复解析二进制 docx，且强绑定 docx/OnlyOffice，违背版式解耦；只存语义结构则丢失文秘要带走精排的保真原件。故二者并存、经版本关联。

### D-4：权限与密级一律调用 c04，不另立口径（遵 ADR-0001 D10、§2 密级防越权）

"能否打开/编辑/采纳/导出"四类动作均在动作入口调用 c04 的 **RBAC（能做什么操作）+ 密级（能看/能外发什么数据）叠加判定**，二者正交、任一不通过即拒绝（不得因单层通过而放行）。判定未通过时：不下发 OnlyOffice config、不签发预签名 URL、不登记采纳、不生成导出文件，并提示无权限、记审计。

本 change 是浏览器内人工编辑与导出，**不触发对公网模型的调用，因此不直接经过脱敏网关与密级路由**（D7）；但成稿落生成稿桶、决策与元数据落 PostgreSQL，故密级防越权由 c04 的 Postgres 预过滤（权威 ACL）裁定本动作可否进行。预签名 URL 必须有限有效期、过期重签，绝不向前端/编辑器下发 MinIO 长期凭据——避免绕过 c04 直接拉对象。

### D-5：统一导出抽象 export(docx) / 预留 export(ofd)（遵 ADR-0001 D8、D9、§2）

导出对业务以目标格式参数化：`Export(versionID, format)`。

- `format=docx`：MVP 实现。优先复用 OnlyOffice 在编辑场景内的转换/下载能力得到与所导出版本一致的 docx；若需在 OnlyOffice 编辑场景之外以纯 Go 生成 docx，则用 **MIT 许可的 `mmonterroca/docxgo`**，**排除 AGPL 的 `unioffice`、`fumiama/go-docx`**（污染闭源单二进制，遵 D8）。
- `format=ofd`：**预留接口位但 MVP 不实现**（遵 D9）。请求 ofd 必须返回明确的"暂不支持 OFD"结果，**不得静默失败、不得用非 OFD 内容冒充 OFD**。未来接 OFD 时（Go 无成熟原生 OFD 库，届时定位为可旁挂独立服务，遵 D9），仅在导出抽象下补后端实现，业务调用契约不变。

导出权限同样经 c04 叠加判定（D-4）。

### D-6：采纳决策四档登记与回流信号（对齐 PRD 采纳率口径，回流落地交 c03）

采纳决策结论取自固定四档枚举 **直接用 / 小改 / 大改 / 弃用**（对齐 PRD 验收 rubric 可用度采纳口径），落 PostgreSQL，每条关联对应初稿与其当前成稿版本引用；非四档枚举拒绝并返回校验错误。

采纳率口径遵 PRD：**"直接用 + 小改"计入采纳，"大改 / 弃用"不计入**，作为主指标"初稿采纳率"的权威数据来源、支撑试点 UAT 量化。

回流边界：本 change 是采纳决策持有方，对计入采纳（直接用/小改）的成稿**产出回流决策信号**供 c03 消费入库；弃用不产出信号。回流信号字段固定为四项：**成稿引用（MinIO 对象键/chunk 标识）、采纳结论（直接用/小改/大改/弃用；「直接用+小改」计入采纳）、密级、文种**。其中**密级来源 = 继承原生成任务密级，或在采纳/核稿时由文秘显式确认**——保证信号送达 c03 时密级不悬空（c03 对缺密级的回流一律阻断）。信号经**与 c03 约定的同一套 outbox/事件表**交付（**不直调 c03 API**，避免双写），由 c03 **单点**消费入 Milvus 与对账。本 change **不直接执行向量入库**——避免与 c03 的 outbox 异步管线（§2 跨库一致性、主键全局 `chunk_id`）重复落地、产生双写不一致。

## Risks / Trade-offs

- **[OnlyOffice AGPL v3 + 禁去品牌、正式交付仍需商业授权]** → 仅经 API/JS 集成、不改源码、保留品牌以守住 AGPL 边界（遵 D8）；MVP 不承诺去品牌/SLA，商业授权与 AGPL 法务确认登记为后置成本（D8、ADR-0001 §6），不在本 change 解决。
- **[信创认证缺口：统信/龙芯无一手认证，OnlyOffice 无 LoongArch64 官方镜像]** → 仅登记（遵 ADR-0001 §5/§6）；MVP 公网阶段不阻塞，明确"勿向客户承诺龙芯"，认证与 ARM64/LoongArch64 镜像 PoC 由信创阶段处理。
- **[伪造回调篡改文档状态]** → 每次 config 下发与回调均签发/校验 JWT；JWT 缺失/签名错误/过期一律拒绝、不拉取不写入、记鉴权失败审计（D-1）。
- **[预签名 URL 泄露或长凭据外泄绕过 c04]** → URL 有限有效期、过期重签、绝不下发长期凭据；所有动作前置 c04 叠加判定（D-4），密级防越权由 Postgres 预过滤权威裁定。
- **[三库一致性：docx 落 MinIO 与语义结构/版本落 Postgres 双写不原子]** → 以 Postgres 成稿版本记录为权威、追加式不可变；写回顺序为先写对象再登记版本，版本登记成功才视为保存完成（回调返回 error=0）；孤儿对象由对账 job 清理，复用 §2 软删 + 定期对账思路，不引入跨库事务。
- **[回流双写：本 change 与 c03 都可能写入库管线导致 Milvus/Postgres 不一致]** → 本 change 只产信号不入库（D-6），入库与对账单点归 c03，经约定 outbox/事件表交付，主键沿用全局 `chunk_id`。
- **[OnlyOffice 协同保存丢失一方编辑]** → 依赖 OnlyOffice 协同会话机制，回调以会话最终 docx 为准写回单一新版本；trade-off：MVP 不自建 CRDT/逐字段合并，接受 OnlyOffice 的协同语义。
- **[docx 解析语义结构不全/文种差异大]** → MVP 先存最小公文要素集，结构随文种规范演进；语义结构与版式解耦，演进只动适配层解析逻辑、不动版式原件。

## Migration Plan

本 change 为新增能力（无 Modified Capabilities），无既有行为迁移；部署步骤与回滚：

1. 部署 **OnlyOffice Document Server 社区版**原版镜像（不改源码、带品牌），配置 JWT 密钥（与后端共享）、与后端回调地址互通。
2. 后端上线：OnlyOffice config 下发 + JWT 签发/校验、MinIO 预签名 URL 签发（生成稿桶）、保存回调处理、成稿版本与公文语义结构持久化（PostgreSQL）、`export(docx)`（预留 `export(ofd)`）、采纳决策登记与回流信号产出；所有动作前置接入 c04 判定。
3. 前端上线：写作工作台内嵌 OnlyOffice 编辑器组件、采纳决策四档交互、导出入口。
4. 与 c03 对齐回流信号的 outbox/事件表契约后开通信号产出。
- **回滚**：编辑器组件与回调接口可整体下线，回退到"仅查看初稿"；成稿版本与采纳决策为追加式、回滚不破坏已落数据；OnlyOffice 为独立运行时组件，可单独停用而不影响 c01–c07。

## Open Questions

- OnlyOffice AGPL v3 商用法务确认与社区版/商业授权切换时点（遵 D8，后置）。
- 公文语义结构最小要素集的精确字段与各文种差异化结构的演进边界（与 c05/c07 文种规范对齐）。
- 回流决策信号契约【已定】（遵本 design D-6 回流信号契约 + c03 corpus-ingestion 单点入库 + ADR-0001 §2/D5）：信号字段固定为成稿引用（MinIO 对象键/chunk 标识）+ 采纳结论 + 密级 + 文种，密级来源 = 继承原生成任务密级或采纳/核稿时文秘显式确认，交付走与 c03 约定的同一套 outbox/事件表、由 c03 单点入库；仅 outbox 表的物理列布局与 `chunk_id` 生成时点属 c03 落地细节，以 c03 schema 为准对接，不影响本 change 信号契约。
- 多人协同下的并发会话与权限实时变更（编辑中被 c04 撤权）的处理策略。
- 统信/龙芯官方信创认证与 OnlyOffice LoongArch64 镜像可用性 PoC 结论（遵 ADR-0001 §5/§6，后置）。
