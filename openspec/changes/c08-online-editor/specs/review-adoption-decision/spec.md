## ADDED Requirements

### Requirement: 采纳决策四档登记
系统必须（MUST）允许文秘对每篇 AI 初稿登记一条采纳决策，结论必须取自固定四档枚举：直接用 / 小改 / 大改 / 弃用（对齐 PRD「验收 rubric 草案」可用度采纳口径）。每条决策必须关联到对应初稿与其当前成稿版本引用，并落 PostgreSQL 持久化。登记采纳决策的权限必须以 c04 登记的 `adopt.decide` 权限点（在线核稿动作以 `review.online`）经 c04 的 RBAC 与密级叠加判定校验，不另立口径。

#### Scenario: 登记一条采纳决策
- **WHEN** 具备采纳权限的文秘对某篇初稿选择四档之一（直接用 / 小改 / 大改 / 弃用）并提交
- **THEN** 系统必须在 PostgreSQL 中登记该决策结论及其关联的初稿与成稿版本引用

#### Scenario: 拒绝非四档枚举的结论
- **WHEN** 提交的采纳结论不属于直接用 / 小改 / 大改 / 弃用四档之一
- **THEN** 系统必须拒绝该登记并返回校验错误，不得写入非法结论

#### Scenario: 无采纳权限被拒绝
- **WHEN** 不具备采纳权限（RBAC 或密级判定不通过）的用户尝试登记采纳决策
- **THEN** 系统必须拒绝该操作并提示无权限，不得写入任何决策

### Requirement: 采纳率口径权威数据来源
采纳决策记录必须（MUST）作为主指标"初稿采纳率"的权威数据来源；采纳口径必须遵 PRD 规定——"直接用 + 小改"计入采纳，"大改 / 弃用"不计入采纳。系统必须可按该口径统计采纳情况以支撑试点 UAT 量化。

#### Scenario: 直接用与小改计入采纳
- **WHEN** 按采纳率口径统计一批已登记决策
- **THEN** 结论为"直接用"或"小改"的篇目必须计入采纳，结论为"大改"或"弃用"的篇目不得计入采纳

#### Scenario: 决策可追溯到具体成稿
- **WHEN** 查询某篇初稿的采纳决策
- **THEN** 返回的记录必须包含四档结论及其指向的成稿版本引用，使采纳率统计可逐篇追溯

### Requirement: 被采纳成稿产出回流决策信号
对于计入采纳（直接用 / 小改）的成稿，作为采纳决策持有方的本 change 必须（MUST）产出回流决策信号，其字段固定为：成稿引用（MinIO 对象键 / chunk 标识）、采纳结论（直接用 / 小改 / 大改 / 弃用；「直接用 + 小改」计入采纳）、密级、文种。其中密级来源必须为继承原生成任务密级，或在采纳 / 核稿时由文秘显式确认，不得悬空。该信号必须经与 c03 约定的同一套 outbox / 事件表交付（不直调 c03 API，避免双写），由 c03 单点消费入 Milvus 与对账；本 change 仅产出成稿与信号，向量入库与 RAG 回流落地由 c03 负责，本 change 不得直接执行入库。

#### Scenario: 采纳触发回流信号
- **WHEN** 一篇初稿被登记为"直接用"或"小改"
- **THEN** 系统必须产出包含成稿引用（MinIO 对象键 / chunk 标识）、采纳结论、密级、文种四项字段的回流决策信号，并经与 c03 约定的同一套 outbox / 事件表交付供 c03 消费入库

#### Scenario: 回流信号密级来源确定
- **WHEN** 为计入采纳的成稿产出回流信号
- **THEN** 信号中的密级必须取自原生成任务继承密级或采纳 / 核稿时文秘的显式确认，不得为空（c03 对缺密级的回流一律阻断）

#### Scenario: 弃用不产出回流信号
- **WHEN** 一篇初稿被登记为"弃用"
- **THEN** 系统不得为其产出回流入库信号

#### Scenario: 不在本 change 直接入库
- **WHEN** 回流决策信号已产出
- **THEN** 本 change 必须仅停留在产出成稿与信号，向量入库与 RAG 回流落地必须交由 c03 经约定 outbox / 事件表单点完成，本 change 不得直调 c03 入库 API
