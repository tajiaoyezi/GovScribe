## 1. 适配层接口与公文域数据模型（先于一切实现）

- [ ] 1.1 定义编辑器可替换适配层的稳定接口（公文域语义）：`OpenSession(docID, user) → EditorConfig`、`OnSaved(callback) → newVersion`、`Export(versionID, format) → object`；接口契约只暴露文档/成稿版本/语义结构/采纳决策等公文域概念，不在签名与返回结构中泄露 OnlyOffice 专有的 config 字段、callback `status` 语义或 JWT 细节（遵 D-2、§2「编辑器可替换」）。验证：接口包内无 OnlyOffice 类型出现在对外签名上，业务层仅依赖该接口。
- [ ] 1.2 设计成稿版本表（PostgreSQL）：追加式不可变，含 docx 对象引用、语义结构快照引用、来源初稿引用、版本指纹（用作 OnlyOffice `document.key`）、创建人、时间戳（遵 D-3）。验证：同一文档多次保存生成多条版本、历史版本不被覆盖。
- [ ] 1.3 设计公文语义结构存储模型（PostgreSQL）：标题、主送、正文结构、落款等最小公文要素集，字段表达独立于 OnlyOffice 内部格式与版式表现（遵 D-3）。验证：语义结构可在不解析 docx 二进制的前提下被查询返回。
- [ ] 1.4 设计采纳决策表（PostgreSQL）：四档枚举（直接用/小改/大改/弃用）、关联初稿引用与当前成稿版本引用、登记人、时间戳；枚举值约束在库层与应用层双重校验（遵 D-6）。验证：写入非四档值被拒绝。
- [ ] 1.5 确认 MinIO 生成稿桶与范文桶双桶隔离配置，成稿原件与导出产物仅落生成稿桶（遵 §2、D-3）。验证：写回路径目标桶为生成稿桶、范文桶只读不被本 change 写入。

## 2. OnlyOffice 接入与 JWT 回调鉴权

- [ ] 2.1 部署 OnlyOffice Document Server 社区版原版镜像（不改源码、保留品牌标识），配置与后端共享的 JWT 密钥及回调地址互通（遵 D-1、D8 合规边界）。验证：镜像为未修改的官方社区版、编辑器界面保留 OnlyOffice 品牌。
- [ ] 2.2 实现 `OpenSession`：组装 OnlyOffice document config（`document.url`=MinIO 预签名读取 URL、`document.key`=成稿版本指纹、`editorConfig.callbackUrl`=后端回调地址），对整个 config 签发 JWT 后下发前端（遵 D-1）。验证：下发的 config 携带有效 JWT、`document.key` 与版本指纹一致。
- [ ] 2.3 实现保存回调接口与 JWT 校验：对每条回调先校验 JWT，仅对 `status=2`（MustSave）/`status=6`（强制保存）依回调内文档 URL 拉取最新 docx；校验失败一律拒绝、不拉取不写入、返回拒绝并记鉴权失败审计（遵 D-1、规约「JWT 缺失/非法回调被拒绝」）。验证：伪造/过期/缺失 JWT 的回调被拒绝且文档状态无任何变化。
- [ ] 2.4 实现 `OnSaved`：写回顺序为先经预签名写入 URL 写 docx 到生成稿桶、再登记新成稿版本，版本登记成功才返回 OnlyOffice 约定成功应答 `{"error":0}`（遵 D-1、D-3 三库一致性）。验证：合法 `status=2` 回调后生成新版本且 docx 落生成稿桶、返回 error=0。
- [ ] 2.5 保存回调后由适配层从写回的 docx 解析公文语义要素并落 PostgreSQL，与该成稿版本关联（遵 D-3）。验证：保存后可查到该版本的语义结构且与 docx 内容对应。

## 3. MinIO 预签名 URL 与文档读写

- [ ] 3.1 实现有限有效期的 MinIO 预签名读取 URL 签发，注入 OnlyOffice config 供编辑器加载 docx（遵 D-1、规约「文档读写经 MinIO 预签名 URL」）。验证：URL 含有效期、过期后不可用。
- [ ] 3.2 实现预签名写入 URL 签发用于回调写回生成稿桶，绝不向前端或编辑器下发 MinIO 长期凭据（遵 D-4）。验证：前端/编辑器侧不持有任何 MinIO 长期 access key。
- [ ] 3.3 实现预签名 URL 过期后重新签发：文秘再次打开同一文档时重签新 URL、不复用已过期 URL（遵规约「预签名 URL 过期后重新签发」）。验证：过期后再打开拿到新 URL。

## 4. 权限与密级判定接入 c04（每个动作入口）

- [ ] 4.1 在打开/编辑/在线核稿/采纳/导出各动作入口，分别以 c04 登记的权限点 `document.open` / `document.edit` / `review.online` / `adopt.decide` / `document.export` 调用 c04 的 RBAC + 密级叠加判定，任一不通过即拒绝、不得因单层通过而放行，亦不得绕过或在本 change 硬编码权限名（遵 D-4、c04 权限点登记中心（rbac-authorization）+ D-4，不另立口径）。验证：各动作以对应权限点经 c04 校验；构造 RBAC 通过但密级不通过（及反之）两组样例，均被拒绝。
- [ ] 4.2 判定未通过时的拒绝行为：不下发 OnlyOffice config、不签发预签名 URL、不登记采纳、不生成导出文件，提示无权限并记审计（遵 D-4）。验证：无权限用户打开文档时无 config/URL 下发、不返回文档内容。

## 5. 公文语义结构化存储与版式解耦

- [ ] 5.1 落地语义结构查询能力，返回以公文要素表达、不依赖 OnlyOffice 私有数据格式的结构（遵 D-3、规约「语义结构不绑定具体编辑器」）。验证：查询结果中无 OnlyOffice 内部字段。
- [ ] 5.2 校验"双表示"经成稿版本记录关联：docx 原件（MinIO）与语义结构快照（PostgreSQL）由同一版本记录串联，版本追加不可变（遵 D-3）。验证：按版本可同时取到对应 docx 引用与语义结构快照。
- [ ] 5.3 实现孤儿对象对账 job：写回 docx 成功但版本登记失败产生的孤儿对象由对账清理，复用 §2 软删 + 定期对账思路，不引入跨库事务（遵 D-3 风险项）。验证：模拟版本登记失败后对账可识别并清理孤儿对象。

## 6. 统一导出抽象与 docx 导出

- [ ] 6.1 实现统一导出抽象 `Export(versionID, format)`，对业务以目标格式参数化、与具体导出后端解耦（遵 D-5、§2）。验证：业务侧仅按 format 调用、不感知后端实现。
- [ ] 6.2 实现 `export(docx)`：优先复用 OnlyOffice 编辑场景内转换/下载得到与所导出版本一致的 docx；需在编辑场景之外纯 Go 生成时用 MIT 许可的 `mmonterroca/docxgo`，并在依赖中排除 AGPL 的 `unioffice`、`fumiama/go-docx`（遵 D-5、D8）。验证：导出 .docx 内容与所导出成稿版本一致；依赖扫描无 AGPL docx 库。
- [ ] 6.3 预留 `export(ofd)` 接口位但 MVP 不实现：请求 ofd 时返回明确"暂不支持 OFD"结果，不得静默失败、不得用非 OFD 内容冒充 OFD（遵 D-5、D9）。验证：以 format=ofd 调用返回明确暂不支持。
- [ ] 6.4 导出动作前置经 c04 叠加判定（复用第 4 组），无权限拒绝且不生成任何文件（遵 D-4、D-5）。验证：无导出权限用户请求导出不产生文件。

## 7. 采纳决策登记与回流信号

- [ ] 7.1 实现采纳决策登记接口：四档枚举校验、关联初稿与当前成稿版本引用、落 PostgreSQL；非四档枚举返回校验错误（遵 D-6）。验证：合法四档可登记、非法枚举被拒。
- [ ] 7.2 实现采纳率口径统计：按 PRD 口径"直接用 + 小改"计入采纳、"大改 / 弃用"不计入，支撑试点 UAT 量化并可逐篇追溯到成稿版本（遵 D-6、PRD 采纳率主指标）。验证：给定一批决策，统计计入项与 PRD 口径一致。
- [ ] 7.3 实现回流决策信号产出：对计入采纳（直接用/小改）的成稿，经与 c03 约定的同一套 outbox/事件表交付字段固定为「成稿引用（MinIO 对象键/chunk 标识）+ 采纳结论 + 密级 + 文种」的信号，其中密级取自继承原生成任务密级或采纳/核稿时文秘显式确认（不得悬空）；弃用不产出信号；不直调 c03 API、不直接执行向量入库（遵 D-6 回流信号契约 + c03 corpus-ingestion 单点入库 + ADR-0001 §2/D5，入库与对账单点归 c03）。验证：直接用/小改产出含四项字段且密级非空的信号、弃用不产出、本 change 无 Milvus 写入路径与对 c03 入库 API 的直调。
