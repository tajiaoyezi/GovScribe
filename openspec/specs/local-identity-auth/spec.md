# local-identity-auth

## Purpose

定义本地账号密码认证、会话 JWT、登录失败处理与可插拔认证 Provider 抽象，为后续 RBAC 和访问控制叠加判定提供统一 principal。

## Requirements

### Requirement: 本地账号密码登录

系统应当（SHALL）提供本地账号密码登录入口，由 principal 提交用户名与密码完成身份认证。账号由系统管理员维护，系统必须（MUST）不提供面向终端用户的自助注册流程（契合单租户私有部署，口径见 ADR-0001 D10）。密码必须以加盐哈希形式存储于 PostgreSQL，系统必须不以明文或可逆加密形式持久化密码，且任何接口与日志都不得回显密码原文或完整哈希。

#### Scenario: 凭证正确登录成功

- **WHEN** 一个已启用账号提交与存储哈希匹配的用户名与密码
- **THEN** 系统应当判定认证通过，并进入会话凭证签发流程（见「会话凭证 JWT 签发与校验」需求）

#### Scenario: 密码错误拒绝登录

- **WHEN** principal 提交存在的用户名但密码与存储哈希不匹配
- **THEN** 系统必须拒绝登录、不签发任何会话凭证，并返回不区分"用户名不存在"与"密码错误"的统一失败提示（避免账号枚举）

#### Scenario: 密码以加盐哈希落库

- **WHEN** 系统管理员创建账号或重置密码而写入 PostgreSQL
- **THEN** 持久化的密码字段必须为加盐哈希值，且数据库中不存在该密码的明文或可逆密文副本

### Requirement: 会话凭证 JWT 签发与校验

登录成功后系统应当（SHALL）签发 JWT 作为会话凭证，后端各接口必须（MUST）据其识别请求所属 principal（含用户标识与所属角色信息，供下游 RBAC 与访问控制叠加判定消费）。JWT 必须设置有效期，过期或签名校验失败的凭证必须被拒绝。受保护接口在缺失或携带无效凭证时必须返回未认证错误，并且不得执行任何业务逻辑。

#### Scenario: 携带有效 JWT 访问受保护接口

- **WHEN** 请求携带未过期且签名有效的 JWT 访问受保护接口
- **THEN** 系统应当解析出 principal 标识与角色信息并放行至后续授权判定阶段

#### Scenario: JWT 过期被拒绝

- **WHEN** 请求携带的 JWT 已超过有效期
- **THEN** 系统必须判定为未认证、拒绝该请求并不执行业务逻辑，提示需重新登录

#### Scenario: JWT 签名无效被拒绝

- **WHEN** 请求携带的 JWT 签名校验失败（被篡改或非本系统签发）
- **THEN** 系统必须拒绝该请求并记为未认证，不得据其中声明的身份执行任何操作

#### Scenario: 缺失凭证访问受保护接口

- **WHEN** 请求未携带任何会话凭证却访问受保护接口
- **THEN** 系统必须返回未认证错误并终止处理，不得返回任何业务数据

### Requirement: 登录失败处理与会话失效

系统应当（SHALL）对连续登录失败进行限制以防暴力破解，达到阈值后必须（MUST）临时拒绝该账号的后续登录尝试。被系统管理员停用的账号必须无法通过认证；账号停用后，系统应当使该账号既有会话凭证失效或在下次校验时被拒绝，停用账号不得继续访问任何受保护接口。

#### Scenario: 连续失败触发登录限制

- **WHEN** 同一账号在限定窗口内连续登录失败次数达到阈值
- **THEN** 系统必须临时拒绝该账号的进一步登录尝试，并在审计中留痕（留痕承载见 rbac-authorization 的用户管理需求）

#### Scenario: 停用账号无法登录

- **WHEN** 一个已被系统管理员停用的账号提交正确的用户名与密码
- **THEN** 系统必须拒绝认证、不签发会话凭证

#### Scenario: 停用后既有会话失效

- **WHEN** 账号被停用后，其此前签发且尚未过期的 JWT 再次访问受保护接口
- **THEN** 系统必须拒绝该请求，不得因凭证未到期而放行

### Requirement: 可插拔认证 Provider 抽象

系统应当（SHALL）将认证抽象为统一的认证 Provider 接口：输入为凭证或外部回调，输出为已认证的 principal（含用户标识与角色映射所需信息）。MVP 必须（MUST）实现本地账号密码 Provider；系统必须为信创 / 政企终局对接客户统一身份认证（CAS / OIDC / LDAP / 国产 IAM）预留 Provider 实现的接入位（与 ADR-0001 D1 模型接入窄抽象同思路）。切换或新增 Provider 时，下游业务代码、RBAC 判定与密级判定均不得改动；不同 Provider 认证出的 principal 必须收敛为同一内部身份模型，供统一的会话与授权逻辑消费。

#### Scenario: MVP 使用本地账号密码 Provider

- **WHEN** 部署仅启用本地账号密码 Provider 且 principal 通过其完成认证
- **THEN** 系统应当输出统一内部身份模型的已认证 principal，后续会话签发与授权判定按统一逻辑进行

#### Scenario: 切换为外部 IAM Provider 不改业务代码

- **WHEN** 终局部署将认证 Provider 由本地账号密码切换为 CAS / OIDC / LDAP / 国产 IAM 之一
- **THEN** 仅认证适配层发生变化，业务 handler、RBAC 判定服务与密级判定逻辑的代码不得因此改动，且认证产出的 principal 仍收敛为同一内部身份模型

#### Scenario: 外部 Provider principal 收敛为统一身份模型

- **WHEN** 外部 IAM 回调返回其自有的用户标识与属性
- **THEN** Provider 必须将其映射为系统统一身份模型（用户标识 + 角色映射所需信息），下游不感知具体 Provider 来源
