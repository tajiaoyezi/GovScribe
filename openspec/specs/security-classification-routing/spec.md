# security-classification-routing

## Purpose

定义内容密级到公网或私有模型路径的路由决策、与脱敏网关 fail-closed 状态机的协同，以及密级路由配置的可维护性与默认安全策略。

## Requirements

### Requirement: 按密级决定调用路径

路由决策器应当（SHALL）依据数据密级决定模型调用路径：非密直接走公网、敏感走脱敏后公网、涉密强制走私有化模型。密级 → 模型路由的映射表必须不出域。涉密内容必须强制改道私有路径，不得以任何配置开关使其走公网或脱敏后公网。

#### Scenario: 非密走公网

- **WHEN** 待调用内容密级为非密
- **THEN** 路由决策器应当允许直接走公网模型（仍按网关三层识别处置，但不强制改道）

#### Scenario: 敏感走脱敏后公网

- **WHEN** 待调用内容密级为敏感
- **THEN** 路由决策器必须要求先经脱敏网关识别与占位替换后再走公网

#### Scenario: 涉密强制私有

- **WHEN** 待调用内容密级为涉密
- **THEN** 路由决策器必须强制改道私有化模型，禁止走公网或脱敏后公网

### Requirement: 与脱敏网关协同导向不出域路径

密级路由必须（MUST）与脱敏网关协同：当网关因 fail-closed 触发"转私有"或当密级判定为涉密时，路由决策器应当把该请求导向不出域的私有路径；在 NER 不可用且无私有模型可用的高风险场景下，路由与网关共同保证非降级路径默认为阻断而非放行原文。

#### Scenario: NER 不可用且无私有触发阻断

- **WHEN** 脱敏网关判定 NER 不可用、无可用私有模型，且当前密级未开启降级
- **THEN** 路由与网关协同必须阻断该公网调用，不得回退到发送原文

#### Scenario: 涉密在 NER 不可用时仍走私有

- **WHEN** 内容为涉密且外置 NER 不可用
- **THEN** 路由决策器必须仍强制走私有模型，不因 NER 状态改变涉密的私有路由结论

#### Scenario: 内容密级缺失或未知按涉密 fail-closed

- **WHEN** 上游（经 c05 / c07 → c01）传入的内容密级缺失或未知
- **THEN** 网关必须按涉密 fail-closed 处置——强制转私有，无可用私有则阻断，涉密永不降级，并入审计；不得缺省按非密放行公网

#### Scenario: 转私有定向运行时失败收敛为阻断

- **WHEN** 转私有定向成功发起但私有端点运行时调用失败（c01 返回端点不可达 / 超时 / 上游报错）
- **THEN** 路由必须按 fail-closed 收敛为阻断（涉密绝不回退公网、非密不静默降级原文），入审计；与「无已启用私有配置」信号为两条不同入口、同归阻断

### Requirement: 密级路由配置可维护且不出域

密级 → 模型路由配置（含各密级是否允许降级发送的开关）应当（SHALL）落 PostgreSQL 并可由系统管理员维护；该配置与映射表均不出 Go 进程边界对外发送。降级开关默认应当为关闭（fail-closed），仅在客户为某非涉密密级显式开启后方可对该密级生效。

#### Scenario: 降级开关默认关闭

- **WHEN** 客户未对某敏感密级显式配置降级开关
- **THEN** 该密级的降级开关必须默认为关闭，NER 不可用且无私有时按阻断处置

#### Scenario: 管理员修改路由配置

- **WHEN** 系统管理员更新某密级的路由 / 降级配置
- **THEN** 路由决策器后续请求必须按更新后的配置生效，且配置变更应当可被审计
